Threads, Thread Pools & ExecutorService How Java runs work concurrently — and why you almost never manage threads yourself.

SCOPE · 165 min · 6 stages — Week 2 opens. The execution machinery every server-side system is built on.
WEEK 2 BEGINS — CONCURRENCY MASTERY

Week 1 taught you how memory works and how the JVM behaves under load. Week 2 is about doing many things at once correctly. Day 5 gave you the rules (the memory model, happens-before); now you learn the machinery that runs concurrent work: threads, thread pools, and the JVM’s ExecutorService framework. The architect’s lesson today is counterintuitive: you will spend almost no time creating threads, and almost all your time deciding how many, what queue, and what happens when the system is overwhelmed. Those three decisions are the difference between a service that degrades gracefully and one that falls over.

STAGE 1 25 MIN

Threads — what they are, what they cost, and their lifecycle

A Java platform thread is a thin wrapper over an operating-system thread. When you call thread.start(), the JVM asks the OS for a real kernel thread, and the OS scheduler time-slices it across cores. That 1:1 mapping has consequences:

  • Memory: each thread reserves a stack (~512KB–1MB by default, -Xss). 10,000 threads = gigabytes of stacks before any work.
  • Creation cost: a syscall into the kernel — expensive relative to the work most tasks do.
  • Context-switch tax: the OS switching between far more threads than cores burns CPU on bookkeeping instead of your code.

This is why thread pools exist: threads are costly enough that you reuse a fixed set rather than spawning one per task. (Day 8’s virtual threads change this calculus for I/O-bound work — but platform threads, and pools, remain the model for CPU-bound work and the foundation you must understand first.)

The thread lifecycle — the states you read in a thread dump

Every thread is in exactly one state. These are the same states you diagnosed in Day 6’s thread dumps:

StateMeaning
NEWCreated but start() not yet called
RUNNABLEExecuting or ready to execute (includes running in native/I/O)
BLOCKEDWaiting to acquire a monitor lock held by another thread
WAITINGParked indefinitely (wait(), join(), park()) until signalled
TIMED_WAITINGParked with a timeout (sleep, timed wait/join)
TERMINATEDFinished or threw

Runnable, Callable & Future — describing work

// Runnable: returns nothing, can't throw checked exceptions
Runnable r = () -> log.info("done");
// Callable: returns a value, can throw checked exceptions
Callable<BigDecimal> c = () -> price(symbol);
// Future: a handle to a result that may not exist yet
Future<BigDecimal> f = executor.submit(c);
BigDecimal p = f.get(2, TimeUnit.SECONDS);   // blocks (with timeout!); may throw

Future.get() blocks the calling thread until the result is ready — always prefer the timed overload so a stuck task can’t hang you forever. future.cancel(true) requests cancellation (interrupting the running thread).

STAGE 2 25 MIN

ExecutorService — and why the convenience factories are traps

ExecutorService decouples what work runs from how it’s scheduled. You submit tasks; the pool decides which thread runs them and when. Two submit styles matter:

executor.execute(runnable);              // fire-and-forget; uncaught exception → thread's handler (printed)
Future<?> f = executor.submit(runnable);  // returns a Future; exception is CAPTURED in the Future

The swallowed-exception trap. A task submitted with submit() that throws does not print anything — the exception is stored in the Future and only surfaces when you call get(). If you never call get() (fire-and-forget via submit), the failure vanishes silently. Use execute() for fire-and-forget, or always inspect the Future.

The Executors factory methods — convenient and dangerous

FactoryWhat you getThe hidden danger
newFixedThreadPool(n)n threads, unbounded LinkedBlockingQueueQueue grows without limit under load → OutOfMemoryError
newCachedThreadPool()Unbounded threads, SynchronousQueueOne thread per task under burst → thread explosion, OOM
newSingleThreadExecutor()1 thread, unbounded queueSerial; same unbounded-queue risk
newScheduledThreadPool(n)Delayed / periodic tasksAn uncaught exception silently kills a periodic task (Stage 5)

The architect’s rule: don’t use the Executors factories in production — build a ThreadPoolExecutor explicitly.

Both newFixedThreadPool and newCachedThreadPool hide an unbounded resource — an unbounded queue or unbounded threads. Under sustained overload, “unbounded” means “until the JVM dies.” The factories are fine for examples and tests; for a real service you construct the pool yourself so you control the queue size and the saturation behavior (Stages 3–4). This is one of the most cited pieces of production Java advice.

🎯 INTERVIEW QUESTIONS

Q: Why avoid Executors.newFixedThreadPool in production? — It uses an unbounded LinkedBlockingQueue; if tasks arrive faster than they’re processed, the queue grows until OOM. Build a ThreadPoolExecutor with a bounded queue instead.

Q: Difference between execute() and submit()? — execute takes a Runnable and returns nothing (exceptions go to the uncaught handler); submit takes Runnable/Callable, returns a Future, and captures exceptions in it — silent unless you call get().

QUICK CHECK

A service uses newCachedThreadPool for incoming requests and falls over under a traffic spike with thousands of threads. What exactly happened, and which factory parameter caused it?

STAGE 3 30 MIN

ThreadPoolExecutor internals — the real engine

Every factory above just calls this constructor with different arguments. Understand these seven parameters and you understand all thread pools:

new ThreadPoolExecutor(
    8,                              // corePoolSize — threads kept alive even when idle
    16,                             // maximumPoolSize — hard ceiling on threads
    60, TimeUnit.SECONDS,           // keepAliveTime — idle non-core threads die after this
    new ArrayBlockingQueue<>(1000),   // workQueue — BOUNDED queue holding waiting tasks
    namedThreadFactory,             // threadFactory — names threads, sets daemon/handler
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection handler — what to do when full (Stage 4)
);

How a submitted task flows — the exact algorithm

This sequence is the single most important thing to know about thread pools, and it’s a frequent interview question:

  1. If fewer than corePoolSize threads are running, start a new thread for the task — even if other threads are idle.
  2. Else, try to enqueue the task in the work queue.
  3. If the queue is full, start a new thread up to maximumPoolSize.
  4. If the pool is at max and the queue is full, reject the task via the saturation policy.

The counterintuitive consequence: max pool size does nothing with an unbounded queue. Because step 3 only fires when the queue is full, an unbounded queue (the newFixedThreadPool default) can never fill — so the pool never grows past corePoolSize, and maximumPoolSize is silently ignored. To actually use extra threads under load, you need a bounded queue. This trips up almost everyone.

The work queue choices

  • ArrayBlockingQueue — bounded, fixed capacity. The architect’s default: it gives backpressure.
  • LinkedBlockingQueue — optionally bounded; unbounded by default (the trap from Stage 2).
  • SynchronousQueue — zero capacity; each task must be handed directly to a thread (used by newCachedThreadPool, forcing thread creation).
  • PriorityBlockingQueue — unbounded, ordered by priority.
🎯 INTERVIEW QUESTIONS

Q: Walk me through what a ThreadPoolExecutor does with a submitted task. — Core threads first, then queue, then grow to max, then reject. Recite the four-step algorithm.

Q: I set corePoolSize=4, maxPoolSize=50, but only ever see 4 threads under heavy load. Why? — The queue is unbounded, so it never fills, so step 3 never triggers — maxPoolSize is dead. Use a bounded queue.

STAGE 4 25 MIN

Saturation policies & backpressure — what happens when you’re full

When the pool is at max threads and the queue is full, the next task hits the rejection handler (saturation policy). Choosing it is an architectural decision about how your service behaves under overload.

PolicyBehaviorWhen to use
AbortPolicy (default)Throws RejectedExecutionExceptionFail fast; caller must handle / retry
CallerRunsPolicyRuns the task on the submitting threadBackpressure — slows the producer naturally
DiscardPolicySilently drops the taskLossy-OK work (e.g. best-effort metrics)
DiscardOldestPolicyDrops the oldest queued task, retriesLatest-value-wins (e.g. live quotes)

The backpressure pattern: bounded queue + CallerRunsPolicy.

This is the single most useful production combination. When the pool saturates, the task runs on the caller’s thread — which means the thread that submits work is now busy doing work and can’t submit more. The producer is automatically throttled to the rate the pool can sustain, without dropping anything and without unbounded memory growth. The system slows down instead of falling over. That graceful degradation under overload is exactly what an architect designs for.

The alternative — an unbounded queue — doesn’t apply backpressure; it absorbs the overload into memory until the JVM dies. “Slow down” is almost always better than “crash later.”

🎯 INTERVIEW QUESTIONS

Q: How do you apply backpressure with a thread pool? — A bounded queue plus CallerRunsPolicy: saturation pushes work back onto the producer thread, throttling intake to sustainable throughput without dropping tasks or growing memory.

Q: When would you use DiscardOldestPolicy? — When only the freshest data matters and stale queued tasks are worthless — live market quotes, sensor readings — so dropping the oldest and keeping the newest is correct.

QUICK CHECK

Your ingestion service must never lose an event but also must not OOM under a burst. Which queue + policy combination do you choose, and what does the producer experience during the burst?

STAGE 5 25 MIN

Sizing, scheduling & graceful shutdown

How many threads? The sizing formula

The wrong size is either starvation (too few) or context-switch tax (too many). Brian Goetz’s formula (from Little’s Law):

threads = cores × targetUtilization × (1 + waitTime / computeTime)
  • CPU-bound work (wait ≈ 0): threads ≈ cores + 1. More threads just thrash the scheduler — there’s no more CPU to use.
  • I/O-bound work (wait » compute): many more threads, because most are parked waiting on I/O, not using a core. A task that waits 9ms per 1ms of compute can keep ~10× cores busy.

This is also why virtual threads exist (Day 8): for I/O-bound work the “right” platform-thread count is huge and expensive — virtual threads make that count free. For CPU-bound work, a small platform-thread pool sized near the core count remains correct.

ScheduledExecutorService — periodic work

ses.scheduleAtFixedRate(task, 0, 1, SECONDS);    // every 1s from START time (rate)
ses.scheduleWithFixedDelay(task, 0, 1, SECONDS); // 1s gap AFTER each finishes (delay)
  • fixedRate aims for a steady cadence; if a run overruns the period, the next starts immediately (runs can bunch up / overlap-pressure).
  • fixedDelay always leaves a fixed gap between end and next start — safer when runs vary in length.

The silent-death pitfall: if a scheduled task throws an uncaught exception, the scheduler cancels all future executions of that task — silently. Your periodic job just stops, with no error. Always wrap the task body in try/catch so an exception can never escape.

Graceful shutdown — the step everyone forgets

executor.shutdown();                              // stop accepting new tasks; finish queued ones
if (!executor.awaitTermination(30, SECONDS))     // wait for in-flight work
    executor.shutdownNow();                       // interrupt running tasks, drop the queue

shutdown() is graceful (drains the queue); shutdownNow() is forceful (interrupts and returns undone tasks). A pool whose threads are non-daemon will keep the JVM alive if you never shut it down — a classic “my app won’t exit” and resource-leak bug. Also name your threads via a ThreadFactory so dumps (Day 6) are readable.

🎯 INTERVIEW QUESTIONS

Q: Size a pool for CPU-bound vs I/O-bound work. — CPU-bound: ~cores+1 (more just thrashes). I/O-bound: cores × (1 + wait/compute) — many more, since threads are mostly parked.

Q: scheduleAtFixedRate vs scheduleWithFixedDelay? — fixedRate measures the period from each start (steady cadence, can overlap-pressure); fixedDelay measures the gap after each finish (safe for variable durations).

Q: shutdown() vs shutdownNow()? — shutdown stops intake and drains the queue; shutdownNow interrupts running tasks and returns the undone ones. Pair shutdown with awaitTermination, then escalate to shutdownNow.

The challenge — pick the right executor for three workloads

The capstone. For each workload, design the executor (type, queue, size, saturation policy) and justify it. Answer all three parts (50+ characters each), then reveal the model answers.

SCENARIO · THREE DIFFERENT WORKLOADS

One framework, three jobs

A. An image-resizing service: each task is pure CPU (decode, resize, encode), runs ~200ms, on a 16-core box. Requests can burst far above capacity.

B. An order-enrichment service: each task makes 3 sequential REST calls (~150ms total wait, ~5ms compute) and must never drop an order, but must not OOM under a spike.

C. A market-data fan-out: pushes the latest quote to subscribers every 100ms; if the system falls behind, stale quotes are worthless — only the newest matters.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Platform threads are expensive (~1MB stack, kernel-backed) — pools exist to reuse them. Know the six lifecycle states (they’re what you read in a thread dump).
  • Don’t use the Executors factories in production — they hide unbounded queues/threads that OOM under load. Build ThreadPoolExecutor explicitly.
  • Task flow: core → queue → max → reject. Max pool size is dead with an unbounded queue.
  • Bounded queue + CallerRunsPolicy = backpressure — the system slows instead of crashing.
  • Size by workload: CPU-bound ≈ cores+1; I/O-bound much higher (or virtual threads). Always shut pools down gracefully, and never let a scheduled task throw.
