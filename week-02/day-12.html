Phase 2 · DSA · Week 2 / Day 12

Atomics, CAS & Lock-Free Patterns Thread-safety without locks — how compare-and-swap works, where it wins, and the failure modes (ABA) that bite the unwary.

schedule 170 min · layers 6 stages
RECAP FROM DAY 11

Day 11 made shared state safe with locks — pessimistic concurrency: “assume a conflict, exclude everyone else.” That’s correct but it serializes, and contention silently caps throughput. Today is the other philosophy: optimistic concurrency — “assume no conflict, do the work, and only retry if someone beat me.” Its primitive is compare-and-swap (CAS), a single hardware instruction, and it’s the foundation of every atomic, every lock-free data structure, and the entire java.util.concurrent package. The architect’s lesson: lock-free isn’t magic free-speed — it’s a different trade-off, with its own contention behavior and a famous, subtle failure mode called ABA.

STAGE 1 30 MIN

Compare-and-swap — the foundation

CAS is one atomic CPU instruction with three operands: a memory location, an expected value, and a new value. Its meaning: “if the location still holds the expected value, set it to the new value and report success; otherwise change nothing and report failure.” The compare and the swap happen indivisibly — no other thread can sneak in between.

// The shape of every atomic update — the CAS retry loop
int oldVal, newVal;
do {
    oldVal = counter.get();          // 1. read the current value
    newVal = oldVal + 1;            // 2. compute the new value
} while (!counter.compareAndSet(oldVal, newVal));  // 3. swap iff unchanged, else retry

If another thread changed the value between steps 1 and 3, compareAndSet returns false and the loop retries with the fresh value. No thread ever blocks; a thread only ever redoes its own work. Under the hood this is a single instruction (CMPXCHG on x86, load-linked/store-conditional on ARM) — cheaper than a lock, which needs OS arbitration when contended.

CAS gives all three Day-5 guarantees for a single variable. The atomic classes are built on volatile fields plus CAS, so: atomicity (the read-modify-write is indivisible), visibility (the volatile read/write flushes caches), and ordering (a successful CAS establishes a happens-before edge, like a lock release). This is why an AtomicInteger needs no synchronized — the hardware does the mutual exclusion for that one word.

Optimistic vs pessimistic — the mental model

A lock is pessimistic: acquire first, assume a conflict, block others. CAS is optimistic: do the work, then check; pay only when there’s an actual conflict. Optimistic wins when conflicts are rare (the common case) — no blocking, no context switches, no scheduler involvement. The catch you must internalize: under contention, CAS doesn’t block, it spins and retries, burning CPU. We’ll see in Stage 3 that under very high contention a lock can actually win.

🎯 INTERVIEW QUESTIONS

Q: What is CAS and what does it guarantee? — An atomic compare-and-set instruction (expected, new). It atomically updates only if the value is unchanged, giving atomicity, visibility, and ordering for a single variable without a lock.

Q: How does a non-blocking counter work? — A CAS retry loop: read, compute, compareAndSet; on failure re-read and retry. No thread blocks; a thread only redoes its own computation.

Q: Optimistic vs pessimistic locking? — Pessimistic (locks) blocks up front assuming conflict; optimistic (CAS) proceeds and retries only on actual conflict. Optimistic wins under low contention; locks can win under heavy contention.

STAGE 2 25 MIN

The atomic family — counters, references & functional updates

java.util.concurrent.atomic wraps CAS in friendly classes: AtomicInteger, AtomicLong, AtomicBoolean, and the general AtomicReference<T>. The convenience methods hide the retry loop:

counter.incrementAndGet();              // atomic ++i (CAS loop inside)
counter.getAndAdd(10);                 // returns old, adds 10, atomically
flag.compareAndSet(false, true);       // one-shot guard (do-once patterns)

Functional updates (Java 8) — CAS loops you don’t write

// updateAndGet / accumulateAndGet run a CAS loop with YOUR function
max.accumulateAndGet(candidate, Math::max);     // atomic "running max"
ref.updateAndGet(s -> s.withLastSeen(now()));     // atomic transform of a reference

Caution: the function you pass may be retried (the loop re-runs it on CAS failure), so it must be pure and side-effect-free — no logging, no mutation, no I/O inside it. This is a frequent subtle bug.

AtomicReference + immutable snapshots — the architect’s pattern

This is the lock-free way to manage compound state safely — and it ties Day 5 and Day 8 together:

record Config(String url, int timeout) {}            // immutable (Day 8)
AtomicReference<Config> cfg = new AtomicReference<>(initial);
cfg.updateAndGet(c -> new Config(c.url(), newTimeout));  // swap a whole new snapshot atomically

Readers do cfg.get() and see a consistent, fully-built snapshot (safe publication, Day 5) — never a torn half-update — with zero locking. Updates create a new immutable object and CAS the reference. This is copy-on-write at the reference level, and it’s how you publish configuration, routing tables, and feature flags in a hot path.

(For the last drop of performance there are AtomicIntegerFieldUpdater / AtomicReferenceFieldUpdater, which apply CAS to a volatile field directly — avoiding a wrapper object per field. Niche, but worth knowing the name.)

🎯 INTERVIEW QUESTIONS

Q: Implement a thread-safe counter without synchronized. — AtomicInteger with incrementAndGet() (a CAS loop). For very high write contention, prefer LongAdder (Stage 3).

Q: Why must the lambda passed to updateAndGet be side-effect-free? — It can be re-executed on CAS failure (the retry loop), so side effects would run multiple times.

Q: How do you safely publish a config object that changes occasionally, lock-free? — An immutable record in an AtomicReference; readers get() a consistent snapshot, writers CAS a new instance.

STAGE 3 25 MIN

Contention & LongAdder — when CAS spins too much

CAS’s weakness is its strength inverted. Under low contention it’s near-free. Under high write contention — many threads hammering one AtomicLong — they all read the same value, all try to CAS, all but one fail, and all retry. The hot cache line ping-pongs between cores (“cache-line bouncing”), and throughput collapses even though nobody is blocked. Spinning isn’t free; it’s wasted CPU.

LongAdder beats AtomicLong for high-contention counters — by not contending.

LongAdder keeps an array of cells; under contention, different threads update different cells, so their CAS operations rarely collide. The total is computed as the sum of all cells on read. You trade a single hot location for many cold ones. The cost: sum() is not a perfectly-atomic instantaneous snapshot and is slightly slower than a single get(). So: LongAdder for write-heavy counters you mostly increment and rarely read (metrics, request counts); AtomicLong when you need the exact current value frequently or when contention is low. (LongAccumulator/DoubleAdder generalize this to other reductions.)

When a lock actually beats CAS

The “lock-free is always faster” belief is wrong. Under sustained extreme contention on a single value, a lock can outperform a CAS loop: a lock puts losers to sleep (one cache-line transfer, then quiet), whereas CAS losers keep spinning and bouncing the cache line. The architect’s rule: measure. Optimistic for low/moderate contention; striped structures (LongAdder) or a lock for the genuinely hot single point.

🎯 INTERVIEW QUESTIONS

Q: LongAdder vs AtomicLong? — LongAdder stripes across cells to avoid contention, summed on read — far faster for high-frequency increments, but sum() isn’t a precise atomic snapshot. AtomicLong is better for low contention or when you read the exact value often.

Q: Is lock-free always faster than locking? — No. Under extreme contention CAS loops spin and bounce the cache line, while a lock parks losers. Measure; sometimes a lock or a striped counter wins.

STAGE 4 25 MIN

The ABA problem

CAS asks “is the value still what I expected?” But “the same value” is not the same as “nothing happened.” If a value goes A → B → A between your read and your CAS, the CAS succeeds — it sees A and assumes nothing changed — even though the world moved underneath you. That’s the ABA problem.

Why ABA is usually harmless for numbers but dangerous for references. For a counter, A→B→A genuinely leaves the value at A — no harm. The danger is with pointers in lock-free structures: a node can be removed and a different node recycled to the same memory address, so the reference is “equal” (A) but now points at a different logical object. Your CAS succeeds and you splice corrupted links — a lost node, a cycle, or a resurrected element.

The fix: version stamps

// AtomicStampedReference pairs the reference with an int "version"
AtomicStampedReference<Node> top = new AtomicStampedReference<>(head, 0);

int[] stampHolder = new int[1];
Node cur = top.get(stampHolder);          // read value AND stamp
int stamp = stampHolder[0];
...
top.compareAndSet(cur, next, stamp, stamp + 1);  // must match value AND stamp; bump stamp

AtomicStampedReference makes the CAS check both the reference and a monotonically increasing stamp. Now A→B→A changes the stamp each transition, so the comeback-to-A no longer looks unchanged — the CAS fails and you retry correctly. (AtomicMarkableReference is the lighter cousin: a single boolean “mark” bit instead of a counter — enough when you only need to flag “logically deleted.”)

🎯 INTERVIEW QUESTIONS

Q: What is the ABA problem? — A value changes A→B→A between a thread’s read and its CAS; the CAS succeeds because the value matches, missing that intervening changes occurred — dangerous for pointer-based lock-free structures where a recycled address looks identical.

Q: How do you fix ABA? — AtomicStampedReference (reference + version stamp) so any change bumps the stamp and a returned-to-A value fails the CAS; or AtomicMarkableReference for a single mark bit. Languages with GC also reduce (but don’t eliminate) ABA by preventing immediate address reuse.

STAGE 5 25 MIN

Lock-free structures, progress guarantees & when not to roll your own

The Treiber stack — lock-free in 6 lines

AtomicReference<Node> top = new AtomicReference<>();
void push(T v) {
    Node n = new Node(v);
    do { n.next = top.get(); } while (!top.compareAndSet(n.next, n));
}
T pop() {
    Node cur;
    do { cur = top.get(); if (cur == null) return null; }
    while (!top.compareAndSet(cur, cur.next));   // ← the ABA-vulnerable line
    return cur.value;
}

This is the canonical lock-free stack: each operation is a CAS retry loop on the head pointer. It’s elegant — and the pop() CAS is exactly where ABA can strike (Stage 6’s challenge).

Progress guarantees — the vocabulary

  • Lock-free: the system always makes progress — some thread completes in a bounded number of steps. Individual threads may starve (keep losing CAS), but the structure never stalls. (CAS loops are lock-free.)
  • Wait-free: stronger — every thread completes in a bounded number of steps. No starvation possible. Hard to build; rare.
  • Obstruction-free: weakest — a thread completes if it runs in isolation long enough. (StampedLock’s optimistic read is in this family.)

The architect’s rule: almost never hand-write a lock-free structure. They are extraordinarily hard to get right (ABA, memory reclamation, subtle reorderings) and even harder to debug. Reach instead for the battle-tested lock-free collections in java.util.concurrent — ConcurrentLinkedQueue/Deque (lock-free, the right answer to “I need a concurrent stack/queue”), ConcurrentHashMap, LongAdder. For the extreme high-throughput case, study the LMAX Disruptor — a ring buffer using CAS and cache-line padding that powers HFT systems — but use the library, don’t reinvent it. Roll your own only with a very good reason and a lot of testing (jcstress).

(The low-level engine has modernized: hand-written lock-free code historically used sun.misc.Unsafe; the supported API since Java 9 is VarHandle, which exposes CAS plus fine-grained memory-ordering modes — plain, opaque, acquire/release, volatile. You almost never need it, but that’s what the JDK’s own concurrent classes use.)

🎯 INTERVIEW QUESTIONS

Q: Lock-free vs wait-free? — Lock-free guarantees system-wide progress (some thread advances); wait-free guarantees per-thread progress (every thread finishes in bounded steps, no starvation). CAS loops are lock-free, not wait-free.

Q: I need a concurrent stack/queue — do I write a Treiber stack? — No. Use ConcurrentLinkedDeque/ConcurrentLinkedQueue — already lock-free and correct. Hand-rolling invites ABA and reclamation bugs.

QUICK CHECK

A CAS retry loop is lock-free but not wait-free. Construct the scenario: which thread could, in theory, never complete — and why does the structure still count as making progress?

STAGE 6 35 MIN

The challenge — spot the ABA bug

The capstone. Study the lock-free stack and the interleaving, then answer all three parts (50+ characters each) and reveal the model answers.

SCENARIO · READ CAREFULLY BEFORE ANSWERING

A recycled-node stack

A Treiber stack (the pop() from Stage 5) is used with an object pool that recycles popped Node objects back for reuse (to reduce allocation). Stack contents, top first: A → B → C.

An interleaving:

  1. Thread 1 enters pop(): reads cur = A, reads cur.next = B. Then it is paused by the scheduler, just before its CAS.
  2. Thread 2 runs: pop() A, pop() B (both recycled to the pool), then push() a reused node that happens to be the same object A. Stack is now A → C.
  3. Thread 1 resumes and executes compareAndSet(A, B).
YOUR ANSWERS · MINIMUM 50 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 50 characters
MODEL ANSWER

The CAS succeeds — and that’s the bug. When Thread 1 resumes, top is once again the object A (re-pushed by Thread 2), so compareAndSet(A, B) sees the expected value and sets top = B. But B was already popped and recycled — it’s no longer in the stack. The real stack was A → C; after the CAS it becomes top = B, whose next pointer is stale/garbage (it pointed at C at read time, but B has since been reused). C is lost (leaked or unreachable), and the stack now exposes a node that was supposed to be out of circulation — structural corruption. The CAS “passed” an equality check while the world had completely changed: textbook ABA.

0 / 50 characters
MODEL ANSWER

Primitive fix: replace AtomicReference<Node> with AtomicStampedReference<Node>. Read the node and the stamp; CAS on both and bump the stamp. Now Thread 2’s pops/push each advance the stamp, so when Thread 1 does its CAS the stamp no longer matches — it fails and retries against the real current state. Don’t-hand-roll fix: use ConcurrentLinkedDeque (or ConcurrentLinkedQueue) — a tested, lock-free j.u.c. structure that handles these hazards for you; deleting the custom stack deletes the bug class. The precondition: the bug required node recycling — reusing the same object identity. Without the object pool, garbage collection alone makes ABA far less likely here, because a popped node can’t be reused (and re-appear as the “same” reference) while Thread 1 still holds a reference to it. The object pool, added to “reduce allocation,” is what reintroduced ABA — a cautionary tale about premature optimization.

0 / 50 characters
MODEL ANSWER

First counter: a synchronized block for a single increment is pessimistic overkill — it forces lock acquisition (and, under contention, parking/waking) where a CAS would suffice. Revert to AtomicLong.incrementAndGet() (or better, see below): non-blocking, no monitor, much higher throughput for an independent increment. Second counter: the AtomicLong is the profile hot spot because of extreme write contention — all threads CAS the same word, fail, retry, and bounce the cache line between cores. The fix is LongAdder: it stripes increments across cells so threads rarely collide, summing on read. You give up a precise instantaneous atomic snapshot (sum() is eventually consistent), which is fine for a metric read once a second. The through-line: match the tool to the contention — lock for compound invariants, CAS/AtomicLong for low-contention single values, LongAdder for hot single-value writes.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • CAS is optimistic concurrency — read, compute, compare-and-set, retry. One hardware instruction; gives atomicity + visibility + ordering for a single variable, no lock.
  • Atomics wrap CAS; updateAndGet/accumulateAndGet run your (pure!) function in the loop. AtomicReference + immutable snapshot = lock-free compound state.
  • LongAdder for hot counters (stripes to avoid contention); AtomicLong for low contention / exact reads. Lock-free isn’t always faster — measure.
  • ABA: A→B→A fools CAS; harmless for numbers, corrupting for pointers. Fix with AtomicStampedReference.
  • Don’t hand-roll lock-free structures — use ConcurrentLinkedQueue/Deque, ConcurrentHashMap, the Disruptor. Lock-free = system progress; wait-free = per-thread progress.
