Synchronization Primitives: synchronized, Lock & ReentrantLock How to make shared mutable state safe — from the intrinsic monitor to explicit locks, conditions, and deadlock-free design.

schedule 170 min · layers 6 stages
RECAP FROM DAYS 5 & 10

Day 5 gave you the rules: shared mutable state needs a happens-before edge or you have a data race. Day 10 gave you the execution machinery: threads and pools running work concurrently. Today connects them — the synchronization primitives that actually create those happens-before edges and enforce mutual exclusion. The architect’s theme: correctness is the floor, not the goal. Any lock makes code correct; the skill is making it correct and scalable — because lock contention silently destroys throughput, and the difference between a coarse lock and a well-chosen one is often 10× under load.

STAGE 1 30 MIN

synchronized & the intrinsic monitor

Every Java object has an invisible lock attached to it — its intrinsic lock or monitor. synchronized acquires it on entry and releases it on exit (normally or via exception):

synchronized (lockObject) { ... }        // explicit monitor
synchronized void m() { ... }             // locks 'this'
static synchronized void s() { ... }      // locks the Class object

Under the hood the compiler emits monitorenter / monitorexit bytecodes; the lock state lives in the object header’s mark word. (Historically the JVM optimized with biased and lightweight locking; biased locking was removed in JDK 15+, so don’t cite it as current.)

synchronized provides three things, not one. Most people name only the first.

  • Mutual exclusion — one thread in the block at a time per monitor.
  • Visibility — on release, all writes are flushed; on acquire, the next thread sees them.
  • Ordering — the unlock happens-before the next lock of the same monitor (Day 5). That edge is what makes ordinary fields written inside the block visible to the next thread.

So a thread that only reads a shared field must also synchronize (on the same monitor) to be guaranteed the latest value — exclusion on the writer alone isn’t enough; visibility needs the matching acquire.

Reentrancy

Intrinsic locks are reentrant: a thread that holds a monitor can re-acquire it without deadlocking itself (the JVM tracks a hold count, decremented on each exit). This is why a synchronized method can call another synchronized method on the same object. Without reentrancy, that would self-deadlock instantly.

Choosing the lock object

Don’t lock on something a caller can also lock (e.g. synchronized(this) on a public class, or a String literal / boxed Integer that may be interned and shared). Prefer a private final lock object: private final Object lock = new Object(); — it can’t be locked by anyone else, so you can reason about who contends for it.

🎯 INTERVIEW QUESTIONS

Q: Does synchronized only give mutual exclusion? — No. It also gives visibility and ordering via the unlock→lock happens-before edge. A reader that doesn’t synchronize can see a stale value even if the writer does.

Q: What does “reentrant” mean? — A thread can acquire a lock it already holds (hold count increments); it releases only when the count returns to zero. Without it, recursive or nested synchronized calls would deadlock.

Q: Why not synchronized(this) in a library class? — Callers can lock your instance too, causing surprise contention or deadlock. Use a private final lock object you fully control.

QUICK CHECK

Thread A increments a counter inside synchronized; Thread B reads it with a plain (unsynchronized) getter. Is B guaranteed to see A’s latest value? Why or why not — in happens-before terms?

STAGE 2 25 MIN

wait / notify — coordinating threads on a monitor

Mutual exclusion stops threads from colliding; condition signalling lets them coordinate — “wait until the queue is non-empty.” The monitor provides wait(), notify(), and notifyAll(), and they may only be called while holding that monitor.

synchronized (lock) {
    while (queue.isEmpty()) {      // MUST be a while, never an if
        lock.wait();              // releases the lock, parks, re-acquires on wake
    }
    return queue.remove();
}
// producer:
synchronized (lock) { queue.add(item); lock.notifyAll(); }

The three rules everyone gets wrong:

  1. Always wait in a while loop, never an if. On wake-up you must re-check the condition: another thread may have raced in and changed it, and spurious wakeups are permitted by the spec — wait can return without any notify.
  2. wait() atomically releases the lock and parks, then re-acquires it before returning. That release is what lets the producer get in.
  3. Prefer notifyAll() over notify() unless you can prove all waiters are interchangeable. notify() wakes one arbitrary waiter; if it’s waiting on a different condition, your needed waiter sleeps forever (a lost-wakeup / missed-signal bug).

This low-level pattern is the foundation of every blocking queue and bounded buffer — but you rarely write it by hand anymore. You reach for java.util.concurrent (Day 13’s BlockingQueue) or the Condition objects in the next stage, which make the intent explicit and the bugs harder.

🎯 INTERVIEW QUESTIONS

Q: Why must wait() be in a loop? — To re-check the condition after waking: the state may have changed before you re-acquired the lock, and spurious wakeups are allowed by the JVM spec.

Q: notify() vs notifyAll()? — notify wakes one arbitrary waiter (risk: it’s the wrong one, and the right one starves). notifyAll wakes all; they re-check and the right ones proceed. Default to notifyAll unless waiters are provably uniform.

QUICK CHECK

A teammate replaces the while guard around wait() with an if, “because it’s only woken when ready.” Name the two distinct ways this now breaks.

STAGE 3 30 MIN

ReentrantLock & Condition — explicit locks with superpowers

ReentrantLock does everything synchronized does (it’s reentrant, same memory semantics) but as an explicit object — which unlocks capabilities the keyword can’t express. The cost: you must release it yourself.

private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    ... // critical section
} finally {
    lock.unlock();   // ALWAYS in finally — or an exception leaks the lock forever
}

The try/finally rule is non-negotiable. synchronized auto-releases on any exit, including exceptions. An explicit lock does not — if the critical section throws and you didn’t unlock in finally, the lock is held forever and every other thread blocks. This is the #1 ReentrantLock bug.

Why prefer ReentrantLock over synchronized — the four superpowers

CapabilityWhat it enables
tryLock()Acquire only if free, else do something else — never block
tryLock(timeout)Give up after a bound — the primary deadlock-avoidance tool (Stage 5)
lockInterruptibly()A blocked acquirer can be interrupted/cancelled (synchronized can’t)
fairness + multiple ConditionsOptional FIFO ordering; separate wait-sets per condition

Fair vs unfair locks

new ReentrantLock(true) is fair: the longest-waiting thread acquires next (FIFO), preventing starvation. The default is unfair: a thread may “barge” ahead of the queue. Unfair is the default because it has much higher throughput — barging avoids costly thread hand-offs. Use fair only when starvation is a real, observed problem; you pay for it in throughput.

Condition — named wait-sets

final Condition notFull  = lock.newCondition();
final Condition notEmpty = lock.newCondition();
// put(): while(full) notFull.await();  ... notEmpty.signal();
// take(): while(empty) notEmpty.await(); ... notFull.signal();

A single monitor has one wait-set, so notifyAll wakes everyone, including threads waiting on an unrelated condition. Condition objects give a lock multiple, separate wait-sets, so a bounded buffer can signal “not full” producers and “not empty” consumers independently — targeted wake-ups, far less wasteful contention.

🎯 INTERVIEW QUESTIONS

Q: When prefer ReentrantLock over synchronized? — When you need tryLock/timed acquisition (deadlock avoidance), interruptible locking, fairness, or multiple condition wait-sets. Otherwise synchronized is simpler and auto-releases — prefer it by default.

Q: Fair vs unfair — default and why? — Default is unfair, for throughput (barging avoids hand-off overhead). Fair guarantees FIFO and prevents starvation but is slower. Use fair only when starvation is observed.

Q: Why are Condition objects better than one monitor’s wait-set? — Multiple wait-sets on one lock let you wake only the relevant waiters (producers vs consumers) instead of notifyAll waking everyone.

QUICK CHECK

Two services must update each other’s accounts (transfer A→B and B→A) and occasionally deadlock. Which single ReentrantLock method, used how, lets a thread back off instead of hanging?

STAGE 4 25 MIN

ReadWriteLock & StampedLock — scaling read-heavy state

A plain lock serializes everything, even reads that don’t conflict. For read-heavy data (a config, a cache, a routing table read thousands of times per write), that’s wasteful: concurrent reads are safe.

ReentrantReadWriteLock

private final ReentrantReadWriteLock rw = new ReentrantReadWriteLock();
// reads: rw.readLock().lock()  — many readers concurrently
// writes: rw.writeLock().lock() — exclusive, blocks all readers and writers
  • Multiple readers may hold the read lock simultaneously; a writer is exclusive.
  • Downgrade is allowed (acquire write, then read, then release write) — safe. Upgrade is NOT (hold read, try to acquire write) — it deadlocks, because the writer waits for readers to release, including itself.
  • Write starvation: under a constant stream of readers, a writer may never get in. Use the fair constructor if writes must make progress.

Net win only when reads greatly outnumber writes and the critical section is long enough that the read-write bookkeeping pays off. For short sections, a plain lock can actually be faster.

StampedLock (Java 8) — optimistic reads

long stamp = sl.tryOptimisticRead();   // no lock taken — just a version stamp
int x = this.x, y = this.y;          // read fields
if (!sl.validate(stamp)) {            // a writer intervened?
    stamp = sl.readLock();            // fall back to a real read lock
    try { x = this.x; y = this.y; } finally { sl.unlockRead(stamp); }
}

StampedLock adds an optimistic read: you read without acquiring anything, then validate that no write happened; only on a (rare) conflict do you fall back to a real read lock. For overwhelmingly read-heavy data this beats ReadWriteLock by avoiding even read-lock contention. The catch: it is not reentrant and not directly tied to Conditions, so it’s a specialized tool, not a drop-in.

🎯 INTERVIEW QUESTIONS

Q: Why can’t you upgrade a read lock to a write lock? — The write lock waits for all readers to release; if you hold the read lock while requesting write, you wait for yourself — deadlock. Downgrade (write→read) is fine.

Q: When does ReadWriteLock NOT help? — Short critical sections (its bookkeeping costs more than it saves) or write-heavy workloads (writers are exclusive anyway). It pays off only for read-dominated, longer sections.

Q: What does StampedLock add? — Optimistic reads: read without locking, then validate; fall back only on a real conflict. Best for very read-heavy data — but it’s non-reentrant.

STAGE 5 25 MIN

Deadlock, contention & lock granularity

Deadlock — the four conditions

A deadlock needs all four Coffman conditions to hold simultaneously; break any one and deadlock is impossible:

  1. Mutual exclusion — the resource can’t be shared.
  2. Hold and wait — a thread holds one lock while requesting another.
  3. No preemption — locks can’t be forcibly taken away.
  4. Circular wait — a cycle: A holds L1 wants L2, B holds L2 wants L1.

The two practical fixes (each breaks a condition): (1) Global lock ordering — always acquire multiple locks in the same total order (e.g. by account id), which makes a circular wait impossible. (2) Timed tryLock — if you can’t get the second lock within a timeout, release the first and retry, breaking “hold and wait.” Lock ordering is the cleaner design; timed tryLock is the robust fallback when ordering isn’t feasible.

Related hazards: livelock (threads keep responding to each other and make no progress — two people stepping aside in a hallway), and starvation (a thread never gets the lock — the fairness problem from Stage 3).

Diagnosing deadlock & contention

Tie back to Day 6: a thread dump auto-detects deadlocks (“Found one Java-level deadlock”) with the exact cycle. Contention (not deadlock) shows up as many threads BLOCKED on the same monitor, or a hot lock in a lock profile — the system runs, but slowly, because threads queue for one lock.

Granularity — the throughput lever

One coarse lock around a whole data structure is correct but serial — throughput is capped at one thread. Finer-grained locking scales: lock per-bucket or per-entry (lock striping, how the old ConcurrentHashMap worked — 16 segment locks instead of one), so independent operations proceed in parallel. The architect’s instinct: hold locks for the shortest time over the smallest scope, and prefer the lock-free java.util.concurrent collections (Days 12–13) over hand-rolled locking whenever they fit.

🎯 INTERVIEW QUESTIONS

Q: How do you prevent deadlock? — Break a Coffman condition: impose a global lock-acquisition order (kills circular wait), or use timed tryLock with back-off (kills hold-and-wait). Also: hold fewer locks, for less time.

Q: How do you tell deadlock from contention? — Deadlock: threads stuck forever, no progress; a thread dump names the cycle. Contention: progress continues but slowly; many BLOCKED threads on one hot monitor.

Q: Why do finer-grained locks scale better? — They let independent operations run in parallel instead of serializing on one lock; lock striping (per-bucket locks) is the classic example.

QUICK CHECK

Two transfer threads deadlock: one does lock(A); lock(B), the other lock(B); lock(A). Which Coffman condition does a consistent lock ordering eliminate, and how would you order the locks here?

SCENARIO · READ CAREFULLY BEFORE ANSWERING

The contended quote cache

public class QuoteCache {
    private final Map<String,Quote> map = new HashMap<>();

    public synchronized Quote get(String sym) { return map.get(sym); }
    public synchronized void put(String sym, Quote q) { map.put(sym, q); }
    public synchronized int size() { return map.size(); }
}

Reads (get) outnumber writes (put) ~10,000 : 1. Under market load, profiling shows threads piling up BLOCKED on this object’s monitor, and throughput is capped far below CPU capacity.

MODEL ANSWER

Every method is synchronized on the same monitor (the QuoteCache instance), so the lock serializes all access — including the 10,000 reads that don’t conflict with each other at all. Only one thread can be inside any method at a time, so concurrent gets queue behind one another even though reading a HashMap concurrently would be safe if nobody were writing. That queueing is exactly the BLOCKED pile-up in the profile: threads waiting to acquire the one monitor. Throughput is capped at “one read at a time,” far below what the cores could do — classic lock contention from coarse granularity, not a CPU or GC problem.

Best first choice: replace the whole thing with ConcurrentHashMap and drop every synchronized — it gives lock-free reads, fine-grained (striped) write locking, and a correct concurrent size(), with zero hand-rolled locking (Days 12–13). That’s the architect’s answer: prefer the proven concurrent collection over rolling your own. If you must keep a plain map (e.g. custom eviction logic), use a ReentrantReadWriteLock: get/size under readLock (many concurrent readers), put under writeLock (exclusive) — correctness rule: every access, including reads, must go through the lock, or readers lose the visibility guarantee. With reads at 10,000:1 and a read-heavy access pattern, a StampedLock with optimistic reads would scale even better, at the cost of more complex, non-reentrant code. Order of preference: ConcurrentHashMap > StampedLock > ReadWriteLock > the original.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • synchronized gives three things — exclusion, visibility, ordering — via the unlock→lock happens-before edge. Readers must synchronize too. It’s reentrant; lock on a private final object.
  • wait in a while loop (spurious wakeups); prefer notifyAll. You rarely hand-roll this — use j.u.c.
  • ReentrantLock adds tryLock/timeout, interruptible lock, fairness, and multiple Condition wait-sets — but you must unlock in finally. Default is unfair (throughput).
  • Read-heavy? ReadWriteLock (no upgrade!) or StampedLock (optimistic reads, non-reentrant). Often ConcurrentHashMap beats both.
  • Deadlock needs all four Coffman conditions; break one with lock ordering or timed tryLock. Finer granularity scales; coarse locks silently cap throughput.
