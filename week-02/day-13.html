CompletableFuture & Async Composition From callback hell to composable pipelines — how to chain, fan-out, combine, and handle errors in asynchronous Java without blocking a thread.

SCOPE · 60 min · 5 stages — the async composition layer that sits between raw threads (Days 10–12) and full reactive (Days 20–21).
RECAP FROM DAY 12

Days 10–12 gave you the machinery: thread pools for managing OS threads, locks for mutual exclusion, and CAS atomics for contention-free updates on single variables. All of that is low-level — you controlled threads directly. Today lifts the abstraction. CompletableFuture lets you describe what to compute and how results feed into each other, without owning threads yourself. It is the foundation of composable async in Java — used inside Spring WebClient, Kafka clients, AWS SDK v2, and every reactive adapter in the ecosystem.

STAGE 1 12 MIN

Why CompletableFuture? The limits of plain Future

Java’s original Future<T> (Java 5) had one job: represent a computation happening on another thread. But it had three fatal flaws for composable async work:

  • Blocking get(). To use a result you had to call future.get(), which blocks the calling thread — defeating the purpose of async.
  • No chaining. You couldn’t say “when this finishes, do that” without blocking first. Every pipeline step required a thread stall.
  • No exception composability. Errors had to be caught at the get() call; there was no way to handle them mid-pipeline.

CompletableFuture (Java 8) implements both Future<T> and CompletionStage<T>. The CompletionStage interface is the key: it is a node in a computation pipeline that can be chained, combined, and error-handled without ever blocking the calling thread. Think of it as a promise that knows its downstream continuations.

Starting a CompletableFuture

// supplyAsync — async computation that returns a value
CompletableFuture<OptionChain> chainFuture =
    CompletableFuture.supplyAsync(() -> kiteClient.getOptionChain("NIFTY"));

// runAsync — async side-effect that returns void
CompletableFuture<Void> logFuture =
    CompletableFuture.runAsync(() -> auditLog.record("chain-fetched"));

// completedFuture — already-done value; great for tests and cache hits
CompletableFuture<Price> cached =
    CompletableFuture.completedFuture(priceCache.get(symbol));

// failedFuture (Java 9+) — already-failed; for error injection in tests
CompletableFuture<Price> bad =
    CompletableFuture.failedFuture(new MarketClosedException());

The hidden default thread pool: ForkJoinPool.commonPool()

When you call supplyAsync(fn) with no executor argument, the task runs on the JVM-wide ForkJoinPool.commonPool() — the same pool shared with parallel streams (Day 7). Its size defaults to Runtime.getRuntime().availableProcessors() - 1. Starve this pool in one place (e.g., a slow blocking HTTP call) and every parallel stream and un-customized CompletableFuture in the JVM queues behind it. Always pass a dedicated executor for production async work.

// The right way — own your thread pool
ExecutorService ioPool = Executors.newFixedThreadPool(20,
    new ThreadFactoryBuilder().setNameFormat("kite-io-%d").build());

CompletableFuture<OptionChain> cf =
    CompletableFuture.supplyAsync(() -> kiteClient.getOptionChain("NIFTY"), ioPool);
🎯 INTERVIEW QUESTIONS

Q: Why replace Future with CompletableFuture? — Future.get() blocks; CompletableFuture provides non-blocking chaining via CompletionStage, exception handling mid-pipeline, and fan-out/fan-in combinators without ever parking the calling thread.

Q: What thread pool does supplyAsync use by default, and why is that a problem? — ForkJoinPool.commonPool() — a JVM-global pool shared with parallel streams. Blocking calls (I/O, sleep) inside unguarded supplyAsync tasks can starve all parallel stream usage. Always pass a dedicated executor for I/O-bound work.

STAGE 2 12 MIN

Pipeline operators — transform, consume, run

Once you have a CompletableFuture, you describe what to do with its result using continuation operators. The three families:

// thenApply  — transform T → U  (like Stream.map)
CompletableFuture<List<Strike>> strikes = chainFuture
    .thenApply(chain -> chain.filteredByDelta(0.3, 0.5));

// thenAccept — consume T, return Void  (like Stream.forEach)
strikes.thenAccept(list -> dashboard.render(list));

// thenRun    — no input, no output, just a side-effect after completion
strikes.thenRun(() -> metrics.increment("strikes.fetched"));

Sync vs async variants — which thread runs the continuation?

Every operator has two variants: the plain form and an Async suffix form. This is the most common CF pitfall in production code:

thenApply — continuation runs on whichever thread completed the prior stage thenApplyAsync — continuation dispatched to executor (default: commonPool) 
// Dangerous: thenApply with a blocking HTTP call
chainFuture
    .thenApply(chain -> kiteClient.getLTP(chain.atmStrike()))  // blocks IO pool thread!
    .thenApply(price -> price * 1.02);

// Safe: thenApplyAsync so the blocking call frees the prior pool thread
chainFuture
    .thenApplyAsync(chain -> kiteClient.getLTP(chain.atmStrike()), ioPool)
    .thenApply(price -> price * 1.02);  // cheap math: plain thenApply is fine
Architect lens — the async suffix rule

Use thenApply (no suffix) only for cheap, non-blocking transformations: mapping a field, formatting a string, filtering a list. The moment a continuation does I/O, blocks on a lock, or calls another service — use thenApplyAsync with an explicit executor. Failure to do this creates a hidden coupling: a slow downstream call monopolizes the thread that completed the upstream stage, stalling anything else waiting on that thread.

This pattern is identical to the Day 7 parallel-stream trap: the danger is not the CF itself but unintended thread-pool sharing.

🎯 INTERVIEW QUESTIONS

Q: What is the difference between thenApply and thenApplyAsync? — thenApply runs the function on whichever thread completed the prior stage (inline). thenApplyAsync submits the function to a thread pool (default: commonPool, or a provided executor). Use thenApply for fast pure transforms; thenApplyAsync for anything that may block.

Q: I used thenApply for a database call and it works fine in tests but degrades in production. Why? — In tests there is no concurrency; the continuation runs fine inline. Under production load, the completing thread (e.g., an IO pool worker) is held by the blocking DB call, unable to pick up new tasks — pool starvation under contention.

STAGE 3 12 MIN

Composition & fan-out — thenCompose, thenCombine, allOf, anyOf

Chaining transformations is easy; the harder problems are sequential dependency (A then B where B uses A’s output) and parallel fan-out (start A and B concurrently, combine when both finish).

thenCompose — sequential dependency (flatMap for CF)

If the continuation itself returns a CompletableFuture, use thenCompose, not thenApply. thenApply(fn) where fn returns a CF gives you a CF<CF<T>> — a nested future you have to unwrap. thenCompose flattens it automatically, exactly like flatMap on a stream.

// thenApply mistake: produces CF<CF<Order>>
CompletableFuture<CompletableFuture<Order>> nested =
    chainFuture.thenApply(chain -> placeOrderAsync(chain.bestStrike()));

// thenCompose correct: produces CF<Order>  (flatMap — unwraps one level)
CompletableFuture<Order> orderFuture =
    chainFuture.thenCompose(chain -> placeOrderAsync(chain.bestStrike()));

thenCombine — zip two independent CFs

// Fetch spot and IV in parallel — combine when both land
CompletableFuture<Double> spotFuture = CompletableFuture.supplyAsync(this::fetchSpot, ioPool);
CompletableFuture<Double> ivFuture   = CompletableFuture.supplyAsync(this::fetchIV, ioPool);

CompletableFuture<Greeks> greeksFuture =
    spotFuture.thenCombine(ivFuture, (spot, iv) -> blackScholes(spot, iv, strike, expiry));

allOf & anyOf — fan-in from N futures

List<CompletableFuture<Quote>> quotes = symbols.stream()
    .map(s -> CompletableFuture.supplyAsync(() -> fetchQuote(s), ioPool))
    .collect(Collectors.toList());

// allOf: wait for ALL to complete (returns CF<Void> — must join manually)
CompletableFuture<Void> allDone = CompletableFuture.allOf(
    quotes.toArray(new CompletableFuture[0]));

CompletableFuture<List<Quote>> allQuotes = allDone.thenApply(v ->
    quotes.stream().map(CompletableFuture::join).collect(Collectors.toList()));

// anyOf: resolve as soon as ANY completes (returns CF<Object> — cast required)
CompletableFuture<Object> fastest = CompletableFuture.anyOf(
    quotes.toArray(new CompletableFuture[0]));

allOf returns CF<Void> — not the values. This surprises everyone the first time. After allOf() completes, you must manually call .join() (not .get()) on each individual future to extract its result — .join() is identical to .get() but throws unchecked exceptions. The pattern is: allOf(futures).thenApply(v -> futures.stream().map(CF::join).collect(...)).

🎯 INTERVIEW QUESTIONS

Q: Chain 3 async API calls where the second depends on the first. — supplyAsync(call1).thenCompose(r1 -> supplyAsync(() -> call2(r1))).thenCompose(r2 -> supplyAsync(() -> call3(r2))). Use thenCompose (not thenApply) because each continuation returns a CF — thenApply would produce CF<CF<T>>.

Q: thenApply vs thenCompose? — thenApply: fn returns a plain value (T → U). thenCompose: fn returns a CF (T → CF<U>) and flattens it to CF<U>. Same as map vs flatMap in streams. Use thenCompose whenever the continuation itself kicks off another async operation.

Q: What does allOf return and how do you get the results? — CF<Void> — no results embedded. After allOf completes, call .join() on each individual CF in a stream to collect results. Alternatively, collect into a CF<List<T>> via allOf(...).thenApply(v -> futures.stream().map(CF::join).toList()).

STAGE 4 12 MIN

Exception handling & timeouts

A CF pipeline without error handling is incomplete. Exceptions propagate through the chain: if any stage throws, all downstream stages are skipped and the final CF completes exceptionally. Three operators let you intercept errors at different points in the pipeline:

The three exception handlers

// exceptionally — recover from exception, supply a fallback value
// Runs ONLY on failure; success skips it entirely
chainFuture
    .thenApply(chain -> chain.atmStrike())
    .exceptionally(ex -> {
        log.warn("Chain fetch failed: {}", ex.getMessage());
        return FALLBACK_STRIKE;   // must return same type T
    });

// handle — runs on BOTH success and failure; gives you the value OR the exception
// (whichever is non-null — the other is null)
chainFuture.handle((chain, ex) -> {
    if (ex != null) return OptionChain.empty();
    return chain;
});

// whenComplete — observe result OR exception, cannot change the pipeline value
// Good for logging/metrics on every outcome
chainFuture.whenComplete((chain, ex) -> {
    if (ex != null) metrics.increment("chain.error");
    else           metrics.increment("chain.success");
});
OperatorRuns onCan change value?Use for
exceptionallyFailure onlyYes (supply fallback)Fallbacks, defaults
handleSuccess OR failureYes (both paths)Unified transformation + recovery
whenCompleteSuccess OR failureNo (observer only)Logging, metrics, cleanup

Timeouts (Java 9+)

// orTimeout — complete exceptionally with TimeoutException after N units
chainFuture
    .orTimeout(500, TimeUnit.MILLISECONDS)
    .exceptionally(ex -> OptionChain.empty());

// completeOnTimeout — complete with a fallback VALUE (not exception) after N units
chainFuture
    .completeOnTimeout(OptionChain.empty(), 500, TimeUnit.MILLISECONDS);
Architect lens — always set a timeout; never let a CF hang forever

A CompletableFuture that never completes holds its downstream chain open indefinitely. In a trading system fetching option chains from Kite Connect, a network partition can stall the socket without throwing. Without orTimeout(), every stage downstream waits forever, threads piled in the pool cannot be freed, and the service quietly deadlocks under load while appearing “up” to health checks. The rule: every CF that touches an external system gets a timeout.

Prefer completeOnTimeout(fallback, ...) over orTimeout(...).exceptionally(...) when you have a safe default — it is cleaner and avoids the exception-handling overhead. Use orTimeout when you want the timeout to be visible as an error and trigger a different code path.

🎯 INTERVIEW QUESTIONS

Q: How does exception propagation work in a CF chain? — If any stage throws, its downstream thenApply/thenCompose stages are bypassed and the exception propagates to the final CF. An exceptionally/handle at the end (or at any intermediate point) can intercept it. Without a handler, the exception surfaces when you call .get()/.join().

Q: Difference between handle and exceptionally? — exceptionally runs only on failure and must return a recovery value of type T. handle runs on both success and failure, receives (T result, Throwable ex) (one null), and can map to any value — equivalent to a finally-with-transform. Use handle when you need a unified transformation of both paths.

Q: How do you add a timeout to a CompletableFuture (Java 9+)? — .orTimeout(ms, MILLISECONDS) fails exceptionally with TimeoutException; .completeOnTimeout(fallback, ms, MILLISECONDS) completes with a default value. Both avoid hanging the chain. Always use one for any CF touching an external system.

A Kite Connect order pipeline

You have three async helper methods, each returning a CompletableFuture:

CompletableFuture<OptionChain> fetchChain(String symbol);          // ~200ms I/O
CompletableFuture<Strike>      selectBestStrike(OptionChain c);  // ~5ms CPU
CompletableFuture<OrderId>    placeOrder(Strike s);             // ~150ms I/O

A junior developer wrote this blocking cascade to sequence them:

// ❌ Callback hell / blocking cascade — do NOT write code like this
try {
    OptionChain chain = fetchChain("NIFTY").get();          // blocks thread 350ms
    Strike      best  = selectBestStrike(chain).get();       // blocks again
    OrderId     id    = placeOrder(best).get();              // blocks again
    System.out.println("Placed: " + id);
} catch (Exception e) { log.error(e); }

Additionally, for a separate risk check, you need to fetch spotPrice and impliedVolatility in parallel (independent calls), then combine them to compute Greeks — and the whole thing must timeout in 600 ms with a fallback of Greeks.ZERO.

YOUR ANSWERS · MINIMUM 50 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 50 characters
MODEL ANSWER

Use thenCompose (not thenApply) at each step because every continuation returns a CompletableFuture — thenApply would produce CF<CF<T>>, requiring manual unwrapping. Add orTimeout and whenComplete for observability:

fetchChain("NIFTY")                           // CF<OptionChain>
    .thenCompose(chain -> selectBestStrike(chain))  // CF<Strike>
    .thenCompose(strike -> placeOrder(strike))       // CF<OrderId>
    .orTimeout(600, TimeUnit.MILLISECONDS)
    .whenComplete((id, ex) -> {
        if (ex != null) log.error("Order pipeline failed", ex);
        else           log.info("Placed: {}", id);
    });

Why thenCompose: each helper already kicks off async I/O and returns a CF. thenCompose flat-maps (unwraps) the nested CF, giving a single-level chain. The original .get() calls blocked the calling thread for ~350 ms total; the CF chain releases every thread as soon as its stage completes. whenComplete logs both outcomes without altering the result (it passes the value/exception through unchanged).

MODEL ANSWER

Without an explicit executor, all supplyAsync calls (and any thenApplyAsync calls) use ForkJoinPool.commonPool(). Its size is Runtime.availableProcessors() - 1 — typically 3–7 threads. During market open, the order pipeline and the risk check plus any parallel streams in the JVM all queue on those same few threads. A slow Kite Connect response (network jitter is common in Indian retail brokers during 9:15 am open) holds a commonPool thread, and the risk check — on the same pool — queues behind it. Both pipelines may breach their 600 ms timeout simultaneously.

Fix: dedicate a named ExecutorService per concern:

private static final ExecutorService kiteIoPool =
    Executors.newFixedThreadPool(10,
        r -> { Thread t = new Thread(r, "kite-io"); t.setDaemon(true); return t; });

Pass kiteIoPool to every supplyAsync and thenApplyAsync. Now Kite I/O contention does not bleed into parallel streams, and you can tune each pool independently (e.g., bump kite-io during market open, shrink at close).

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • CompletableFuture = Future + CompletionStage. The CompletionStage interface enables non-blocking chaining. Always prefer CF over raw Future for any composed async work.
  • Default pool trap. supplyAsync(fn) without an executor uses ForkJoinPool.commonPool() — the same JVM-global pool as parallel streams. Pass a dedicated executor for all production I/O async tasks.
  • Plain vs Async suffix. thenApply runs on the completing thread (fine for cheap transforms). thenApplyAsync dispatches to an executor. Never put blocking calls inside plain thenApply.
  • thenCompose vs thenApply. If the continuation returns a CF, use thenCompose (flatMap). thenApply nests to CF<CF<T>> — a type-system hint you used the wrong one.
  • Fan-out pattern. Start independent CFs immediately; combine with thenCombine (two) or allOf().thenApply(v -> futures.stream().map(CF::join)...) (N). allOf returns CF<Void> — you must extract values manually.
  • Exception operators. exceptionally for recovery fallbacks; handle for unified success+failure transform; whenComplete for side-effect observation. Always add a timeout (orTimeout or completeOnTimeout) for external calls.
