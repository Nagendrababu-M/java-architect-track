STAGE 1 25 MIN

┃ JAVA 8 (2014, LTS) — LAMBDAS & METHOD REFERENCES

Behavior as data — and what the JVM really does with it

Functional interfaces (SAM types)

A functional interface has exactly one abstract method — a SAM (Single Abstract Method) type. default and static methods don’t count, and neither do abstract methods that override Object’s public methods (which is why Comparator can declare equals and stay functional). The @FunctionalInterface annotation changes nothing at runtime; it makes the compiler fail if a second abstract method is ever added — a contract annotation like @Override.

The java.util.function package ships the core four, plus ~40 primitive specializations to avoid autoboxing:

InterfaceSignatureUsed by
Supplier<T>() → Tlazy values, factories
Consumer<T>T → voidforEach, side effects
Function<T,R>T → Rmap, transforms
Predicate<T>T → booleanfilter, validation

The primitive cousins (IntFunction, ToLongFunction, DoublePredicate…) exist purely to dodge boxing: Stream<Integer>.map(i -> i*2) boxes and unboxes on every element; IntStream.map works on raw ints. Over millions of elements that’s the difference between memory-bound and CPU-bound.

Lambdas & the four kinds of method reference

// Lambda
Comparator<Trade> c = (a, b) -> a.price().compareTo(b.price());

// 1. Static            2. Bound instance       3. Unbound instance      4. Constructor
Integer::parseInt;      myString::length;       String::length;          ArrayList::new;

The distinction interviewers test is bound vs unbound: myString::length is bound to a specific object and takes no args; String::length is unbound — the receiver becomes the first parameter, so it works as a Function<String,Integer>.

Capture semantics: effectively final

A lambda may only capture locals that are effectively final (never reassigned). Captured locals are copied into the lambda at creation; permitting reassignment would let the copy and the original silently diverge, so Java forbids it. Fields and statics are not captured — they’re read live through this — which is why they needn’t be final, and why mutating a field from a lambda inside a parallel stream is a data race.

Lambda internals: a lambda is NOT an anonymous class.

The compiler does not emit a class file per lambda. The body becomes a private static method, and the call site is an invokedynamic instruction. On first execution, that instruction calls a bootstrap method — LambdaMetafactory — which spins a small implementation class at runtime and links the call site; subsequent calls hit the linked site directly. Consequences worth stating in an interview: non-capturing lambdas are cached singletons (x -> x*2 is allocated once, ever); capturing lambdas allocate per evaluation (they must carry the captured values); and because the strategy lives in the JVM, the platform can optimize lambdas without recompiling your code. An anonymous class, by contrast, is a fixed .class file with an allocation on every new.

🎯 INTERVIEW QUESTIONS

Q: What does invokedynamic do for lambdas? — Defers lambda-class creation to runtime via LambdaMetafactory: no compile-time class-per-lambda, singleton caching of non-capturing lambdas, and JVM freedom to evolve the strategy.

Q: When does a lambda allocate? — Only if it captures. Non-capturing = cached singleton; capturing = an object per evaluation, because it carries its captured state.

Q: Why must captured locals be effectively final? — They’re copied by value; allowing reassignment would let the copy and the variable diverge, so the language bans it.

STAGE 2 25 MIN

┃ JAVA 8 (2014, LTS) — STREAM API & ITS INTERNALS

Declarative pipelines, and how they actually execute

The shape of a pipeline

Map<String, BigDecimal> notional = trades.stream()   // source
    .filter(t -> t.side() == Side.BUY)              // intermediate (lazy)
    .filter(t -> t.qty() > 10_000)                  // intermediate (lazy)
    .collect(groupingBy(Trade::symbol,                 // terminal (triggers)
             reducing(BigDecimal.ZERO, Trade::value, BigDecimal::add)));

Intermediate operations do nothing. They build a linked chain of stages. All work happens when the terminal operation pulls.

When collect() runs, the pipeline is fused: each element flows through all stages one at a time — filter, then map, then accumulate — before the next element starts. There is no intermediate collection between stages. One pass, no materialization, and short-circuit ops (findFirst, anyMatch, limit) can stop pulling the instant they’re satisfied — which is why a filtered findFirst terminates on an infinite stream.

Spliterators — the engine underneath

Every source provides a Spliterator (“splittable iterator”). tryAdvance(consumer) pushes one element through the pipeline (sequential traversal); trySplit() carves off a chunk for another thread (parallel decomposition). It reports characteristics — SIZED, SUBSIZED, ORDERED, SORTED, DISTINCT, IMMUTABLE — that let the pipeline skip work: distinct() on a DISTINCT source is a no-op; toArray() on a SIZED source preallocates exactly.

Stateless vs stateful; one-pass & non-reusable

  • Stateless intermediate (filter, map, flatMap): each element independent — fuses and parallelizes perfectly.
  • Stateful intermediate (sorted, distinct, limit): must see some/all elements before emitting — sorted buffers the entire stream, breaking the one-pass model and crushing parallel scalability.
  • One-shot: a stream is a view over a spliterator, not a data structure. The terminal op consumes it; a second terminal op throws IllegalStateException. Re-create from the source, or hold a Supplier<Stream>.

map vs flatMap

stream.map(Order::status)                // 1 → 1
stream.flatMap(o -> o.lines().stream())  // 1 → many, flattening nested streams
intStream.mapToObj(Integer::toString) // primitive stream → object stream
🎯 INTERVIEW QUESTIONS

Q: Why are streams not reusable? — A stream is a one-time view over a spliterator; the terminal op consumes it. Reuse throws IllegalStateException.

Q: Does peek always run? — No. Laziness + short-circuiting means un-pulled elements are never peeked, and ops can be elided entirely (count() on a SIZED source may skip traversal). Never use peek for required side effects.

Q: Stream API vs imperative loops — trade-offs? — Streams: declarative, composable, parallelizable, no index bugs; cost: harder step-debugging, overhead on tiny collections, awkward early-exit-with-side-effects. A plain loop is still clearer for genuinely imperative work.

STAGE 3 25 MIN

┃ JAVA 8 (2014, LTS) — COLLECTORS & PARALLEL STREAMS

Mutable reduction, composition, and the parallel trap

grouping, partitioning & downstream composition

trades.stream().collect(groupingBy(Trade::symbol, counting()));         // count per symbol
trades.stream().collect(groupingBy(Trade::symbol, groupingBy(Trade::side))); // 2-level
trades.stream().collect(partitioningBy(t -> t.qty() > 10_000));            // boolean split
trades.stream().collect(joining(", ", "[", "]"));                          // string build

The 5-function Collector contract

Collector.of(
    () -> new StringJoiner(", "),   // 1. supplier — mutable container
    (j, s) -> j.add(s),               // 2. accumulator — fold one element in
    (j1, j2) -> j1.merge(j2),         // 3. combiner — merge two containers (PARALLEL)
    StringJoiner::toString            // 4. finisher — container → result
    // 5. characteristics — IDENTITY_FINISH, CONCURRENT, UNORDERED
);

The interview point: the combiner is load-bearing. Sequential streams never call it, so a broken combiner “works” in every test, then silently produces garbage under parallelStream().

reduce vs collect. reduce is immutable accumulation — each step produces a new value (right for ints, O(n²) for strings because every concat copies). collect is mutable reduction — one container per thread, mutated in place, merged. Folding strings with reduce(String::concat) is quadratic; joining() is linear.

Parallel streams — how they run, and the common-pool trap

The spliterator trySplit()s the source into chunks, chunks run as ForkJoin tasks, results merge via the collector’s combiner. Critically, those tasks run on ForkJoinPool.commonPool() — one pool, shared JVM-wide, sized cores−1. Every parallel stream and every default CompletableFuture async shares it; one parallel stream doing blocking I/O starves every other parallel computation in the process.

Reach for parallel only if all hold: large N × per-element cost; well-splitting source (ArrayList/array, not LinkedList/iterate); stateless, non-blocking ops; cheap merge; no encounter-order dependence.

// BROKEN under parallel — ArrayList isn't thread-safe
List<String> out = new ArrayList<>();
data.parallelStream().forEach(out::add);          // ✗ data race
List<String> out = data.parallelStream().collect(toList()); // ✓ collector owns state
🎯 INTERVIEW QUESTIONS

Q: toMap throws on duplicate keys — fix? — The 3-arg form with a merge function: toMap(Trade::symbol, Trade::price, (a, b) -> b). Forgetting it is a top-5 production stream bug.

Q: Why was my parallel stream slower than sequential? — Too few/cheap elements, badly-splitting source, a stateful op serializing the pipeline, expensive merge, or common-pool contention from elsewhere in the JVM.

STAGE 4 25 MIN

┃ JAVA 8 (2014, LTS) — OPTIONAL, INTERFACE EVOLUTION, java.time, CompletableFuture

The rest of the Java 8 toolkit

Optional<T>

// The signature itself says "absence is a normal outcome"
Optional<User> findUser(String id);
String name = findUser(id).map(User::name)
                          .filter(n -> !n.isBlank())
                          .orElseGet(() -> expensiveDefault());  // lazy fallback
Anti-patternWhy wrongInstead
if (o.isPresent()) o.get()Recreates the null check; get() can throwmap/ifPresent/orElseThrow
Optional as a fieldNot Serializable; box per read; the Optional itself can be nullNullable field, Optional-returning getter

The one-line philosophy: Optional is a return type — a contract that absence is normal — not a general null replacement.

Default & static interface methods

Interfaces can finally carry implementations — which is how Streams were retrofitted onto every existing collection without breaking implementors:

public interface Collection<E> {
    default Stream<E> stream() { return StreamSupport.stream(spliterator(), false); }
}

The trade-off to know: a class implementing two interfaces with the same default method must override it to resolve the conflict (the “diamond”), and a class’s own method always wins over an inherited default.

java.time (JSR-310)

LocalDate expiry  = LocalDate.of(2026, Month.JUNE, 25);
ZonedDateTime open = expiry.atTime(9, 15).atZone(ZoneId.of("Asia/Kolkata"));
Duration left   = Duration.between(Instant.now(), open.toInstant());

The fix over Date/Calendar: every type is immutable and thread-safe, so instances can be shared freely. Instant is a machine timestamp (UTC); LocalDateTime has no zone; ZonedDateTime binds a zone — knowing which to store is an interview favorite (store Instant for events, LocalDate for birthdays).

CompletableFuture

CompletableFuture.supplyAsync(() -> fetchSpot(sym))
    .thenCombine(supplyAsync(() -> fetchVol(sym)), (spot, vol) -> price(spot, vol))
    .exceptionally(ex -> fallbackPrice(sym))
    .thenAccept(quoteCache::put);

Composable async: thenApply (transform), thenCompose (flatMap for futures), thenCombine (join two), exceptionally/handle (recover). Caveat to state: the *Async variants without an explicit executor run on the common pool — the same trap as parallel streams.

🎯 INTERVIEW QUESTIONS

Q: When should Optional NOT be used? — As a field, method parameter, or collection element; and never get() without checking. It’s a return-type vehicle for “no result.”

Q: orElse vs orElseGet? — orElse(x) always evaluates x, even when the value is present; orElseGet(supplier) only runs on empty. Use orElseGet when the fallback is expensive.

Q: thenApply vs thenCompose? — thenApply is map (returns a value); thenCompose is flatMap (returns another future) — use it to avoid CompletableFuture<CompletableFuture<T>>.

STAGE 5 20 MIN

┃ JAVA 11 (2018, LTS) — LANGUAGE & PLATFORM (bundling 9 & 10)

The cleanup LTS — var, HttpClient, modules, conveniences

var — local-variable type inference (10)

var bySymbol = new HashMap<String, List<Trade>>();   // inferred at COMPILE time
for (var e : bySymbol.entrySet()) { ... }
(@Nonnull var a, @Nonnull var b) -> a.merge(b)             // var in lambda params (11)

It is inference, not dynamic typing — the variable has a fixed static type and the bytecode is identical. Locals only: never fields, parameters, or return types. The judgment rule: var should aid readability, not hide the type from a reviewer (var result = service.process() is worse than the explicit type).

Standard HttpClient (11)

HttpClient client = HttpClient.newBuilder().version(Version.HTTP_2).build();
HttpRequest req = HttpRequest.newBuilder(URI.create(url))
                             .timeout(Duration.ofSeconds(2)).build();
HttpResponse<String> r = client.send(req, BodyHandlers.ofString());   // sync
client.sendAsync(req, BodyHandlers.ofString())                       // async, composes with CF
      .thenApply(HttpResponse::body).thenAccept(cache::update);

HTTP/2, WebSocket, and async built in — removed the mandatory Apache HttpClient dependency from most services.

Module system (JPMS, 9) & single-file execution (11)

module com.trading.engine {
    requires com.trading.marketdata;
    exports com.trading.engine.api;   // everything else invisible to consumers
}
// 11: run a single source file with no javac step
$ java TradeReport.java

Brief by design — most applications stay on the classpath. The lasting impact is that the JDK itself is modularized, which later enabled jlink custom runtimes and the lockdown of sun.misc.* internals.

String, File & collection conveniences (9–11)

List<String> syms = List.of("NIFTY", "BANKNIFTY");   // (9) immutable, null-rejecting
"  x  ".strip();  "".isBlank();  "a\nb".lines();  "-".repeat(40);  // (11)
String cfg = Files.readString(Path.of("app.conf"));            // (11) whole file
🎯 INTERVIEW QUESTIONS

Q: Is var dynamic typing? — No. Compile-time inference; fixed static type; identical bytecode. Locals only.

Q: We’re on Java 8 — what does moving to 11 buy us, and what breaks? — Buys: free production JFR, the standard HttpClient, var, collection factories, the stream/Optional gap-fillers (Stage 6). Breaks: JAXB/JAX-WS/CORBA removed (re-add as Maven deps); reflection into JDK internals starts warning.

STAGE 6 20 MIN

┃ JAVA 11 (2018, LTS) — STREAM, OPTIONAL & COLLECTOR GAP-FILLERS (9–11)

The functional API grows up

Java 8 shipped the functional model with real gaps — you couldn’t conditionally stop a stream, Stream.iterate was infinite-only, and bridging an Optional into a stream was ugly. The 9–11 releases (delivered to most fleets as Java 11) filled them.

takeWhile / dropWhile & bounded iterate (9)

// Ordered, conditional truncation — stops at the FIRST failing element
prices.stream().takeWhile(p -> p < limit);
prices.stream().dropWhile(p -> p < limit);   // skip until first passing

// 3-arg iterate: a for-loop as a stream, finally bounded
Stream.iterate(1, i -> i <= 100, i -> i * 2).forEach(System.out::println);

// Together: the pagination pattern — lazy pages until an empty one comes back
Stream.iterate(fetchPage(0), pg -> fetchPage(pg.number() + 1))
      .takeWhile(pg -> !pg.items().isEmpty())
      .flatMap(pg -> pg.items().stream());

Optional grows up (9–11)

repo.findById(id)
    .or(() -> cache.find(id))                // (9) lazy alternative source
    .ifPresentOrElse(this::audit,            // (9) both branches
                     () -> log.warn("missing"));

ids.stream().map(repo::findById)
   .flatMap(Optional::stream)                // (9) the bridge — Java 8's monstrosity, gone
   .toList();

if (result.isEmpty()) { ... }                // (11) the readable negative

Before the stream() bridge, turning an Optional into a 0-or-1 stream required opt.map(Stream::of).orElseGet(Stream::empty). Now flatMap(Optional::stream) over a stream of Optionals keeps the present values and drops the empties in one step.

New downstream collectors & small gems (9–11)

// filtering (9) as downstream — keeps EMPTY groups, unlike pre-filtering
trades.stream().collect(groupingBy(Trade::symbol,
       filtering(t -> t.qty() > 10_000, toList())));

// flatMapping (9) — flatten inside a group
orders.stream().collect(groupingBy(Order::customer,
       flatMapping(o -> o.lines().stream(), toList())));

stream.filter(Predicate.not(String::isBlank));    // (11) negate a method ref
Files.lines(path).filter(l -> l.contains("ERROR")); // lazy file processing

The filtering vs pre-filter distinction is a classic probe: .filter(...).collect(groupingBy(...)) drops a key entirely if nothing passes; groupingBy(..., filtering(...)) keeps the key with an empty list. Different report shapes.

🎯 INTERVIEW QUESTIONS

Q: How did Optional change after Java 8? — 9 added or(), ifPresentOrElse(), and stream(); 10 added no-arg orElseThrow(); 11 added isEmpty(). A candidate who only knows isPresent/get learned Optional in 2014 and stopped.

Q: takeWhile vs filter? — filter tests every element; takeWhile stops at the first failure (order-sensitive). On [1,2,5,1] with < 3: filter → [1,2,1], takeWhile → [1,2].

STAGE 7 15 MIN

The challenge — a Java 8 internals incident

A real on-call scenario built from today’s Java 8 internals. Answer all three parts (50+ characters each), then reveal the model answers.

SCENARIO · READ CAREFULLY BEFORE ANSWERING

The 504s under load

System: a Spring Boot service. A controller handler runs a CPU-light aggregation using list.parallelStream().map(...).collect(...) over a few hundred items per request.

Symptom: under production load the service randomly returns 504 Gateway Timeout. CPU is not pegged. It works perfectly in load tests that hit one endpoint at a time. The same code with a sequential stream never times out.

YOUR ANSWERS · MINIMUM 20 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 20 characters
MODEL ANSWER

Root cause: common ForkJoinPool starvation. parallelStream() submits its tasks to ForkJoinPool.commonPool(), shared across the whole JVM and sized cores−1 (often 3–7 threads). Each in-flight request consumes pool threads for its pipeline. Under concurrent load, many requests compete for that tiny shared pool; requests queue waiting for a thread, latency climbs past the gateway timeout, and you get 504s — while CPU isn’t pegged because threads are waiting, not computing. Single-endpoint tests never contend, so it doesn’t reproduce there. (Any blocking I/O in the map would make it far worse, since blocked threads hold pool slots without doing CPU work.)

0 / 20 characters
MODEL ANSWER

Submit the pipeline from inside a dedicated ForkJoinPool: var pool = new ForkJoinPool(8); pool.submit(() -> list.parallelStream().map(...).collect(...)).get(); A parallel stream run from within a custom pool uses that pool’s threads, not the common pool — so this request can no longer starve unrelated work or be starved by it. Caveats: it relies on documented-by-behavior (not formally specified); you must size and lifecycle-manage the pool (a managed singleton, never one-per-request); and you’ve only contained contention — an undersized dedicated pool just starves within itself. Size for actual concurrency × per-request parallelism.

0 / 20 characters
MODEL ANSWER

Use a sequential stream: list.stream().map(...).collect(...). The workload fails the checklist: only a few hundred elements (N too small), CPU-light per element (Q too small), so fork/join orchestration overhead likely exceeded any speedup even before the contention. And in a web handler the server already provides parallelism by running many requests at once; parallelizing within each request just multiplies pressure on shared threads. Sequential makes each request’s latency predictable and independent — usually right for request-scoped work. Parallel streams are justified for batch/CPU-bound work: large N (100k+), expensive per element, well-splitting source, stateless ops, cheap merge, in a context you control (a background job) — ideally on a dedicated pool.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Java 8 is lambdas (invokedynamic, not anonymous classes), the lazy one-pass Stream pipeline over spliterators, the 5-function Collector contract, Optional as a return-type contract, default methods (which made Streams possible), java.time, and CompletableFuture.
  • Parallel streams share the common pool — never in a request handler; CPU-bound, well-splitting, big-N batch work only.
  • Java 11 (bundling 9–10) is the cleanup LTS: var, the standard HttpClient, modules, collection factories, and the functional gap-fillers (takeWhile, bounded iterate, Optional.or/stream, filtering/flatMapping).
  • Judgment: Optional is a return type only; var serves readability; orElseGet over orElse for expensive fallbacks.
