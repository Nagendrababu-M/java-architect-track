RECAP FROM DAY 1

You already know: local primitives live on the stack, references live on the stack, objects live on the heap. Today we extend that with two new wrinkles: static fields (which break the rules) and instance fields (which follow them).

STAGE 1 12 MIN

Instance fields live where their object lives

On Day 1, you saw that the variable order sits on the stack while the Order object sits on the heap. But what about the fields inside that object — like total and id?

This is where instinct can mislead you. People see int total and think “int = primitive = stack.” That rule only applies to locals inside a method. Fields are a different category entirely.

Instance fields are part of the object that owns them.

If the object lives on the heap (which all objects do), then every field declared inside that object — primitive or reference — lives on the heap too. The field is physically embedded in the object’s memory layout.

See it laid out in memory

class User {
    int age;              // instance field, primitive
    String name;          // instance field, reference
}

// Somewhere in a method:
User u = new User();
u.age = 30;
u.name = "Ana";

Notice what’s happening:

  • The age = 30 primitive is stored directly inside the User object’s memory.
  • The name field is a reference (still 8 bytes), embedded inside the object, that points to another heap object — the String "Ana".

So even a single User u = new User() can already produce two heap objects: the User itself, plus the String it references. This is why heap allocation patterns matter for performance — one line of code can mean multiple allocations.

STAGE 2 15 MIN

Static fields — the rule-breakers

So far the rules have been clean: locals on stack, objects on heap, fields with their object. Now we hit an exception.

Static fields don’t belong to any object. They belong to the class itself. There’s only ever one copy, shared across the entire JVM, regardless of how many instances of the class exist.

class Counter {
    static int totalCount = 0;   // belongs to the CLASS
    int instanceCount = 0;        // belongs to each OBJECT
}

Counter a = new Counter();
Counter b = new Counter();
// Two Counter objects exist. Each has its own instanceCount.
// But there is only ONE totalCount, shared by both.

If static fields don’t belong to an object, where do they live? Not on the heap (no object to live in). Not on the stack (they outlive every method). They live in a third region: Metaspace.

Meet Metaspace

Metaspace is where the JVM stores everything about classes themselves:

  • The class’s structure — field names, types, method signatures
  • The bytecode for every method
  • Static fields — both primitive values and object references
  • The Class<?> object that represents the class itself (for reflection)

It’s loaded once when a class is first used (called “class loading”) and typically lives for the entire lifetime of the JVM. Unlike the heap, it’s not garbage-collected on a regular basis — classes only get unloaded under specific conditions.

Static = class-level. Instance = object-level.

Static fields go to Metaspace (one copy per class). Instance fields go to the heap (one copy per object). This holds whether the field is a primitive or a reference.

One catch — a static reference to an object

What if a static field holds a reference to a String or List? The reference itself is in Metaspace. The object it points to is still on the heap. Same split as before — reference and object live in different places.

class Cache {
    static List<String> entries = new ArrayList<>();
}
// 'entries' reference: METASPACE
// the ArrayList object it points to: HEAP

This is why a static HashMap that you forget to clear is a classic memory leak source. The reference in Metaspace keeps the heap object alive forever — and everything that object references — and the GC can never reclaim any of it.

STAGE 3 8 MIN

The unified decision tree

You now have all three regions. Here’s the decision tree that covers every case you’ll see — commit it to memory:

What you haveWhere it livesWhy
Local primitive inside a method STACK Lives in the method’s frame; dies when method returns
Local reference inside a method STACK The reference itself; the object it points to is elsewhere
Object created with new HEAP Always. No exceptions in standard Java.
Instance field (primitive or reference) HEAP Embedded inside the owning object on the heap
Static field (primitive or reference) METASPACE Belongs to the class, not an instance
Class definition (bytecode, methods) METASPACE Loaded once per JVM by the classloader

If you can run code through this table line by line, you have the JVM memory model. Everything else this week — GC, escape analysis, virtual threads — is just consequences of these rules.

STAGE 4 15 MIN

Java Reflection: reading the class at runtime

You now know that class metadata — the description of fields, methods, annotations, and constructors — lives in Metaspace. The Reflection API (java.lang.reflect) is the official way to read and manipulate that metadata at runtime, after compilation.

Every loaded class has exactly one Class<?> object in Metaspace.

Reflection is how your code reaches into that Class object and inspects or invokes what it describes — without knowing the type at compile time.

The core APIs

// 1. Get the Class object — three ways
Class<?> c1 = String.class;                      // compile-time literal
Class<?> c2 = "hello".getClass();                // from an instance
Class<?> c3 = Class.forName("java.lang.String"); // dynamic — triggers class loading into Metaspace

// 2. Inspect fields — see static vs instance distinction from today
for (Field f : MyService.class.getDeclaredFields()) {
    boolean isStatic = Modifier.isStatic(f.getModifiers());
    System.out.printf("%s  static=%b  type=%s%n", f.getName(), isStatic, f.getType().getSimpleName());
}

// 3. Invoke a method by name at runtime
Method m = String.class.getDeclaredMethod("toUpperCase");
String result = (String) m.invoke("hello");  // "HELLO"

// 4. Access a private field — bypass access control
Field f = String.class.getDeclaredField("value");
f.setAccessible(true);                           // ⚠ opens the door — use sparingly
byte[] bytes = (byte[]) f.get("hello");

How this connects to Metaspace

Class.forName("com.example.OrderService") triggers the full class-loading pipeline: load → verify → prepare → resolve → initialise. The resulting Class object and all its metadata land in Metaspace. Every subsequent reflection call reads from that Metaspace entry — not from the heap.

This is why loading thousands of classes dynamically — plugin systems, hot-reload frameworks, OSGi containers — can exhaust Metaspace even when heap usage looks completely normal.

The performance cost — and the modern fix

Reflection is slow compared to direct calls. Every Method.invoke() bypasses JIT inlining, performs access checks on every call, and boxes/unboxes primitives. In a tight loop, it can be 10–100× slower than a direct invocation.

Java 7 introduced MethodHandle in java.lang.invoke. Java 9 added VarHandle for field access. Both are JIT-friendly, module-safe, and the performance gap nearly disappears after the JVM warms up.

// MethodHandle — the JIT-inlineable, module-safe alternative
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findVirtual(
    String.class,
    "toUpperCase",
    MethodType.methodType(String.class)
);
String up = (String) mh.invoke("hello");  // JIT can inline this

// VarHandle — atomic field access without sun.misc.Unsafe
VarHandle vh = MethodHandles.lookup()
    .in(Counter.class)
    .findVarHandle(Counter.class, "count", int.class);
vh.getAndAdd(counter, 1);  // atomic increment — no synchronized block needed

Module system restrictions (Java 9+)

The JPMS module system broke many frameworks that relied on setAccessible(true) across module boundaries. You now need an explicit opens directive in module-info.java, or pass --add-opens on the command line. This is why you see flags like --add-opens java.base/java.lang=ALL-UNNAMED in Spring Boot startup scripts for older apps running on Java 17+.

public class OrderProcessor {
    private static int counter = 0;            // A
    private String region;                        // B

    public Order process(int quantity) {           // C
        int total = quantity * 10;               // D

INTERACTIVE CHALLENGE · 7 ROWS

Where does each piece of data live?

Use STACK, HEAP, or METASPACE for each labeled line.

Static fields belong to the class, not any object. The class lives in Metaspace.

Instance fields are embedded inside their owning object. The OrderProcessor object lives on the heap, so region lives there too — even though it's a String type. Don't be fooled by the type; look at whether it's a field or a local.

Method parameters live in the current stack frame, same as local variables. They're gone when the method returns.

Local primitive variables live in the stack frame of the method that declares them. Popped and gone on return.

The reference order is a local variable — 8 bytes in the current stack frame pointing at the heap object.

new Order(total) allocates an object. All objects go to the heap — always.

The reference on the stack is gone, but the heap object survives — it's now held by the caller's reference. It will be GC'd only when no more references point to it.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Static fields belong to the class and live in Metaspace — they're never GC'd as long as the class is loaded. Static collections are a classic memory leak pattern.
  • Instance fields are part of the object — they live wherever the object lives (always the heap).
  • Metaspace holds class metadata, not objects — it can grow unbounded by default. Always set -XX:MaxMetaspaceSize in production.
