RECAP FROM DAY 3

You know reachability and GC roots. You understand the generational heap: Eden → Survivor → Old Gen, driven by the weak generational hypothesis. You know the difference between Minor, Major, and Full GC, and why Stop-the-World (STW) pauses are the primary source of production latency spikes. STAGE 1 20 MIN

Five collectors, one question: throughput vs latency

Every GC algorithm ever written faces the same fundamental trade-off, and there is no escape from it:

Throughput vs Latency — the GC impossibility triangle.

Throughput: maximize total work done per second. Tolerate occasional long pauses — batch work finishes faster overall even if there are 2-second hiccups. Latency: minimize the worst-case pause time. Accept some throughput loss to keep every pause under a ceiling. No collector wins both. Every algorithm makes an explicit choice about which side of this line to stand on.

Five collectors exist in the modern JVM. Each makes a different choice:

CollectorJava versionPause modelBest for
Serial All Stop-the-world (single thread) Single-core, small heaps, containers with 1 CPU
Parallel (Throughput) All Stop-the-world (parallel threads) Batch jobs, max throughput, pause-tolerant workloads
CMS Deprecated Java 9, removed Java 14 Mostly concurrent Legacy — do not use in new systems
G1 Default since Java 9 Short STW + concurrent phases General purpose, balanced latency/throughput
ZGC Production Java 15+ Sub-millisecond STW Low-latency services, large heaps
Shenandoah OpenJDK 12+ Sub-millisecond STW Same as ZGC, different algorithm approach (concurrent compaction via Brooks pointers)

Serial GC

Uses a single GC thread. When a collection is needed, the entire application stops, one thread does all the work, then the application resumes. Simple, predictable, low overhead. Great for containers with a single CPU — there is no benefit to parallel GC when you only have one core. Activated with -XX:+UseSerialGC.

Parallel GC

Uses multiple GC threads during STW phases — all application threads still pause, but the GC work itself is parallelized across all available cores. If you have 8 cores, 8 threads compact your heap simultaneously. This maximizes throughput: the overall pause duration is shorter than Serial because the work is split, and application threads spend more time running. The downside: pauses on large heaps can still reach seconds, because you’re stopping the world — just with more workers. Activated with -XX:+UseParallelGC.

QUICK CHECK

A nightly batch job processes 10 million records. It runs for 4 hours overnight with no SLA on response time — it just needs to finish as fast as possible. Which collector do you choose: Serial, Parallel, or ZGC? Why is the other option actively wrong here?

STAGE 2 30 MIN

G1 — the region-based collector that changed everything

G1 (“Garbage First”) replaced the old fixed generational layout with a completely different design. Instead of one large contiguous Eden, one Survivor, and one large Old Gen, G1 divides the entire heap into hundreds of equal-sized regions — each between 1MB and 32MB (a power of 2, chosen by the JVM based on heap size, or set with -XX:G1HeapRegionSize).

Regions are labeled dynamically. At any moment, a region is either Eden, Survivor, Old Gen, or Humongous (for objects larger than 50% of a region). There’s no fixed boundary between generations — G1 decides at runtime how many regions to allocate to each role.

The four G1 concepts that matter

1. Remembered Sets (RSets) — Each region maintains a small table recording which other regions hold references into it. When G1 collects a region, it only needs to scan that region’s RSet to find all external pointers — it doesn’t need to scan the entire heap. This is what makes region-level collection possible without scanning everything.

2. Collection Sets (CSets) — G1 doesn’t collect the whole heap at once. It selects a subset of regions — the ones with the highest proportion of garbage — and collects only those. That’s the literal meaning of “Garbage First”: collect the regions with the most garbage first to maximize reclaim rate per unit of pause time.

3. Concurrent Marking — G1 marks live objects concurrently while your application runs. The initial mark (just scanning roots) is a short STW. The bulk of marking happens alongside your app. A second short STW — called the Remark — finalizes what changed during concurrent marking. Then a short Cleanup STW accounts for the results.

4. Pause Target — G1’s killer feature. You specify -XX:MaxGCPauseMillis=200 and G1 uses historical pause data to predict how many regions it can collect within that budget. It adjusts the CSet size accordingly. This is a soft goal, not a hard guarantee — if Old Gen is nearly full, G1 may overshoot to avoid an OOM.

Mixed GC — how G1 handles Old Gen

When the Old Gen reaches a certain occupancy (default 45%, controlled by -XX:InitiatingHeapOccupancyPercent), G1 starts a concurrent marking cycle. After marking completes, G1 enters “mixed” collection mode: it collects Young regions plus a selection of Old Gen regions, targeting the most garbage-dense ones. Mixed GC continues for several cycles until Old Gen is sufficiently cleaned, then returns to normal Young-only collection.

Humongous allocations — G1’s weakness

Objects larger than 50% of a region size (large arrays, big byte buffers) skip Young Gen entirely and go straight to Old Gen as Humongous objects, spanning one or more contiguous regions. This bypasses the generational design entirely. If your application allocates a lot of large objects, G1’s advantage over Parallel GC diminishes. Increase region size (-XX:G1HeapRegionSize=16m) to push the Humongous threshold higher.

Key G1 flags

# Enable G1 (default Java 9+, explicit for clarity)
-XX:+UseG1GC

# Pause target — soft goal in milliseconds
-XX:MaxGCPauseMillis=200

# Region size — power of 2, 1–32 MB
-XX:G1HeapRegionSize=16m

# Start concurrent marking when heap is N% occupied
-XX:InitiatingHeapOccupancyPercent=45

# GC log — always enable in production
-Xlog:gc*:file=gc.log:time,uptime,level,tags

Reading a G1 GC log line

With -Xlog:gc* enabled, a typical Minor GC line looks like:

[2.456s][info][gc] GC(42) Pause Young (Normal) (G1 Evacuation Pause) 512M->128M(2048M) 45.231ms
  • 2.456s — JVM uptime at the pause start
  • GC(42) — the 42nd GC event since JVM start (monotonically increasing)
  • Pause Young (Normal) — a normal Young Gen evacuation (Eden + Survivors copied out)
  • 512M→128M(2048M) — heap before → heap after (max heap capacity)
  • 45.231ms — total pause duration, including STW time

STAGE 3 25 MIN

ZGC — when even 200ms is too slow

ZGC was designed with a single non-negotiable goal: pause times under 1ms regardless of heap size. Not 200ms. Not 50ms. Sub-millisecond, on heaps measured in terabytes. It achieves this through two techniques that are unlike anything in G1.

Colored pointers

In a standard 64-bit JVM, a pointer is 64 bits, but modern hardware only addresses 48 bits of that space (or 52 bits on recent CPUs). ZGC steals the spare bits and uses them to embed GC state directly in every reference. Each pointer carries bits indicating: is this object currently being marked? Has it been relocated? Is it in a remapped state? The GC state travels with the pointer itself. This means the GC can communicate with your running application through the pointer values — no need to stop and scan a separate side table.

Load barriers

Every time your Java code reads an object reference (a load instruction), ZGC inserts a tiny check — a load barrier — that runs inline. The barrier inspects the colored bits. If the pointer is stale (the object was relocated), the barrier fixes it on the fly, updating the reference before returning it to your code. Your code never sees a dangling pointer. This is how ZGC relocates live objects while your application is running without pausing it: relocation and pointer fixup happen lazily, on each load, instead of all at once in a STW stop.

What ZGC still stops for

ZGC has three STW phases, all very short:

  1. Pause Mark Start — scan GC roots (stacks, static fields). Duration: O(thread count), not O(heap size).
  2. Pause Mark End — finalize marking. Duration: also O(thread count).
  3. Pause Relocate Start — start the relocation phase. Very brief.

Because all three pauses scale with the number of threads, not with heap size, pause time stays under 1ms even as you scale from 8GB to 8TB heaps.

ZGC trade-offs

  • Throughput cost: load barriers execute on every reference load. The JIT compiles them efficiently, but there is still a real cost: typically 5–15% lower throughput than G1. For latency-sensitive services this is the right trade.
  • Heap sizing requirement: ZGC needs room to relocate objects while the app runs. The heap should be at least 8× the live set size. If you have 4GB of live data, ZGC wants at least a 32GB heap.
  • 64-bit only: colored pointers require 64-bit address space. No 32-bit JVMs.

Key ZGC flags

# Enable ZGC
-XX:+UseZGC

# ALWAYS set min == max in production — heap resize causes pauses
-Xms8g -Xmx8g

# Soft max — ZGC tries to stay under this, GCs more aggressively if needed
-XX:SoftMaxHeapSize=6g

# GC collection interval — 0 = adaptive (recommended)
-XX:ZCollectionInterval=0

-Xlog:gc*:file=gc.log:time,uptime,level,tags

For the Nifty options engine: ZGC is the correct choice. A G1 pause of 200ms during market open is 200ms of missed prices and unprocessed order signals. The market doesn’t stop. Your engine does. Every filled order during that 200ms goes to a competitor with a faster JVM.

ZGC keeps pauses under 1ms. Set -Xms equal to -Xmx (prevents JVM heap resize, which triggers its own pause). Run the engine on a dedicated host (no noisy neighbours causing OS-level scheduling jitter). With ZGC, your pauses are now in the microsecond range — small enough that market data latency dominates, not GC.

STAGE 4 20 MIN

The 6 flags every architect must know

Most JVM tuning guides list fifty flags. In practice, these six flags cover 90% of what you will ever need to touch in production. Know these cold.

FlagWhat it doesRule of thumb
-Xms / -Xmx Minimum and maximum heap size Set equal in production to eliminate heap resize pauses. -Xms8g -Xmx8g
-XX:NewRatio=N Old:Young size ratio (default 2 = Old is 2× Young) Lower value = larger Young Gen = fewer premature promotions to Old
-XX:MaxGCPauseMillis G1 pause target in milliseconds 200ms general-purpose; 50ms for latency-sensitive services; ignored by ZGC
-XX:+UseZGC Enable ZGC collector Use for latency-sensitive services on Java 15+. Check heap is 8× live set.
-Xlog:gc*:file=gc.log:time,uptime,level,tags Enable structured GC logging to file Always enable in production — non-negotiable. Zero performance cost.

Sizing Young Gen — the highest-impact tuning for high-throughput services

The most common production GC problem — and the one with the highest return on tuning effort — is Young Gen sized too small. When Eden fills faster than Minor GC can clear it, objects get promoted to Old Gen before they would naturally die. Old Gen fills. Major GC fires. Latency spikes.

  • Young Gen too small → objects promoted before they die → Old Gen fills fast → frequent Major GC
  • Young Gen too large → Minor GC takes longer (more live objects to copy) → longer individual pauses
  • Goal: Young Gen should be large enough that 95%+ of allocations die before surviving to Old Gen. If 95% of your objects die within 500ms, your Eden should hold at least 500ms of allocation rate.

Concrete sizing example

Your service allocates at 100 MB/s. Average object lifetime is 500ms. To ensure objects die in Young Gen before a Minor GC promotes them:

# Minimum Eden to hold 500ms of allocation
100 MB/s × 0.5s = 50 MB minimum Eden

# Add 2× headroom for allocation bursts
-XX:NewSize=128m -XX:MaxNewSize=256m

Think of Young Gen as the inbox on your desk. If your inbox holds 2 hours of mail but the postman delivers every 30 minutes, some letters will sit overnight and get “promoted” to the filing cabinet (Old Gen) even though you would have thrown them away tomorrow. A bigger inbox means letters stay visible longer and get thrown away before filing. The filing cabinet fills more slowly. You clean your desk (Major GC) less often.

STAGE 5 30 MIN

You can’t tune what you can’t see

GC flags without GC logs are guesswork. This stage teaches you to read the evidence and gather more of it.

Part A — Reading GC Logs

A healthy G1 service looks like this — short Young GC pauses, heap recovers well after each:

[2.456s][info][gc] GC(42) Pause Young (Normal) (G1 Evacuation Pause)
                   512M->128M(2048M) 45.231ms

Fields decoded: 2.456s = JVM uptime. GC(42) = 42nd GC event. Pause Young (Normal) = normal Minor GC. 512M→128M = heap reduced from 512MB to 128MB. (2048M) = heap capacity. 45.231ms = total pause.

Now the concerning log — this is what a memory leak looks like in a GC log:

[847.2s][info][gc] GC(891) Pause Full (Ergonomics)
                   1890M->1820M(2048M) 4823.ms

Walk through the diagnosis:

  • Pause Full — a Full GC. All other work stops. This is never good.
  • Ergonomics — triggered by the JVM’s internal heuristics, not you. Heap was almost full.
  • 1890M→1820M(2048M) — the Full GC freed only 70MB from a 2048MB heap. That’s 3.5% reclaimed after 4.8 seconds of the world being stopped.
  • 4823.ms — 4.8 seconds. Your service was completely unresponsive for 4.8 seconds.

A Full GC that barely frees memory is not a GC sizing problem. It is a memory leak.

Old Gen is full of live objects — things the GC can still reach, so it can’t collect them. The GC is working correctly. Your code is holding references it shouldn’t. The fix is not more heap or a different GC — it’s finding and fixing the leak. A heap dump at the time of OOM (enabled by -XX:+HeapDumpOnOutOfMemoryError) is your starting point.

Part B — Java Flight Recorder (JFR)

JFR is the JVM’s built-in continuous profiling system. Since Java 11 it is completely free and always-on has near-zero overhead (well under 1% in most workloads). It records a continuous ring buffer of JVM events. You can dump it at any time — even on a production service that’s currently behaving badly.

# Attach to a running process and record for 60 seconds
jcmd <PID> JFR.start duration=60s filename=recording.jfr

# Or enable at JVM startup
java -XX:StartFlightRecording=duration=60s,filename=recording.jfr MyApp

# Stop and dump an ongoing recording
jcmd <PID> JFR.stop name=1 filename=recording.jfr

Open the .jfr file in JDK Mission Control (JMC). The four views you always check first:

  • GC events — pause duration per event, GC type, heap occupancy before and after, which collector phase fired
  • Memory allocation — which classes are allocating most bytes, allocation rate over time (look for sustained climbs that indicate leaks)
  • Hot methods (flame graph) — where CPU time is spent. Find hot loops, unexpected serialization, regex compilation
  • Lock contention — which locks are blocking which threads for how long

async-profiler — for when JFR isn’t enough

# Profile for 30 seconds and write an interactive flame graph
./profiler.sh -d 30 -f flamegraph.html <PID>

JFR has a subtle weakness called safepoint bias. The JVM can only sample at safepoints — points where all threads are in a known state. If your hot code path never naturally hits a safepoint, JFR never samples it, and it disappears from the profile. async-profiler uses OS signals (SIGPROF) to interrupt threads at arbitrary points, regardless of safepoints. It catches what JFR misses. For CPU profiling of tight loops and JNI-heavy code, async-profiler is the tool.

A Full GC is the most thorough collection the JVM can do — it scans everything, collects everywhere. If it freed only 160MB (4% of the heap), 96% of that heap is reachable. MODEL ANSWER

1. Fix the leak (code). Take a heap dump: add -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/app/ and wait for the next OOM, or run jcmd <PID> VM.heap_dump now. Open in Eclipse MAT: “Leak Suspects” report will identify the largest retained heap holders. Find the unbounded collection or cache, add eviction (Guava Cache / Caffeine with maximumSize or expireAfterAccess), or fix the listener that never deregisters.

2. Lower IHOP so G1 starts marking earlier (flags). Set -XX:InitiatingHeapOccupancyPercent=30 (down from 45). G1 will start concurrent marking when the heap is 30% full, giving it more lead time to clean Old Gen before things get critical. This doesn’t fix the leak, but it delays the Full GC for a while.

3. Add a heap utilization alert (operations). If Old Gen occupancy > 70% for > 60s, page the on-call engineer. The leak grows over time — catching it at 70% heap gives you time to take a heap dump before the service OOMs, when you still have a live JVM to inspect. A dead JVM gives you nothing but the heap dump file; a live JVM gives you thread dumps, JFR recordings, and the ability to trigger a GC and observe what survives.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • G1 is the right default for most services: balanced, predictable, works well on heaps from 4GB to 32GB. The 200ms pause target covers the vast majority of production workloads.
  • ZGC is the right choice when latency is the primary constraint: trading systems, real-time pricing, gaming backends. Sub-millisecond pauses at the cost of 5–15% throughput.
  • Always log GC in production (-Xlog:gc*). The log is zero-cost and invaluable when something goes wrong. The JVM is generating this data regardless — you just need to write it somewhere.
  • A Full GC that barely frees memory is a memory leak, not a sizing problem. The fix is a heap dump and code change, not a bigger heap.
