Modern Java Deep Dive: Records, Sealed Classes, Pattern Matching & the LTS Timeline Java 8 → 11 → 17 → 21 — the features interviewers ask about, and the design thinking behind them.

SCOPE · 210 min · 7 stages — interview-prep session: every stage ends with the questions interviewers actually ask
INTERVIEW CONTEXT

Modern Java questions are now standard in senior interviews. The interviewer is not testing whether you memorized syntax — they’re testing whether you understand why these features exist and when you would reach for them. Records, sealed classes, and pattern matching are not three separate features: together they form algebraic data types — a way to model your domain so the compiler enforces correctness. That’s the story you tell in the interview, and that’s the story of today’s session.

STAGE 1 30 MIN

Records — transparent carriers of immutable data

A record is a class whose state is its API. Declared in one line, the compiler generates the constructor, accessors, equals(), hashCode(), and toString():

public record Trade(String symbol, long quantity, BigDecimal price) {}

That single line replaces ~50 lines of a classic value class. But the important part isn’t the brevity — it’s the semantics:

A record is a promise: this type is a transparent, immutable carrier of its components.

All fields are private final. There are no setters. The accessor for symbol is symbol(), not getSymbol(). equals() and hashCode() are component-wise — two records with equal components are equal. Records cannot extend a class (they implicitly extend java.lang.Record) and cannot be extended (implicitly final). They can implement interfaces.

Compact constructors — validation without ceremony

The compact constructor omits the parameter list. It runs before field assignment, and you can normalize values by reassigning the parameters:

public record Trade(String symbol, long quantity, BigDecimal price) {
    public Trade {  // compact constructor — no parameter list
        Objects.requireNonNull(symbol);
        if (quantity <= 0) throw new IllegalArgumentException("quantity must be positive");
        symbol = symbol.toUpperCase();  // normalization: reassigns the parameter, not the field
    }
}

Custom accessors and static factories

public record Range(int lo, int hi) {
    // override an accessor — rare, but legal. Must not break lo() == lo invariant expectations
    public int lo() { return Math.min(lo, hi); }

    // static factory for derived construction
    public static Range of(int a, int b) { return new Range(Math.min(a,b), Math.max(a,b)); }
}

What records are NOT

  • Not shallowly-magic immutable: if a component is a List, the reference is final but the list is still mutable. Defensive-copy in the compact constructor: items = List.copyOf(items);
  • Not JavaBeans: no no-arg constructor, no setters. Some old frameworks (and some serializers without record support) choke on them.
  • Not entities: JPA entities need mutability and identity-based equality. Records have value-based equality. Use records for DTOs, API payloads, map keys, and domain values — not for ORM entities.
🎯 INTERVIEW QUESTIONS — ANSWER OUT LOUD

Q: When would you NOT use a record? — Mutable state, JPA entities, classes needing inheritance, beans requiring no-arg constructors. Records model values; entities model identity over time.

Q: Are records immutable? — Shallowly. The fields are final, but a mutable component (a List, a Date) can still be mutated through its reference. Defensive copies in the compact constructor make them deeply immutable.

Q: How do records interact with serialization? — Records deserialize through the canonical constructor, so your validation in the compact constructor always runs — unlike regular classes where deserialization bypasses constructors. This makes records strictly safer for untrusted data.

QUICK CHECK

A teammate writes record Config(Map<String,String> settings) and claims it’s immutable so it’s safe to share across threads. What’s the flaw, and what one-line fix do you suggest?

STAGE 2 30 MIN

Sealed classes — closing the type hierarchy

Before Java 17, a class hierarchy had exactly two settings: open to everyone (public class) or closed to everyone (final). Sealed types add the middle ground: open to exactly the types I name.

public sealed interface PaymentResult
        permits Approved, Declined, Pending {}

public record Approved(String txnId, BigDecimal amount)        implements PaymentResult {}
public record Declined(String reason, boolean retryable)       implements PaymentResult {}
public record Pending(String txnId, Instant checkAfter)        implements PaymentResult {}

Every permitted subtype must choose one of three modifiers: final (the branch ends here), sealed (the branch continues, controlled), or non-sealed (the branch re-opens to anyone). Records are implicitly final, which is why records + sealed interfaces pair so naturally.

Sealed + records = algebraic data types. This is the headline answer in interviews.

A sealed interface with record implementations is a sum type: a PaymentResult is exactly one of Approved, Declined, or Pending — and the compiler knows the complete list. That knowledge is what unlocks exhaustive switch in Stage 3: no default branch needed, and when someone adds a fourth result type, every switch over PaymentResult in the codebase fails to compile until it handles the new case. The compiler becomes your refactoring safety net.

Replacing the visitor pattern

The classic OO way to dispatch over a closed hierarchy was the Visitor pattern — double dispatch, an accept method on every class, a visitor interface that grows forever. Sealed types + pattern matching replace all of it with a switch expression. If an interviewer asks “how has modern Java changed your design patterns?” — this is the example: visitor is dead for closed hierarchies.

Rules to remember

  • Permitted subtypes must be in the same module (or same package if unnamed module).
  • If subtypes are nested in the same file, the permits clause can be omitted — the compiler infers it.
  • Sealed works on interfaces and classes; enum is conceptually a special case of sealed (fixed set of instances rather than fixed set of subtypes).
🎯 INTERVIEW QUESTIONS

Q: Sealed classes vs enums — when each? — Enum when every variant carries the same shape of data (or none). Sealed hierarchy when variants carry different data: Approved has a txnId, Declined has a reason. Enums are a fixed set of instances; sealed types are a fixed set of shapes.

Q: Why not just use a final class? — Final closes the hierarchy completely. Sealed lets you have polymorphism with a known, compiler-verified set of implementations — you keep the extensibility you control while denying it to everyone else.

QUICK CHECK

You’re designing the result type for an HTTP client library that external teams will extend with custom result types. Is sealed the right choice here? Why or why not?

STAGE 3 35 MIN

Pattern matching — from instanceof to deconstruction

1. Pattern matching for instanceof (Java 16)

// Before: test, then cast — the cast can silently drift from the test
if (obj instanceof String) {
    String s = (String) obj;
    ...
}

// After: test and bind in one step. s is in scope only where the test passed
if (obj instanceof String s && s.length() > 5) {
    ...
}

The binding variable uses flow scoping: it’s in scope exactly where the compiler can prove the test succeeded — including after an early return: if (!(obj instanceof String s)) return; /* s is in scope here */. That flow-scoping detail is a favourite interview probe.

2. Switch expressions (Java 14)

// Arrow labels: no fall-through, each case is an expression
int numLetters = switch (day) {
    case MONDAY, FRIDAY, SUNDAY -> 6;
    case TUESDAY                -> 7;
    default -> {
        int len = day.toString().length();
        yield len;  // yield returns a value from a block case
    }
};
  • Expression, not statement — it produces a value, assignable directly.
  • No fall-through with arrow labels — the #1 historical switch bug, gone.
  • Exhaustiveness checked — an expression must cover all cases (enums and sealed types make this provable without default).

3. Pattern matching for switch + record patterns (Java 21)

This is where everything from Stages 1–2 converges:

String describe(PaymentResult result) {
    return switch (result) {
        // record pattern: matches the type AND deconstructs its components
        case Approved(String txnId, BigDecimal amt)   -> "OK " + txnId + " for " + amt;

        // guarded pattern: 'when' clause adds a condition (Java 21)
        case Declined(String reason, boolean retryable) when retryable
                                                       -> "Retry later: " + reason;
        case Declined(String reason, boolean r)        -> "Hard decline: " + reason;

        case Pending p                                  -> "Check at " + p.checkAfter();
        // NO default needed — sealed guarantees exhaustiveness
    };
}

Record patterns nest. Given record Line(Point start, Point end) and record Point(int x, int y):

if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    // deconstructed two levels deep in one pattern
}

Ordering and dominance

Cases are checked top-to-bottom. A more general pattern placed above a more specific one makes the specific one unreachable — the compiler rejects this as a dominance error. Guarded patterns (when) must come before their unguarded form, as in the Declined example above. Also: case null is now allowed in switch (Java 21); without it, a null selector still throws NullPointerException.

🎯 INTERVIEW QUESTIONS

Q: Why does exhaustive switch matter architecturally? — It moves a class of runtime bugs to compile time. Add a new variant to a sealed type and the compiler points at every switch that must handle it. With if/else chains, you’d find those spots in production.

Q: What happens if you switch over a sealed type and pass null? — NPE, unless you add case null explicitly. Exhaustiveness covers types, not null.

Q: When do you still need default? — When the selector type is open (e.g. switching over Object or a non-sealed interface), or when you deliberately want to collapse remaining cases — at the cost of losing the compile-time check when new variants appear.

QUICK CHECK

You have case Declined(String r, boolean retryable) followed by case Declined(String r, boolean retryable) when retryable. What does the compiler say, and why?

STAGE 4 25 MIN

Text blocks & Sequenced Collections

Text blocks (Java 15)

String query = """
        SELECT t.symbol, SUM(t.quantity)
        FROM trades t
        WHERE t.executed_at > :since
        GROUP BY t.symbol
        """;

The details interviewers probe:

  • Incidental indentation stripping: the closing """ position determines the left margin. Common whitespace to the left of it is stripped; everything to the right is kept.
  • \ at end of line suppresses the newline (one logical line split for readability); \s preserves a trailing space that would otherwise be stripped.
  • It’s still a plain String — same type, same methods, compiled to the same constant pool entry.

Sequenced Collections (Java 21)

Before Java 21, “get the first element” had a different spelling on every collection: list.get(0), deque.getFirst(), sortedSet.first() — and LinkedHashSet had no way at all to get the last element without iterating. JEP 431 retrofits three interfaces into the hierarchy:

InterfaceMethodsImplementors
SequencedCollection<E> getFirst/getLast, addFirst/addLast, removeFirst/removeLast, reversed() List, Deque
SequencedSet<E> same + Set semantics; addFirst on an existing element moves it LinkedHashSet, SortedSet
SequencedMap<K,V> firstEntry/lastEntry, putFirst/putLast, reversed() LinkedHashMap, SortedMap

The headline method is reversed() — it returns a view, not a copy. Mutations on the view write through to the original. for (var e : map.reversed().entrySet()) iterates a LinkedHashMap backwards with zero allocation — previously impossible without copying keys into a list.

🎯 INTERVIEW QUESTIONS

Q: What does addFirst do on a SortedSet? — Throws UnsupportedOperationException. A sorted set’s order is defined by the comparator; you can’t impose a position. The interface is retrofitted onto it for reading ends, not for positional writes.

Q: Is reversed() a copy? — No, a live view. Cheap to create, reflects and propagates mutations.

QUICK CHECK

You maintain an LRU-ish access log in a LinkedHashSet. With Sequenced Collections, how do you (a) move an element to the front on re-access, and (b) evict the oldest? One method call each.

STAGE 5 30 MIN

Structured Concurrency & where Java is heading

The problem: unstructured concurrent code leaks

With plain ExecutorService, when you fork two subtasks and the first one fails, the second keeps running — burning CPU, holding connections — unless you write careful, error-prone cancellation code. Threads outlive the code block that created them. That’s the concurrency equivalent of goto.

StructuredTaskScope (Java 21 preview)

Response handle() throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        Subtask<User>  user  = scope.fork(() -> fetchUser(id));
        Subtask<Order> order = scope.fork(() -> fetchOrder(id));

        scope.join()            // wait for both
             .throwIfFailed();  // propagate the first failure

        return new Response(user.get(), order.get());
    }  // scope close GUARANTEES no subtask outlives this block
}
  • ShutdownOnFailure: first subtask to fail cancels all the others. Use for “I need ALL of these.”
  • ShutdownOnSuccess: first subtask to succeed cancels the rest. Use for “I need ANY of these” — hedged requests, racing replicas.
  • The guarantee: the try-with-resources block cannot exit while subtasks run. Concurrency gets the same block structure as the rest of your code — errors propagate up, cancellation propagates down, thread dumps show the parent-child task tree.

The 30-second answers for everything else

FeatureStatusThe one-liner to say in an interview
String templates Preview, withdrawn for redesign “Interpolation through processors like STR."x = \{x}" — processors can validate/escape, preventing SQL injection by construction. API was withdrawn after Java 22 for redesign — knowing that shows you track the platform.”
Project Valhalla In progress “Value classes — objects without identity that can be flattened into arrays and stack-allocated. Eliminates pointer-chasing for things like Point[]: codes like a class, performs like an int.”
Migration strategy — “Replace inheritance hierarchies over data with sealed interface + records; replace getter-chain null checks with record patterns in switch; replace visitor with exhaustive switch.”

For the trading platform: market events are the textbook sealed hierarchy — sealed interface MarketEvent permits Tick, OrderBookUpdate, TradeExecution, SessionStateChange. The event processing switch is exhaustive: when the exchange adds a new event type and you add it to the hierarchy, the compiler finds every handler in the codebase that must change. Structured concurrency handles the fan-out: price one option by fetching spot, volatility surface, and rates in parallel with ShutdownOnFailure — if the vol fetch fails, the other fetches cancel immediately instead of wasting microseconds you don’t have.

STAGE 6 35 MIN

The LTS timeline — Java 8, 11, 17, 21 in depth

“Which Java version do you use, and what came with it?” is a guaranteed interview question. Companies live on the LTS releases — 8, 11, 17, 21 — so organize your knowledge around those four. For each one, know the headline features, the runtime/JVM changes, and the migration pain.

Java 8 (2014) — the functional revolution

The largest language change in Java’s history. Still the baseline interviewers expect total fluency in:

  • Lambdas & functional interfaces — behavior as data. Runnable r = () -> work(); backed by invokedynamic, not anonymous classes (Day 9 covers the internals).
  • Streams API — declarative, lazy, composable collection processing; parallelizable via the ForkJoin common pool.
  • Optional — a return-type contract for “absence is normal”, ending the null-or-not guessing game.
  • Default & static interface methods — interfaces could finally evolve without breaking implementors. This is how Streams could be added: Collection.stream() is a default method.
  • java.time (JSR-310) — immutable, thread-safe dates replacing the broken Date/Calendar. LocalDate, Instant, ZonedDateTime, Duration.
  • CompletableFuture — composable async pipelines: thenApply, thenCompose, exceptionally.
  • JVM: PermGen removed, Metaspace introduced — class metadata moved to native memory; OutOfMemoryError: PermGen space became Metaspace, sized by -XX:MaxMetaspaceSize.

Java 11 (2018) — the cleanup LTS

The first LTS of the 6-month release era; it bundles Java 9/10 changes, so credit those here:

  • Module system (JPMS, from 9) — module-info.java, strong encapsulation of JDK internals. Most apps stay on the classpath, but the JDK itself is modularized — which is what later enabled jlink custom runtimes and the eventual lockdown of sun.misc.* hacks.
  • var (from 10) — local-variable type inference. Compile-time only; the bytecode is identical. Also legal on lambda params in 11 ((var a, var b) -> ...) so you can annotate them.
  • Standard HTTP Client — java.net.http.HttpClient: HTTP/2, async via CompletableFuture, WebSockets. Killed the need for Apache HttpClient in most services.
  • Collection/String conveniences — List.of()/Map.of() immutable factories (9); String.strip(), isBlank(), lines(), repeat(); Files.readString(); Predicate.not().
  • Single-file execution — java Script.java runs without explicit compilation.
  • Removed: Java EE & CORBA modules (JAXB, JAX-WS) — the #1 migration breakage from 8 to 11; you re-add them as Maven dependencies.
  • JVM: ZGC and Epsilon arrive as experimental; JFR is open-sourced and free in production (it was a commercial feature in 8 — a great interview detail).

Java 17 (2021) — the data-modeling LTS

The release that made this whole lesson possible. Bundles 12–16:

  • Records (16) — Stage 1. Transparent immutable carriers with compiler-generated members.
  • Sealed classes (17) — Stage 2. Closed hierarchies; the compiler knows every subtype.
  • Pattern matching for instanceof (16) — Stage 3. Test-and-bind with flow scoping.
  • Switch expressions (14) — arrow labels, yield, exhaustiveness.
  • Text blocks (15) — Stage 4. Triple-quoted multi-line strings.
  • Helpful NullPointerExceptions (14) — the JVM tells you which variable was null in a chain: “Cannot invoke String.length() because the return value of User.getName() is null”.
  • Strong encapsulation by default (17) — --illegal-access ends; reflective access to JDK internals now requires explicit --add-opens. The #1 migration pain from 11 to 17 (old versions of Lombok, Mockito, serializers break until upgraded).
  • JVM: ZGC and Shenandoah production-ready (15); biased locking removed; always-strict floating point.

Java 21 (2023) — the concurrency LTS

  • Virtual threads (final!) — the headline. Millions of cheap threads multiplexed onto carrier threads; blocking code that scales like async. Thread-per-request returns as a sane architecture. (Week 2, Day 14 goes deep.)
  • Pattern matching for switch + record patterns (final) — Stage 3. Exhaustive switch over sealed hierarchies with deconstruction and when guards.
  • Sequenced Collections — Stage 4. getFirst/getLast/reversed() unified across ordered collections.
  • Structured Concurrency (preview) — Stage 5. StructuredTaskScope; subtasks scoped to a block.
  • Scoped Values (preview) — immutable, inheritable per-thread context; the ThreadLocal replacement designed for virtual threads (a million virtual threads × mutable ThreadLocals = memory disaster).
  • Generational ZGC — ZGC gains young/old generations: most workloads see lower CPU and better allocation throughput at the same sub-ms pauses. Ties directly back to Day 4.
  • String templates (preview) — later withdrawn for redesign (Stage 5’s table).

The one-line-per-LTS summary to deliver in interviews:

8 changed how we write Java (lambdas, streams). 11 cleaned the platform (modules, HTTP client, EE removal, free JFR). 17 changed how we model data (records, sealed, pattern matching). 21 changed how we do concurrency (virtual threads, structured concurrency). If you can attach two or three concrete features and one migration pain to each sentence, you’ve answered the version question at a senior level.

🎯 INTERVIEW QUESTIONS

Q: We’re on Java 8 — what does migrating to 17 buy us, and what breaks? — Buys: records/sealed/pattern matching for the domain model, G1 default + ZGC available, free JFR, helpful NPEs, text blocks, HTTP client. Breaks: JAXB/JAX-WS removal (re-add as dependencies), strong encapsulation (--add-opens or upgrade Lombok/Mockito/ASM-based libs), and any code touching sun.misc.Unsafe internals.

Q: Why do companies skip non-LTS versions? — Six-month releases get six months of free updates; LTS gets years of vendor support. Features land in non-LTS releases first (records in 16) but fleets standardize on the LTS that bundles them (17).

Q: What changed about JFR between 8 and 11? — In 8 it was a commercial Oracle feature requiring -XX:+UnlockCommercialFeatures; since 11 it’s open-source and free in production — which is why “always-on JFR” is now standard ops practice.

QUICK CHECK

Without looking up: name the LTS release each of these shipped in as final — records, virtual threads, lambdas, the standard HTTP client, sealed classes, Sequenced Collections.

STAGE 7 30 MIN

The interview challenge — design with the full toolkit

This is a realistic senior-interview design question. Answer all three parts (50+ characters each), then compare with the model answers.

SCENARIO · A REAL INTERVIEW DESIGN QUESTION

“Model the result of an order-matching attempt”

You’re asked to design the types for an order-matching engine’s result. An attempt to match an order can end in exactly four ways: fully filled (with execution price and quantity), partially filled (filled quantity + remaining quantity), rejected (with a reason code and whether retry is allowed), or queued (with queue position). Downstream code must handle every outcome — forgetting one should be impossible.

YOUR ANSWERS · MINIMUM 50 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 50 characters
MODEL ANSWER

sealed interface MatchResult permits Filled, PartiallyFilled, Rejected, Queued — sealed because the outcomes are a closed set the compiler should know about. Each variant is a record: record Filled(BigDecimal price, long qty), record PartiallyFilled(long filledQty, long remainingQty), record Rejected(ReasonCode reason, boolean retryable), record Queued(int position) — records because each outcome is an immutable value with different shape (which rules out an enum). Validation lives in compact constructors: PartiallyFilled asserts both quantities are positive; Queued asserts position ≥ 0. ReasonCode is an enum — fixed set of instances with identical shape. The combination is an algebraic data type: sum type (sealed) of product types (records).

0 / 50 characters
MODEL ANSWER

A switch expression with record patterns and no default branch: return switch (result) { case Filled(var price, var qty) -> settle(price, qty); case PartiallyFilled(var filled, var remaining) -> requeue(remaining, settle(filled)); case Rejected(var reason, var retryable) when retryable -> scheduleRetry(reason); case Rejected(var reason, var r) -> alertDesk(reason); case Queued(var pos) -> notifyPosition(pos); }; Three features cooperate: sealed gives the compiler the complete list of subtypes; switch as an expression requires exhaustiveness; record patterns deconstruct each variant’s components in place. Omitting default is the crucial choice: when a fifth outcome is added to the sealed interface, this switch stops compiling — the compiler walks you to every consumer that needs updating. The guarded pattern (when retryable) must precede the unguarded Rejected case or the compiler reports a dominance error.

0 / 50 characters
MODEL ANSWER

This is the expression problem. Virtual dispatch (abstract handle()) makes adding new variants easy but adding new operations hard — every new operation touches every class. The data-oriented design (sealed + records + switch) makes adding new operations easy — each operation is one self-contained switch — but adding a variant touches every switch (which is exactly what we want here: the compiler enumerates the touch points). For a MatchResult, the variants are stable (four outcomes, defined by the matching algorithm) while operations multiply endlessly — settlement, risk, audit logging, client notification, metrics — and those operations belong to different teams and modules that shouldn’t inject behavior into a shared domain type. Data-oriented wins. Classic polymorphism is better when the variants are open-ended and operations are few and stable — e.g. a plugin system where third parties add new implementations of a fixed interface: there, sealed is exactly wrong (Stage 2’s HTTP-client example).

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Records + sealed + pattern matching are one feature, not three: algebraic data types with compiler-enforced exhaustive handling. That framing is your interview headline.
  • Records are shallowly immutable — defensive-copy mutable components in the compact constructor. And record deserialization runs the canonical constructor, so validation can’t be bypassed.
  • Exhaustive switch without default turns “new variant added” from a production bug hunt into a list of compile errors.
  • Structured concurrency gives threads block scope: subtasks cannot outlive their scope; failure cancels siblings (ShutdownOnFailure) or success cancels the losers (ShutdownOnSuccess).
  • Know the expression problem — it’s the deep answer behind “sealed+switch vs polymorphism” and separates senior candidates from mid-level ones.
