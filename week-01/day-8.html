STAGE 1 25 MIN

The Java 8 era (2014, LTS) — the functional revolution

The largest language change in Java’s history, and still the baseline interviewers expect total fluency in. Everything later builds on this.

Lambdas, functional interfaces & method references

A functional interface has exactly one abstract method (a SAM type). A lambda is its inline implementation; a method reference is an even shorter spelling:

// Before: anonymous class ceremony
Comparator<Trade> c = new Comparator<>() {
    public int compare(Trade a, Trade b) { return a.price().compareTo(b.price()); }
};
// Lambda, then method-reference composition
Comparator<Trade> c = (a, b) -> a.price().compareTo(b.price());
Comparator<Trade> c = comparing(Trade::price).thenComparing(Trade::symbol);

The four kinds of method reference: static (Integer::parseInt), bound instance (myStr::length), unbound instance (String::length — receiver becomes the first arg), and constructor (ArrayList::new).

Stream API — basic operations

// Declarative pipeline: filter → map → collect
List<String> bigBuys = trades.stream()
    .filter(t -> t.side() == Side.BUY && t.qty() > 10_000)
    .map(Trade::symbol)
    .collect(Collectors.toList());

Optional<T>

// The signature itself says "absence is a normal outcome"
Optional<User> findUser(String id);
String name = findUser(id).map(User::name).orElse("anonymous");

Default & static interface methods

Interfaces can finally carry implementations — which is precisely how Streams could be added without breaking every existing collection:

public interface Collection<E> {
    default Stream<E> stream() { return StreamSupport.stream(spliterator(), false); }
}

java.time (JSR-310) & CompletableFuture basics

LocalDate expiry = LocalDate.of(2026, Month.JUNE, 25);   // immutable, thread-safe
Duration left  = Duration.between(Instant.now(), expiry.atStartOfDay(ZoneOffset.UTC).toInstant());

CompletableFuture.supplyAsync(() -> fetchSpot(sym))
    .thenApply(spot -> price(spot))
    .exceptionally(ex -> fallback(sym));

STAGE 2 20 MIN

The Java 9 → 11 era — cleanup & convenience

┃ JAVA 9 (2017)

// Collection factory methods — immutable, reject nulls
List<String> syms = List.of("NIFTY", "BANKNIFTY");
Map<String,Integer> lots = Map.of("NIFTY", 50, "BANKNIFTY", 15);

// Stream additions
prices.stream().takeWhile(p -> p < limit);   // stop at first failing element
prices.stream().dropWhile(p -> p < limit);   // skip until first passing element
Stream.ofNullable(maybeNull);                // 0-or-1 element stream
  • Private interface methods — share helper code between default methods without exposing it.
  • JPMS (module system) — module-info.java with requires/exports. Brief mention only: most applications stay on the classpath; the big impact is that the JDK itself is modularized.

┃ JAVA 10 – 11 (2018, LTS)

// var — local-variable type inference (10). Compile-time only; identical bytecode.
var bySymbol = new HashMap<String, List<Trade>>();
(@Nonnull var a, @Nonnull var b) -> a.merge(b)   // var in lambda params (11)

// Standardized HttpClient (11) — HTTP/2, async, WebSocket
HttpClient client = HttpClient.newHttpClient();
HttpResponse<String> r = client.send(req, BodyHandlers.ofString());

// String methods (11)
"  x  ".strip();  "".isBlank();  "a\nb".lines();  "-".repeat(40);

// File convenience (11)
String cfg = Files.readString(Path.of("app.conf"));
Files.writeString(Path.of("out.txt"), data);
🎯 INTERVIEW QUESTIONS

Q: Is var dynamic typing? — No. It’s compile-time inference; the variable has a fixed static type and the bytecode is identical to writing the type out. Locals only — never fields, params, or return types.

Q: Are List.of() collections just unmodifiable wrappers? — They’re genuinely immutable, reject null elements, and are often more compact than Collections.unmodifiableList(new ArrayList<>()).

STAGE 3 25 MIN

The Java 14–16 era — expressions, records & pattern matching

┃ SWITCH EXPRESSIONS (standard in 14)

int lot = switch (symbol) {
    case "NIFTY"     -> 50;
    case "BANKNIFTY" -> 15;
    default -> { int v = lookup(symbol); yield v; }   // yield from a block
};

Arrow labels mean no fall-through. It’s an expression — produces a value — and must be exhaustive.

┃ TEXT BLOCKS (standard in 15)

String q = """
    SELECT symbol, SUM(qty)
    FROM trades GROUP BY symbol
    """;

Incidental indentation is stripped relative to the closing """; \ at line end suppresses the newline, \s preserves a trailing space.

┃ RECORDS (standard in 16)

public record Trade(String symbol, long qty, BigDecimal price) {
    public Trade {                              // compact constructor: validate + normalize
        Objects.requireNonNull(symbol);
        if (qty <= 0) throw new IllegalArgumentException("qty");
        symbol = symbol.toUpperCase();           // reassigns the PARAMETER, not the field
    }
}

The compiler generates the constructor, accessors (symbol(), not getSymbol()), equals/hashCode (component-wise), and toString. Records are implicitly final, can’t extend a class, but can implement interfaces. Two cautions: they’re only shallowly immutable (defensive-copy a List component with List.copyOf), and record deserialization runs the canonical constructor — so your validation can never be bypassed, unlike normal classes.

┃ PATTERN MATCHING FOR instanceof (standard in 16)

// Test and bind in one step; flow scoping puts 'te' in scope only where proven
if (msg instanceof TradeEvent te && te.qty() > 0) process(te);
if (!(msg instanceof TradeEvent te)) return;  // te is in scope below this line

Also in this era

  • Helpful NullPointerExceptions (14) — “Cannot invoke Address.getCity() because the return value of Customer.getAddress() is null” instead of a bare line number.
  • Stream.toList() (16) — shorthand for collect(toList()) that returns an unmodifiable list (Day 9 covers the migration trap).

STAGE 4 25 MIN

Java 17 (2021, LTS) — sealed classes & the big idea

┃ SEALED CLASSES (standard in 17)

Before 17, a hierarchy was either fully open or fully final. Sealed adds the middle ground — open to exactly the types you name:

public sealed interface PaymentResult
        permits Approved, Declined, Pending {}

public record Approved(String txnId, BigDecimal amount)  implements PaymentResult {}
public record Declined(String reason, boolean retryable) implements PaymentResult {}
public record Pending(String txnId, Instant checkAfter)  implements PaymentResult {}

Each permitted subtype must be final, sealed, or non-sealed. Records are implicitly final, which is why records + sealed interfaces pair so naturally. (Java 17 also added enhanced pseudo-random generators — the RandomGenerator interface family.)

The big idea: records + sealed classes + pattern matching = algebraic data types.

A sealed interface with record implementations is a sum type of product types: a PaymentResult is exactly one of Approved, Declined, or Pending — and the compiler knows the complete list. That knowledge powers exhaustive switch (no default needed), and the day someone adds a fourth result type, every switch over PaymentResult stops compiling until it handles the new case. This replaces visitor patterns and long instanceof chains, and turns “forgot a case” from a production bug into a compile error. (Pattern matching for switch and record deconstruction go fully standard in Java 21 — Day 9.)

🎯 INTERVIEW QUESTIONS

Q: Sealed classes vs enums? — Enum when every variant has the same shape (or none). Sealed hierarchy when variants carry different data: Approved has a txnId, Declined has a reason. Enums = fixed set of instances; sealed = fixed set of shapes.

Q: When is sealed the wrong choice? — When third parties must extend your type (a plugin SPI, a public library result type). Sealing slams that door. Closed domain model: seal it. Open extension point: don’t.

STAGE 5 10 MIN

The architect lens & interview signals

Knowing the syntax is table stakes. Seniority shows in when you reach for each tool and what you refuse to do with it.

The judgment rules that separate senior candidates:

  • Java 8 changed how you express data transformations — functional composition replaces imperative loops, but a loop is still clearer for genuinely imperative work with side effects.
  • Optional<T> is a return type only — never a field, parameter, or collection element. It’s a contract for “no result,” not a general null replacement (it costs an allocation and isn’t Serializable).
  • var should improve readability, not hide the type from a reviewer reading a diff. var x = service.process() is worse than the explicit type; var trades = new ArrayList<Trade>() is fine.
  • Sealed interfaces over abstract classes when the type set is closed and finite — you get exhaustiveness for free.
  • Records for immutable DTOs, value objects, API request/response classes — not for JPA entities, which need mutability and identity-based equality. Records model values; entities model identity over time.
🎯 INTERVIEW SIGNALS — ANSWER OUT LOUD

“Walk me through trade-offs of Stream API vs imperative loops.” — Streams: declarative, composable, parallelizable, no index bugs; cost: harder to debug step-by-step, overhead for tiny collections, awkward for early-exit-with-side-effects.

“When should Optional NOT be used?” — As a field, method parameter, or collection element; and never call get() without checking.

“Difference between map, flatMap, and mapToObj?” — map: 1→1 transform; flatMap: 1→many, flattening nested streams; mapToObj: from a primitive stream (IntStream) back to an object stream.

“Design a payment result type using records and sealed classes” / “Convert this if-else chain to pattern matching for instanceof.” — This is Stage 6.

SCENARIO · READ CAREFULLY

The order state hierarchy

Model a trading order’s lifecycle. An order is in exactly one of these states, and each carries different data:

  • Submitted — no extra data
  • Validated — no extra data
  • Filled(BigDecimal fillPrice)
  • Rejected(String reason)
  • Cancelled — no extra data

Downstream code must produce a status message for every state — and forgetting one should be impossible.

YOUR ANSWERS · MINIMUM 50 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 50 characters
MODEL ANSWER

sealed interface OrderState permits Submitted, Validated, Filled, Rejected, Cancelled — sealed because the set of states is closed and the compiler should know it. States with no data can be records with no components (record Submitted() implements OrderState {}) or, more idiomatically, a single enum-style constant; states with data are records: record Filled(BigDecimal fillPrice) implements OrderState {}, record Rejected(String reason) implements OrderState {} — records because each is an immutable value with a different shape (which rules out a plain enum). Validation lives in compact constructors: Filled asserts fillPrice is non-null and positive; Rejected asserts reason is non-blank. The whole thing is an algebraic data type: a sum type (sealed) of product types (records).

0 / 50 characters
MODEL ANSWER

A switch expression with record patterns and no default branch: return switch (state) { case Submitted s -> "Order submitted"; case Validated v -> "Passed risk checks"; case Filled(var price) -> "Filled at " + price; case Rejected(var reason) -> "Rejected: " + reason; case Cancelled c -> "Cancelled by user"; }; Three features cooperate: sealed gives the compiler the complete subtype list; switch as an expression requires exhaustiveness; record patterns (Filled(var price)) deconstruct the data in place. Omitting default is the crucial choice — it keeps the exhaustiveness check alive. (Pattern matching for switch + record patterns are standard in Java 21; on 17 you’d use the preview or pattern instanceof in if-chains.)

0 / 50 characters
MODEL ANSWER

Every exhaustive switch over OrderState that lacks a default branch stops compiling — the compiler reports that PartiallyFilled is not covered. This is exactly what you want: the compiler hands you a to-do list of every consumer that must be updated (settlement, risk, audit, client notification…), turning a silent “forgot a case” runtime bug into a set of compile errors you cannot ignore. This is the headline benefit of algebraic data types — exhaustiveness as a refactoring safety net. The contrast: with if/else instanceof chains or a default that swallows unknowns, the new state would slip through to production undetected. The only place you’d lose this is a switch that includes default — which is why you omit default on closed hierarchies.

Key Takeaways
  • Know the timeline: lambdas/streams/Optional (8) → collection factories & var & HttpClient (9–11) → switch expressions/text blocks/records/pattern instanceof (14–16) → sealed classes (17).

  • Records are shallowly immutable and deserialize through the canonical constructor (validation can’t be bypassed).
  • Judgment matters: Optional is a return type only; var serves readability; records for values, not JPA entities; sealed for closed hierarchies.
