STAGE 1 20 MIN

┃ JAVA 17 (2021, LTS) — SWITCH EXPRESSIONS & TEXT BLOCKS

The ergonomics that set up records & pattern matching

Switch expressions (standard in 14)

// Arrow labels: no fall-through, each case is an expression that yields a value
int lot = switch (symbol) {
    case "NIFTY", "FINNIFTY" -> 25;       // multiple labels per case
    case "BANKNIFTY"          -> 15;
    default -> {
        int v = lookup(symbol);
        yield v;                          // yield returns a value from a block case
    }
};
  • Expression, not statement — produces a value you assign directly.
  • No fall-through with arrow labels — the #1 historical switch bug, eliminated.
  • Exhaustiveness — a switch expression must cover all inputs; over an enum or sealed type the compiler proves it without a default (the foundation Stage 5 builds on).

Text blocks (standard in 15)

String query = """
        SELECT t.symbol, SUM(t.qty)
        FROM trades t
        WHERE t.executed_at > :since
        GROUP BY t.symbol
        """;
  • Incidental indentation stripping — the position of the closing """ sets the left margin; whitespace to its left is stripped, everything to its right kept.
  • \ at line end suppresses the newline (one logical line split for readability); \s preserves a trailing space that would otherwise be stripped.
  • It is still a plain String — same type, same methods, same constant-pool entry.
🎯 INTERVIEW QUESTIONS

Q: Switch expression vs switch statement? — The expression yields a value, forbids fall-through (arrow labels), and is exhaustiveness-checked. The old statement form still exists for side-effecting branches but is where fall-through bugs live.

Q: Does a text block strip all leading whitespace? — No — only the incidental indentation common to all lines, measured against the closing delimiter. Indentation to the right of that margin is preserved.

STAGE 2 30 MIN

┃ JAVA 17 (2021, LTS) — RECORDS (standard in 16)

Transparent carriers of immutable data — in full

A record is a class whose state is its API. Declared in one line, the compiler generates the canonical constructor, the accessors, equals(), hashCode(), and toString():

public record Trade(String symbol, long quantity, BigDecimal price) {}

A record is a promise: this type is a transparent, immutable carrier of its components.

All fields are private final. There are no setters. The accessor for symbol is symbol(), not getSymbol(). equals()/hashCode() are component-wise — two records with equal components are equal. A record cannot extend a class (it implicitly extends java.lang.Record) and cannot be extended (implicitly final). It can implement interfaces.

Compact constructors — validation & normalization

The compact constructor omits the parameter list. It runs before field assignment, and you normalize by reassigning the parameters:

public record Trade(String symbol, long quantity, BigDecimal price) {
    public Trade {                                   // no parameter list
        Objects.requireNonNull(symbol);
        if (quantity <= 0) throw new IllegalArgumentException("quantity must be positive");
        symbol = symbol.toUpperCase();                // reassigns the PARAMETER, not the field
    }
}

Custom accessors, static factories, extra members

public record Range(int lo, int hi) {
    public Range(int lo, int hi) {                    // canonical ctor (full form) for derived logic
        this.lo = Math.min(lo, hi);
        this.hi = Math.max(lo, hi);
    }
    public int span() { return hi - lo; }              // extra methods are fine
    public static Range unit() { return new Range(0, 1); } // static factory
}

What records are NOT — three traps

  • Not deeply immutable. If a component is a List or array, the reference is final but the contents are still mutable. Defensive-copy in the compact constructor: items = List.copyOf(items); — one line for deep immutability.
  • Not JavaBeans. No no-arg constructor, no get prefix. Frameworks that reflect on bean conventions may choke (modern Jackson/Spring support records; ancient versions don’t).
  • Not JPA entities. Entities need mutability and identity-based equality; records have value-based equality and are final. Use records for DTOs, API payloads, value objects, and map keys — not for ORM entities.

The deserialization-safety angle (a senior signal). Normal Java deserialization bypasses constructors entirely — your validation never runs. Record deserialization goes through the canonical constructor, always. So your compact-constructor validation cannot be skipped, making records strictly safer than regular classes for untrusted/serialized data. Stating this in an interview separates you from candidates who only know “records are short.”

🎯 INTERVIEW QUESTIONS

Q: When would you NOT use a record? — Mutable state, JPA entities, classes needing inheritance, or beans requiring a no-arg constructor. Records model values; entities model identity over time.

Q: Are records immutable? — Shallowly. Fields are final, but a mutable component can be mutated through its reference. Defensive copies in the compact constructor make them deeply immutable.

Q: How do records interact with serialization? — They deserialize through the canonical constructor, so validation always runs — unlike regular classes, where deserialization bypasses constructors.

STAGE 3 25 MIN

┃ JAVA 17 (2021, LTS) — SEALED CLASSES & PATTERN MATCHING instanceof

Closing the hierarchy — and the big idea

Pattern matching for instanceof (standard in 16)

// Test and bind in one step; flow scoping limits the binding to where it's proven
if (msg instanceof TradeEvent te && te.qty() > 0) process(te);
if (!(msg instanceof TradeEvent te)) return;   // te is in scope BELOW this line

The binding variable uses flow scoping: it’s in scope exactly where the compiler can prove the test passed — including after an early return. This kills the historical “test, then cast on the next line” pattern where the cast and the test could silently drift apart.

Sealed classes (standard in 17)

Before 17, a hierarchy was either fully open (public class) or fully closed (final). Sealed adds the middle ground — open to exactly the types you name:

public sealed interface PaymentResult
        permits Approved, Declined, Pending {}

public record Approved(String txnId, BigDecimal amount)  implements PaymentResult {}
public record Declined(String reason, boolean retryable) implements PaymentResult {}
public record Pending(String txnId, Instant checkAfter)  implements PaymentResult {}

Each permitted subtype must choose one modifier: final (the branch ends), sealed (continues, controlled), or non-sealed (re-opens to anyone). Records are implicitly final, which is why records + sealed interfaces pair so naturally. Rules: permitted subtypes live in the same module (or package, unnamed module); if they’re nested in the same file the permits clause can be inferred.

The big idea: records + sealed classes + pattern matching = algebraic data types.

A sealed interface with record implementations is a sum type of product types: a PaymentResult is exactly one of Approved, Declined, or Pending — and the compiler knows the complete list. That knowledge powers exhaustive switch (Stage 5): no default needed, and the day someone adds a fourth result type, every switch over PaymentResult stops compiling until it handles the new case. This replaces the visitor pattern and long instanceof chains, and turns “forgot a case” from a production bug into a compile error.

🎯 INTERVIEW QUESTIONS

Q: Sealed classes vs enums? — Enum when every variant has the same shape (or none). Sealed when variants carry different data: Approved has a txnId, Declined a reason. Enums = fixed set of instances; sealed = fixed set of shapes.

Q: Why not just a final class, or an open hierarchy? — Final closes it to everyone; open lets anyone extend. Sealed gives compiler-verified, controlled polymorphism — you keep the extensibility you choose and deny the rest, which is what unlocks exhaustiveness.

Q: When is sealed the wrong choice? — When third parties must extend your type (a plugin SPI, a public result type). Closed domain model: seal it. Open extension point: don’t.

STAGE 4 15 MIN

┃ JAVA 17 (2021, LTS) — STREAM & API ERGONOMICS (12–16)

The smaller wins bundled into 17

Stream.toList() (16) — and its migration trap

List<String> a = stream.collect(Collectors.toList()); // happens to be a mutable ArrayList
List<String> b = stream.toList();                     // UNMODIFIABLE; also no collector indirection
b.add("x");  // UnsupportedOperationException — mechanical migration can break mutating code

mapMulti (16) & teeing (12)

// mapMulti: allocation-free 1→0-or-many — no per-element Stream object like flatMap
orders.stream().<OrderLine>mapMulti((o, push) -> o.lines().forEach(push)).toList();

// teeing: feed every element to TWO collectors, merge results — min and max in one pass
trades.stream().collect(teeing(
    minBy(comparing(Trade::price)), maxBy(comparing(Trade::price)), PriceRange::new));

Records as stream tuples (16) — including local records

List<String> report(List<Trade> trades) {
    record SymbolValue(String symbol, BigDecimal value) {}   // declared INSIDE the method
    return trades.stream()
        .map(t -> new SymbolValue(t.symbol(), t.value()))
        .sorted(comparing(SymbolValue::value).reversed())
        .map(sv -> sv.symbol() + ": " + sv.value())
        .toList();
}

Before records, one-off intermediate pipeline shapes meant AbstractMap.SimpleEntry or a hand-written class; local records make them free.

Helpful NullPointerExceptions (14)

// Before: "NullPointerException" at a line — which of three calls was null?
order.getCustomer().getAddress().getCity();
// Java 14+: "Cannot invoke Address.getCity() because the return value of
//           Customer.getAddress() is null"

(Also in this era: enhanced pseudo-random generators — the RandomGenerator interface family in 17.)

🎯 INTERVIEW QUESTIONS

Q: Stream.toList() vs collect(Collectors.toList()) — just style? — No. toList() returns an unmodifiable list; Collectors.toList() returns a mutable ArrayList (documented as no-guarantees). Mechanical migration can break code that mutates the result, and toList() can be faster (no accumulator indirection, sized allocation).

STAGE 5 20 MIN

┃ JAVA 21 (2023, LTS) — PATTERN MATCHING FOR SWITCH & RECORD PATTERNS

The data-modeling story reaches its conclusion

The Java 17 foundations (sealed types, records, pattern instanceof) reach their conclusion here: full pattern matching for switch and record deconstruction, both standard in 21.

return switch (result) {
    case Approved(var id, var amt)             -> settle(id, amt);
    case Declined(var r, var retry) when retry -> requeue(r);   // guarded: 'when'
    case Declined(var r, var x)               -> reject(r);
    case null                                  -> handleNull();   // explicit null case
    case Pending p                            -> defer(p);
};  // sealed ⇒ exhaustive ⇒ no default needed

Type patterns & the death of cast-after-test

A type pattern (case Approved a) tests the runtime type and, on match, binds a pre-narrowed variable scoped to that branch — no cast. The long if (x instanceof A) { A a = (A) x; ... } chain, where a stale cast and stale test silently drift apart, collapses into a switch where the compiler guarantees they can never disagree.

Guards & dominance — ordering is a compile-time contract

A guard (when retry) refines a pattern with a boolean. Cases run top-to-bottom and the compiler enforces dominance: a broader pattern above a narrower one it subsumes is a compile error, because the narrower case is unreachable. So the guarded case Declined(...) when retry must precede the unguarded case Declined(...). This is the compiler preventing dead branches, not a style rule.

Null handling & how exhaustiveness is proven

switch still throws NullPointerException on a null selector unless you write case null (optionally case null, default) — because exhaustiveness reasons about types, and null inhabits every reference type. Over a sealed type the compiler walks the permits list recursively, proves total coverage, and lets you omit default; it even synthesizes a hidden MatchException fallback for a subtype added via separate compilation. Omitting default is the correct choice — it preserves the refactoring safety net; a default would silently absorb future subtypes.

Record patterns — deconstruction, nested

case Approved(String txnId, BigDecimal amount) -> settle(txnId, amount);

// Nested: destructure an object graph in one pattern
if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2)))
    double len = Math.hypot(x2 - x1, y2 - y1);   // all four bound at once

Sealed (closed set) + record patterns (destructure each shape) + exhaustiveness (nothing missed) = the complete algebraic-data-type triad, and the full replacement for the visitor pattern. Unlike a visitor, adding a variant doesn’t force every implementor to re-implement an interface — it just lights up compile errors at the switches that care.

🎯 INTERVIEW QUESTIONS

Q: Explain exhaustiveness checking and why it matters for refactoring. — Over a sealed type the compiler proves every subtype is covered; a switch with no default then compiles. Add a subtype and every such switch breaks the build, naming exactly where to update — “forgot a case” moves to compile time. A default defeats this.

Q: Why must a guarded case precede its unguarded form? — Dominance: the unguarded pattern subsumes the guarded one, so reversed, the guarded case is unreachable — a compile error.

Q: Switch over a sealed type with null? — NPE unless you add case null; exhaustiveness covers types, not null.

STAGE 6 25 MIN

┃ JAVA 21 (2023, LTS) — VIRTUAL THREADS, STRUCTURED CONCURRENCY & PLATFORM

The concurrency revolution

Virtual threads (standard) — the headline

try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (Order o : orders) exec.submit(() -> enrich(o));  // 100k blocking calls? Fine.
}
Thread.ofVirtual().start(() -> doWork());

A platform thread wraps an OS thread: ~1 MB stack, kernel-scheduled, expensive — capping you at a few thousand. So high-concurrency I/O went asynchronous (callbacks, reactive), trading readable blocking code for throughput. Virtual threads remove that trade by making blocking cheap.

The M:N scheduling model: a virtual thread is a Java object holding a continuation plus a heap stack — not an OS thread. Many are multiplexed onto a small pool of carrier threads (platform threads):

  • Mount: to run, a virtual thread mounts onto a carrier and executes as normal Java.
  • Unmount: on an instrumented blocking call (socket read, sleep, most j.u.c waits), the runtime captures its continuation, copies its stack to the heap, and unmounts it — freeing the carrier to run another virtual thread immediately.
  • Resume: when I/O completes, it’s rescheduled onto any free carrier and continues from the park point.

Code is written in plain blocking style, but the carrier never blocks — so a handful of OS threads serve hundreds of thousands of concurrent operations. Thread-per-request is sane again.

Pinning — the one failure mode to know cold. A virtual thread that can’t unmount while blocked is pinned to its carrier, which sits idle — reintroducing the exact starvation virtual threads were meant to remove. Causes: (1) blocking inside a synchronized block/method (the JDK can’t unmount across a monitor); (2) blocking in a native (JNI) call. Fix: switch hot blocking sections from synchronized to ReentrantLock (its park is unmount-aware). Detect: -Djdk.tracePinnedThreads=full. Also note: virtual threads don’t speed up CPU-bound work (still core-limited), and you never pool them — create one per task and let it die.

Structured concurrency (preview)

Virtual threads make spawning cheap; structured concurrency makes it safe. With a bare executor, if you fork two subtasks and the first fails, the second runs on orphaned. Structured concurrency gives concurrent tasks block lifecycle:

try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var user  = scope.fork(() -> fetchUser(id));     // each fork → its own virtual thread
    var order = scope.fork(() -> fetchOrder(id));
    scope.join();                                   // wait for ALL
    scope.throwIfFailed();                           // propagate the first failure
    return new Response(user.get(), order.get());
}  // scope close: no subtask can outlive this block
  • ShutdownOnFailure — “I need ALL”: first failure cancels the siblings.
  • ShutdownOnSuccess — “I need ANY”: first success cancels the rest (hedged requests).
  • Cancellation propagates down, exceptions up, and a thread dump shows the real parent–child tree.

Scoped values (preview) — the ThreadLocal replacement

static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();
ScopedValue.where(REQUEST_ID, "req-8861").run(() -> handle());
// Inside handle() and everything it calls: REQUEST_ID.get()

ThreadLocal has three flaws virtual threads make fatal: mutable anywhere downstream, unbounded lifetime (leaks; a forgotten remove() in a pooled thread is a classic bug), and a million mutable maps is a memory disaster. Scoped values flip all three: immutable (rebind only in a nested scope), lifetime = the dynamic extent of the run/call (auto-torn-down), and inherited by forked subtasks in a StructuredTaskScope — so request context flows across the fan-out. The two features are co-designed.

Sequenced collections (standard) & generational ZGC

// Unified first/last/reversed across List, Deque, LinkedHashSet, LinkedHashMap
history.addFirst(latest);  history.removeLast();  var newest = history.getFirst();
for (var e : auditLog.reversed().entrySet()) { ... }   // reversed() is a zero-copy VIEW

reversed() returns a live view (O(1), writes through), so descending iteration of a LinkedHashMap is allocation-free. One sharp edge: addFirst on a SortedSet throws UnsupportedOperationException — its order comes from its comparator. Generational ZGC splits ZGC’s heap into young/old generations, exploiting the weak generational hypothesis (Day 3) for lower CPU and better allocation throughput while keeping the sub-ms pauses from Day 4 — now the recommended ZGC default.

Coming next: Gatherers (Java 22+ preview)

Collectors customize a stream’s terminal step; Stream.gather(...) finally customizes the intermediate step — windowFixed (batching), windowSliding (moving averages), scan (running totals). Java 21 is the current LTS, so know it exists and the gap it fills.

🎯 INTERVIEW QUESTIONS

Q: Walk me through what happens when a virtual thread blocks on a socket read. — The instrumented call captures its continuation, copies its stack to the heap, unmounts it from the carrier (which runs another virtual thread); on completion it’s rescheduled onto a free carrier and resumes. The OS thread never blocked.

Q: What is pinning — cause, detection, fix? — A virtual thread that can’t unmount while blocked, holding its carrier idle. Caused by blocking in synchronized or JNI. Detect with -Djdk.tracePinnedThreads=full; fix by switching to ReentrantLock.

Q: Virtual threads vs Project Reactor? — Virtual threads for blocking I/O you want simple and debuggable; Reactor still wins for backpressure, streaming, and complex async composition. Virtual threads remove the “go reactive just to scale blocking calls” pressure.

Q: Why can’t ThreadLocal work well with virtual threads, and what replaces it? — A million mutable, unbounded-lifetime ThreadLocals is a memory/leak hazard. Scoped values: immutable, scope-bound lifetime, inherited by subtasks.

QUICK CHECK

You move a service to virtual threads and throughput barely improves; a profiler shows carriers mostly idle. One synchronized method guards a call to a slow external API. Explain what’s happening and the fix.

The order state hierarchy

Model a trading order’s lifecycle. An order is in exactly one of these states, each carrying different data:

  • Submitted — no extra data
  • Validated — no extra data
  • Filled(BigDecimal fillPrice)
  • Rejected(String reason)
  • Cancelled — no extra data

Downstream code must produce a status message for every state — and forgetting one should be impossible.

sealed interface OrderState permits Submitted, Validated, Filled, Rejected, Cancelled — sealed because the set is closed and the compiler should know it.
