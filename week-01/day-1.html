The JVM Memory Model Part 1 of 2 — Stack vs. Heap, built from first principles.

SCOPE · 60 min · 5 stages, pause after each
STAGE 1 8 MIN

What does “JVM memory” actually mean?

When you run java MyApp, the operating system gives your JVM process a chunk of RAM. The JVM doesn’t treat this RAM as one big blob — it deliberately divides it into regions, each with a different job.

Why divide it? Because different kinds of data have different lifetimes and access patterns. A method call ends in microseconds. An object you store in a cache might live for hours. Treating them the same would be wasteful and slow.

The regions, at the highest level

There are a few more regions (PC registers, native method stack) but these three carry 95% of the architect-level conversation.

Today, we focus on the first two: heap and stack. Metaspace gets its own day later this week.

STAGE 2 12 MIN

The Stack: a stack of plates, one per thread

The name isn’t metaphorical — it really is a stack data structure (LIFO). Every time a method gets called, the JVM pushes a new frame onto the stack. When the method returns, that frame is popped off and discarded.

Picture a stack of dinner plates. Each plate is a method call. The top plate is the method currently running. When that method calls another method, a new plate gets stacked on top. When a method finishes, its plate is removed from the top. The bottom plate (main()) stays until the program ends.

What’s on each plate (frame)?

  • Local variables declared inside that method
  • Method parameters passed in
  • The return address — where to go back to when this method finishes

Watch a stack grow and shrink

public void a() {
    int x = 10;
    b();              // pushes b's frame on top of a's
}

public void b() {
    int y = 20;
    c();              // pushes c's frame on top of b's
}

public void c() {
    int z = 30;
    // at this exact moment, stack has 3 frames: a, b, c
}

When c() returns, its frame is popped instantly — z is gone. No garbage collector needed. This is why stack allocation is essentially free: pushing/popping is just moving a pointer up or down.

Two consequences worth burning in

  1. Each thread has its own stack. A 100-thread server has 100 stacks. This is why thread count × stack size = real memory cost.
  2. The stack is small. Default ~512KB–1MB per thread. Recurse too deep and you get StackOverflowError — not because your computer is out of RAM, but because that one thread’s tiny stack filled up.

STAGE 3 12 MIN

The Heap: one shared warehouse for all objects

The heap is where every object your program ever creates with new ends up. Every String, every ArrayList, every custom class. Always.

Unlike the stack, the heap is not organized as LIFO. Objects can be created in any order and freed in any order. They live for as long as something still references them — when no references remain, the Garbage Collector eventually reclaims their space.

Think of the heap as a giant warehouse shared by every worker (thread) in the building. Anyone can put a box on a shelf. Anyone can read from any box (if they know the address). A cleanup crew (the GC) periodically walks the warehouse, finds boxes nobody is using anymore, and removes them.

The key distinction that trips up most engineers

When you write Order order = new Order(100);, two things happen in two different places:

The variable named order on the stack is not the object. It’s a reference — a small fixed-size value (typically 4 or 8 bytes) that says “the object lives at this address in the heap.”

This is the single most important idea in today’s lesson. References on stack. Objects on heap. Always.

What about fields inside the object?

This is where people often get confused. Look at the Order object in the diagram above. It contains total = 100 — a primitive int. So is total on the stack?

No. total is a field of the Order object. The Order object lives on the heap. Therefore total lives on the heap too, as part of the object’s body.

The rule: “primitives on stack” only applies when the primitive is a local variable inside a method. If it’s a field of an object, it lives wherever the object lives — which is the heap.

Now let’s lock it in with a direct comparison. Read every row.

Notice the symmetry of trade-offs. The stack is fast and self-cleaning, but small and rigid. The heap is flexible and large, but expensive and needs a GC. Neither is better. Each exists because each is right for what it stores.

STAGE 5 15 MIN

Why an architect must internalize this

If you only needed to write working code, “objects on heap, locals on stack” would be enough. But you’re training for architecture decisions. Here’s where this knowledge starts paying back.

1. Diagnosing production failures

You will, eventually, get paged at 2 AM because production crashed with one of these two errors. The cause and the fix are completely different:

ErrorWhat it meansWhere to look
StackOverflowError A single thread’s stack ran out of frames Recursion that doesn’t terminate, or excessively deep call chains
OutOfMemoryError The shared heap ran out of room Memory leaks, oversized caches, large in-memory data

2. Capacity planning for high-throughput services

Consider a server handling 10,000 concurrent requests. If each request runs on its own platform thread with a 1MB stack:

10,000 threads × 1MB stack = 10 GB just for stacks
                                   // before a single object is allocated on the heap

This is a real production-scale calculation. It’s exactly why virtual threads (Project Loom, Day 12) were invented — they have tiny stacks that live on the heap, sidestepping this entire problem.

3. Allocation patterns affect GC pressure

Every object you create on the heap eventually needs to be collected. Code that creates millions of short-lived objects per second forces the GC to run constantly, which steals CPU and causes latency spikes.

4. Concurrency safety follows from location

Stack data is private to one thread — it is automatically thread-safe. Heap data is shared — it needs locks, atomics, or immutability to stay safe.

INTERACTIVE CHALLENGE · STAGE 5

Where does each piece of data live?

Given this code:

public void m() {
    int a = 5;
    String s = new String("hi");
}

Tell me where three things live. Use the dropdowns — only two regions are possible here (Metaspace doesn’t apply).

Local primitive int a = 5 — the value lives directly in the method's stack frame.

The reference variable s is a local variable — 8 bytes in the method's stack frame. The object it points to is a separate question.

Every object created with new goes to the heap — no exceptions. The variable s is on the stack; the object it references is on the heap.

Key Takeaways
  • Every local variable (primitive or reference) lives on the stack — the stack frame is destroyed when the method returns.
  • Every object created with new lives on the heap — it outlives the method that created it.
  • A reference on the stack is just a pointer (8 bytes) — the object it points to is a separate heap allocation. Never confuse the two.
