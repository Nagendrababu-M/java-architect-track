Profiling in Anger: JFR, async-profiler & Flame Graphs How to find why a service is slow — the methodology, the tools, and how to read the evidence.

schedule 160 min · layers 6 stages
RECAP FROM DAYS 4–5

You can read a GC log and reason about the memory model. But when a service is “slow,” the GC log is only one suspect. Today is the diagnostic discipline: how to figure out where the time actually goes — CPU, locks, I/O, GC, or allocation — using profiling. Metrics tell you a service is slow; profiling tells you which line of code is to blame. An architect who can’t profile is reduced to guessing, and guessing in production is expensive. The goal of today: never again say “I think it’s the database” without evidence.

STAGE 1 25 MIN

The diagnostic mindset — what kind of “slow” is it?

“The service is slow” is not a diagnosis — it’s a symptom. The first job is to classify the bottleneck, because each class has a different tool and a different fix. There are five, and confusing them is why teams burn days optimizing the wrong thing.

Bound bySymptomTool that finds it
CPUCores pegged at 100%, high CPU time per requestCPU profiler (async-profiler, JFR) → flame graph
Lock / contentionCores idle but latency high; threads BLOCKEDLock profiler, thread dumps, off-CPU profiling
I/OThreads WAITING on sockets/DB; low CPUWall-clock profiling, thread dumps, DB/trace spans
GCLatency spikes correlated with pausesGC logs (Day 4), JFR GC events
AllocationHigh GC frequency, high churn, growing pausesAllocation profiler → allocation flame graph

Suspect before you measure — but never conclude before you measure.

Form a hypothesis from the symptom (idle CPU + high latency → suspect locks or I/O, not CPU), then pick the tool that can confirm or kill it. The cardinal sin is optimizing on intuition: the bottleneck is almost never where experienced engineers first guess, which is exactly why the profile exists. “Measure, don’t guess” is the single most repeated rule in performance work.

The diagnostic ladder

Move from cheap-and-broad to expensive-and-precise:

  1. Metrics (dashboards, p99 latency, CPU %, GC time) — tells you that there’s a problem and roughly which resource.
  2. JFR / continuous profile — always-on, low overhead; the first thing you pull when an alert fires.
  3. Targeted profiler (async-profiler in the right mode) — zoom into the suspected dimension (CPU, alloc, lock, wall).
  4. Thread / heap dump — point-in-time snapshots for deadlocks, blocked threads, and leaks.

You don’t start at the bottom. You start at metrics, form a hypothesis, and descend only as far as you need.

STAGE 2 30 MIN

The built-in toolbox — jcmd, thread dumps & heap dumps

Before reaching for a profiler, know the tools shipped inside every JDK. They’re free, production-safe, and often enough on their own.

jcmd — the swiss-army knife

# Discover the PID, then ask the JVM what it can do
jcmd -l                                  # list running JVMs
jcmd <PID> help                          # every diagnostic command available
jcmd <PID> Thread.print                   # thread dump
jcmd <PID> GC.heap_info                   # live heap occupancy
jcmd <PID> VM.flags                       # effective JVM flags
jcmd <PID> JFR.start duration=60s filename=rec.jfr   # start a recording

Thread dumps — for lock & I/O problems

A thread dump is a snapshot of every thread’s stack and state. It’s the fastest way to diagnose a hung or contended service. Read the thread states:

  • RUNNABLE — actually executing (or in native/IO). Many RUNNABLE in the same method = a CPU hot spot.
  • BLOCKED — waiting to acquire a monitor someone else holds. Many BLOCKED on one lock = contention.
  • WAITING / TIMED_WAITING — parked, often on I/O, a pool, or a condition. Many here with idle CPU = I/O-bound or pool-starved.

Take three dumps a few seconds apart: threads stuck in the same frame across all three are your real problem; threads that move are fine. The JVM also auto-detects deadlocks in a thread dump (“Found one Java-level deadlock”) with the exact lock cycle — the fastest deadlock diagnosis there is.

Heap dumps — for leaks & memory bloat

# Capture the full heap (pauses the JVM briefly — use with care in prod)
jcmd <PID> GC.heap_dump /tmp/heap.hprof
# Or automatically, on the OOM that matters most:
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/log/app/

Open the .hprof in Eclipse MAT (Memory Analyzer Tool). Its “Leak Suspects” report ranks the objects holding the most retained heap — the memory that would be freed if that object were collected. Follow the “path to GC roots” to find what is keeping the leak alive (the classic answer from Day 4: an unbounded cache or a static collection).

🎯 INTERVIEW QUESTIONS

Q: A service is unresponsive but CPU is near zero. First move? — Take a few thread dumps (jcmd Thread.print) seconds apart. Look for many BLOCKED threads (contention/deadlock) or WAITING threads (I/O or pool starvation). The JVM flags deadlocks automatically.

Q: Retained vs shallow heap? — Shallow = the object’s own size; retained = everything that becomes unreachable (and thus freeable) if that object is collected. Leak hunting is about retained size.

Q: How do you find a deadlock in production? — A thread dump; the JVM reports the deadlock cycle and the exact locks/threads involved.

QUICK CHECK

STAGE 3 30 MIN

Java Flight Recorder — always-on, production-safe profiling

JFR is the JVM’s built-in event recorder. Since Java 11 it is free and open-source (it was a commercial feature in Java 8 — a detail worth knowing). Its defining trait: near-zero overhead — typically under 1% — low enough to run continuously in production.

What it records

JFR is event-based, not just a sampler. It captures a continuous ring buffer of thousands of event types: GC pauses, allocations, method samples, lock contention, thread parks, I/O, exceptions, JIT compilation, even custom application events. Because it’s a ring buffer, you can dump the last N minutes after the incident already happened — the killer feature for intermittent problems you can’t reproduce on demand.

# Attach to a running process and record 60s
jcmd <PID> JFR.start duration=60s settings=profile filename=rec.jfr

# Or enable at startup, continuous, dump on demand
java -XX:StartFlightRecording=disk=true,maxsize=200m,settings=profile MyApp
jcmd <PID> JFR.dump filename=rec.jfr     # grab the buffer right now

Two built-in configs: default (≈<1% overhead, safe everywhere) and profile (more detail, slightly higher overhead, fine for short investigations).

Analyzing in JDK Mission Control (JMC)

Open the .jfr in JMC. The four views you check first map directly onto Stage 1’s bottleneck classes:

  • GC events — pause durations, frequency, heap before/after. Confirms or kills the GC hypothesis.
  • Memory / TLAB allocation — which classes and call sites allocate the most bytes; sustained climbs reveal leaks and churn.
  • Method profiling (hot methods / flame graph) — where CPU time goes.
  • Lock instances / thread parks — which monitors block which threads, for how long.

Continuous profiling is the modern standard. Always-on JFR (or a continuous profiler like Grafana Pyroscope/Parca built on the same idea) means that when an alert fires at 3 a.m., the evidence is already captured — you dump the buffer instead of trying to reproduce. “We’ll add profiling and wait for it to happen again” is the slow, expensive path. Architects design observability in before the incident.

🎯 INTERVIEW QUESTIONS

Q: Is it safe to profile in production? — With JFR, yes — the default config is <1% overhead and designed for it. That’s the whole point: capture real production behavior, which staging rarely reproduces.

Q: A latency spike happened 5 minutes ago and is gone. How do you investigate? — If continuous JFR is on, dump the ring buffer (JFR.dump) — it still holds the window that covers the spike. No reproduction needed.

Q: What changed about JFR between Java 8 and 11? — It became free and open-source; in 8 it required -XX:+UnlockCommercialFeatures. That’s why always-on production profiling only became standard practice from 11.

STAGE 4 25 MIN

Sampling vs instrumentation, safepoint bias & profiling modes

Two ways to profile — and their trade-off

SamplingInstrumentation
HowPeriodically capture every thread’s stackInject timing code into every method entry/exit
OverheadLow, tunable (e.g. every 10ms)High, and it distorts what it measures
AccuracyStatistical — hot paths dominate samplesExact counts, but skews hot tiny methods
UseDefault for finding hot spots in prodRare; exact call counts of specific code

The instrumentation trap: adding code to every method makes cheap, frequently-called methods look expensive (the measurement cost dominates their real cost), and inlining is defeated — so the profile lies. Sampling is the production default for exactly this reason.

Safepoint bias — why most Java samplers lie.

Traditional Java profilers (and JFR’s method sampler, to a degree) can only capture a stack at a safepoint — a point where the JVM has paused the thread in a known state. But the JIT only inserts safepoints at certain spots (loop back-edges, method calls). A hot tight loop with no safepoint inside never gets sampled there — so the real hot code is invisible and the samples pile up at the next safepoint, blaming the wrong method. This is safepoint bias: the profile is systematically skewed toward safepoint locations.

async-profiler — sampling without the bias

async-profiler sidesteps safepoint bias by using OS-level signals (perf_events / SIGPROF) and the JVM’s AsyncGetCallTrace API to capture a thread’s real stack at any instruction, not just at safepoints. It catches the tight loops and JNI-heavy code JFR can miss, at very low overhead.

# CPU flame graph for 30s
asprof -d 30 -f cpu.html <PID>
# Allocation profiling — which call sites churn the heap (ties to GC, Day 4)
asprof -e alloc -d 30 -f alloc.html <PID>
# Lock contention — where threads block
asprof -e lock -d 30 -f lock.html <PID>
# Wall-clock — includes time spent BLOCKED/WAITING (finds I/O the CPU profile hides)
asprof -e wall -d 30 -f wall.html <PID>

The modes map onto Stage 1’s bottlenecks

This is the connective tissue of the whole day: CPU mode finds CPU-bound hot code; alloc mode finds allocation churn driving GC; lock mode finds contention; wall-clock mode includes blocked/waiting time and so finds I/O and lock waits that a pure CPU profile renders invisible. Pick the mode that matches your hypothesis from Stage 1.

🎯 INTERVIEW QUESTIONS

Q: What is safepoint bias and which tool avoids it? — Samplers that only capture at safepoints miss code that never hits one (tight loops), skewing the profile to safepoint locations. async-profiler avoids it via AsyncGetCallTrace + OS signals, sampling at arbitrary instructions.

Q: Your CPU flame graph looks flat but latency is terrible. What mode do you run next? — Wall-clock (or lock) profiling — the time is being spent off-CPU, blocked or waiting, which a CPU profile cannot show.

Q: Why not instrument every method for exact numbers? — The instrumentation overhead distorts the result, especially for small hot methods, and defeats inlining. Sampling gives a truthful statistical picture at low cost.

STAGE 5 20 MIN

Reading flame graphs

A flame graph is the standard visualization for a sampled profile. Once you can read one, diagnosis becomes fast and visual.

How to read it — four rules

  • Each box is a stack frame (a method). The box below called the box above.
  • Width = time (sample count). A frame twice as wide consumed twice the resource. This is the only axis that means anything quantitatively.
  • Y-axis = stack depth, not time. Left-to-right ordering is alphabetical/merge order, not chronological — do not read it as a timeline.
  • Look for wide plateaus. A wide box near the top with little above it is where the resource is actually spent — the leaf doing the work. Wide-but-deep towers are just call depth; the top of a wide stack is the culprit.

Variations worth knowing

  • CPU flame graph — width = CPU time. Finds compute hot spots.
  • Allocation flame graph — width = bytes allocated. Finds GC-pressure sources (ties to Days 3–4).
  • Off-CPU / wall-clock flame graph — width = time blocked/waiting. Finds I/O and lock waits.
  • Differential flame graph — colors the change between two profiles (before/after a deploy). The fastest way to spot a regression: the new hot frame lights up red.
  • Icicle graph — the same thing flipped (root on top); JMC uses this orientation.

The workflow: widest frame at the top of a tower → that’s where the resource goes → click to zoom → read the call path below it to understand why it’s hot → fix the call site, not the leaf. A method being hot is often the symptom; the bug is the caller invoking it too often (e.g. recompiling a regex per request).

QUICK CHECK

In a flame graph, frame A is very wide but has a tall stack of frames above it; frame B is slightly narrower with nothing above it. Which one is consuming the resource, and why?

MODEL ANSWER

The bug is in the caller, not in Pattern.compile. matchSymbol() is calling Pattern.compile(regex) on every invocation — recompiling the same regular expression for every trade, when compilation is the expensive step and matching is cheap. The fix: compile each pattern once and reuse it — hoist it to a private static final Pattern (or cache compiled patterns by key). Compiled Pattern objects are thread-safe for matching, so a static field is safe. This is the classic “the hot leaf is a symptom; the bug is calling it too often” pattern from Stage 5 — you fix the call site, not the leaf. Expect CPU to drop dramatically and throughput to roughly double, matching the capacity prediction.

This is the Stage 1 classification in action. CPU pegged at 100% with threads RUNNABLE (not BLOCKED or WAITING) rules out lock contention and I/O — the time is being spent computing, not waiting. Had threads been BLOCKED, a CPU profile would have looked misleadingly idle and you’d have needed lock/wall-clock mode; had the GC log been busy, allocation profiling would have been the move. 🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Classify the “slow” first — CPU, lock, I/O, GC, or allocation. Each has a different tool. Suspect, then measure; never conclude on intuition.
  • The built-in toolbox is often enough — jcmd, thread dumps (3 of them, seconds apart) for locks/deadlocks/I/O, heap dumps + Eclipse MAT for leaks.
  • JFR is always-on, <1% overhead, production-safe — dump the ring buffer after the incident. Continuous profiling means the evidence is already captured.
  • Sampling beats instrumentation for finding hot spots; async-profiler avoids safepoint bias and offers CPU / alloc / lock / wall-clock modes matching each bottleneck class.
  • Flame graphs: width = time, y = stack depth, not a timeline. The widest leaf is where the resource goes; fix the call site, not the leaf.
