STAGE 1 20 MIN

Lambda internals — what a lambda really is

Everyone can write a lambda. Senior interviews ask what the JVM does with it.

The compiler does not generate a class file per lambda. It emits an invokedynamic instruction.

The lambda body becomes a private static method in the enclosing class. The invokedynamic call site, on first execution, calls a bootstrap method — LambdaMetafactory — which spins a small implementation class at runtime and links it. Subsequent executions hit the linked call site directly. An anonymous class, by contrast, is a compile-time-fixed .class file with an allocation on every new.

Consequences worth stating in an interview:

  • Non-capturing lambdas are cached singletons — x -> x * 2 is created once, ever. Zero allocation per call.
  • Capturing lambdas allocate per evaluation — x -> x * factor must carry factor, so it builds an object each time the expression runs. This is the hot-path cost model.
  • The strategy lives in the JVM, not your class files — the platform can make lambdas faster without recompiling your code.

Capture semantics: effectively final

A lambda can only capture locals that are effectively final. Captured locals are copied into the lambda at creation; allowing reassignment would mean the copy and the original silently diverge, so Java forbids it. Fields are read live through this (not copied), which is why they don’t need to be final — and why mutating a field from a lambda in a parallel stream is a data race.

🎯 INTERVIEW QUESTIONS

Q: What does invokedynamic do for lambdas? — Defers lambda-class creation to runtime via LambdaMetafactory, avoiding a compile-time class-per-lambda, enabling singleton caching of non-capturing lambdas, and letting the JVM evolve the strategy.

Q: Is a lambda just sugar for an anonymous class? — No. Anonymous class = compile-time class file + allocation per instantiation + its own this. Lambda = invokedynamic, singleton if non-capturing, lexical this.

STAGE 2 20 MIN

Stream internals — the lazy pipeline

Intermediate operations do nothing. They build a linked chain of stages. All work happens when the terminal operation pulls.

When collect() runs, the pipeline is fused: each element flows through all stages one at a time — filter, then map, then accumulate — before the next element starts. There is no intermediate collection between stages. One pass, no materialization, and short-circuit operations (findFirst, anyMatch, limit) can stop pulling the instant they’re satisfied — which is why a filtered findFirst terminates on an infinite stream.

Spliterators — the engine

Every source provides a Spliterator. tryAdvance(consumer) pushes one element through the pipeline; trySplit() carves off a chunk for another thread (the basis of parallel streams). It reports characteristics — SIZED, SUBSIZED, ORDERED, SORTED, DISTINCT, IMMUTABLE — that let the pipeline skip work (distinct() on a DISTINCT source is a no-op; toArray() on a SIZED source preallocates exactly).

One-pass & non-reusable

A stream is a one-shot pipeline over a spliterator, not a data structure. The terminal op consumes the spliterator; a second terminal op throws IllegalStateException. Re-create from the source, or hold a Supplier<Stream>.

Intermediate vs terminal; stateless vs stateful

  • Stateless intermediate (filter, map, flatMap): each element independent — fuses and parallelizes perfectly.
  • Stateful intermediate (sorted, distinct, limit): must see some/all elements first — sorted buffers the entire stream, breaking the one-pass model and killing parallel scalability.
  • Terminal (collect, reduce, forEach, findFirst): triggers execution and produces a result or side effect.
🎯 INTERVIEW QUESTIONS

Q: Why are streams not reusable? — A stream is a one-time view over a spliterator; the terminal operation consumes it. Reusing throws IllegalStateException. Streams aren’t collections.

Q: Does peek always run? — No. Laziness + short-circuiting means un-pulled elements are never peeked, and ops may be elided entirely (count() on a SIZED source can skip traversal). Never use peek for required side effects.

STAGE 3 20 MIN

Collectors deep dive — from groupingBy to your own

grouping & partitioning

// Count per symbol
trades.stream().collect(groupingBy(Trade::symbol, counting()));
// Multi-level: symbol → side → list
trades.stream().collect(groupingBy(Trade::symbol, groupingBy(Trade::side)));
// partitioningBy — a boolean split, always two keys
trades.stream().collect(partitioningBy(t -> t.qty() > 10_000));

Downstream composition: teeing, mapMulti, flatMapping

// teeing (12): two collectors, one pass — min and max together
trades.stream().collect(teeing(
    minBy(comparing(Trade::price)), maxBy(comparing(Trade::price)), PriceRange::new));

// flatMapping (9) as downstream — flatten inside a group
orders.stream().collect(groupingBy(Order::customer,
    flatMapping(o -> o.lines().stream(), toList())));

// mapMulti (16): allocation-free 1→0-or-many, no per-element Stream object
orders.stream().<OrderLine>mapMulti((o, push) -> o.lines().forEach(push)).toList();

Writing a custom Collector — the 5-function recipe

Collector.of(
    () -> new StringJoiner(", "),     // 1. supplier — the mutable container
    (j, s) -> j.add(s),                  // 2. accumulator — fold one element in
    (j1, j2) -> j1.merge(j2),            // 3. combiner — merge two containers (PARALLEL)
    StringJoiner::toString               // 4. finisher — container → result
    // 5. characteristics — IDENTITY_FINISH, CONCURRENT, UNORDERED
);

The interview point: the combiner is load-bearing. Sequential streams never call it, so a broken combiner “works” in every test and then silently produces garbage under parallelStream().

🎯 INTERVIEW QUESTIONS

Q: toMap throws on duplicate keys — fix? — The 3-arg form with a merge function: toMap(Trade::symbol, Trade::price, (a, b) -> b). Forgetting it is a top-5 production stream bug.

Q: reduce vs collect? — reduce = immutable accumulation (new value each step; right for ints, O(n²) for strings). collect = mutable reduction: one container per thread, mutated in place, merged.

STAGE 4 20 MIN

Parallel streams — when they help, when they hurt

Underneath: the spliterator trySplit()s the source into chunks, chunks run as ForkJoin tasks, and results merge via the collector’s combiner.

The common-pool trap: every parallel stream shares ForkJoinPool.commonPool(), sized cores−1, JVM-wide.

Every parallel stream and every default CompletableFuture async shares these threads. One parallel stream doing blocking I/O starves every other parallel computation in the process. This is the #1 production parallel-stream incident.

Parallel only if ALL of these hold

  • N × Q is large — element count times per-element cost. Below ~100k cheap elements, fork/join orchestration costs more than it saves.
  • The source splits well — ArrayList/arrays split perfectly; LinkedList and Stream.iterate split terribly or not at all; LongStream.range is ideal.
  • Operations are stateless & non-blocking — CPU-bound pure functions, no I/O, no locks, no shared mutable state.
  • The merge is cheap — summing longs is O(1); merging HashMaps is O(n). Consider groupingByConcurrent when order doesn’t matter.
  • No encounter-order dependence — limit/findFirst force coordination; findAny/unordered release it.
// BROKEN under parallel — ArrayList isn't thread-safe
List<String> out = new ArrayList<>();
data.parallelStream().forEach(out::add);          // ✗ data race
// CORRECT — the collector owns the state, one container per thread + merge
List<String> out = data.parallelStream().toList();   // ✓
🎯 INTERVIEW QUESTIONS

Q: When would parallel streams hurt a production service? — In a request handler: they grab the shared common pool, so under load they starve the whole JVM (the Stage 7 challenge). Also: small N, badly-splitting source, stateful ops, expensive merge.

Q: Where do parallel-stream threads come from? — ForkJoinPool.commonPool(), sized cores−1, JVM-wide; the calling thread also participates. To isolate, submit the pipeline from inside a dedicated ForkJoinPool.

STAGE 5 15 MIN

Java 21 (2023, LTS) — pattern matching goes all the way

The Day 8 foundations (sealed types, records, pattern instanceof) reach their conclusion here: full pattern matching for switch and record deconstruction, both standard in 21.

┃ PATTERN MATCHING FOR switch (standard)

return switch (result) {
    case Approved(var id, var amt)             -> settle(id, amt);
    case Declined(var r, var retry) when retry -> requeue(r);   // guarded: 'when'
    case Declined(var r, var x)               -> reject(r);
    case null                                  -> handleNull();   // explicit null case
    case Pending p                            -> defer(p);
};  // sealed ⇒ exhaustive ⇒ no default needed

Type patterns and the death of the cast-after-test

A type pattern (case Approved a) does three things at once: tests the runtime type, and if it matches, binds a new variable of that type already narrowed — no explicit cast, and the binding is only in scope on that branch. This is the same flow-scoping you saw with pattern instanceof in Stage 1, now lifted into switch. The practical payoff: the long if (x instanceof A) { A a = (A) x; ... } else if (x instanceof B) ... chain — historically the place where a stale cast and a stale test silently drift apart — collapses into a switch where the compiler guarantees test and binding can never disagree.

Guarded patterns and dominance — order is a compile-time contract

A guard (when retry) refines a pattern with a boolean. The subtlety interviewers push on: cases are evaluated top-to-bottom, and the compiler enforces dominance — a broader pattern placed above a narrower one that it subsumes is a compile error, because the narrower case could never be reached. Concretely, the guarded case Declined(var r, var retry) when retry must appear before the unguarded case Declined(var r, var x); reverse them and the unguarded case dominates, making the guard dead code, and the compiler rejects it. This is not a style rule — it is the compiler preventing a whole class of unreachable-branch bugs.

Null handling — a deliberate semantic change

Historically switch threw NullPointerException on a null selector before any case ran. Java 21 keeps that default unless you write an explicit case null — only then does null flow into the switch. The reasoning: exhaustiveness reasons about types, and null inhabits every reference type, so silently treating null as “some case” would be unsound. Making case null explicit forces you to decide null’s fate consciously. You can also combine it: case null, default -> ... handles null and everything-else together.

Exhaustiveness — how the compiler actually proves it

Over a sealed type, the compiler walks the permits list (recursively, through nested sealed types) and verifies every permitted subtype is matched by some case. If it can prove total coverage, the switch is legal with no default; the compiler even synthesizes a hidden fallback that throws MatchException for the theoretically-impossible case of a subtype added via separate compilation. This is why omitting default is the correct choice on a closed hierarchy: default would silently absorb future subtypes, while its absence turns “a new subtype was added” into a build break that names every switch needing an update.

┃ RECORD PATTERNS (standard)

A record pattern matches a type and deconstructs it into its components in one move, binding each component to a variable. Because records publish their exact component list, the compiler knows the deconstruction is total — and patterns nest, so you can destructure a whole object graph in a single line:

// Flat: match Approved and pull out both components
case Approved(String txnId, BigDecimal amount) -> settle(txnId, amount);

// Nested: destructure two levels deep — no intermediate getters
if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
    double len = Math.hypot(x2 - x1, y2 - y1);   // all four bound at once
}

Two depth points. First, var in a record pattern infers each component’s declared type — you don’t restate BigDecimal. Second, record patterns and sealed switches compose into the complete algebraic-data-type story from Day 8: sealed gives the closed set, the record pattern destructures each shape, exhaustiveness proves nothing is missed. That triad is what replaces the visitor pattern entirely — and unlike a visitor, adding a variant doesn’t require touching an interface every implementor must re-implement; it just lights up the compile errors at the switches that care.

Why this matters beyond syntax. Pattern matching for switch is the feature that finally lets Java do data-oriented programming — behavior kept outside the data, dispatched by shape, verified exhaustive at compile time. It is the deep reason records and sealed classes were worth adding: alone they’re nicer data holders; together with pattern switch they change how you architect a domain model. The Day 8 challenge built the types; this is the machinery that consumes them safely.

🎯 INTERVIEW QUESTIONS

Q: Explain exhaustiveness checking and why it matters for refactoring. — Over a sealed type the compiler walks the permits list and proves every subtype is covered; a switch with no default then compiles. Add a subtype and every such switch breaks the build, naming exactly where to update — “forgot a case” moves from a production incident to a compile error. A default branch silently defeats this, which is why you omit it on closed hierarchies.

Q: Why must a guarded case precede its unguarded form? — Dominance: the unguarded pattern subsumes the guarded one, so if it came first the guarded case is unreachable. The compiler rejects unreachable patterns to prevent silent dead branches.

Q: What happens if you switch over a sealed type and pass null? — NPE before any case, unless you add an explicit case null (optionally case null, default). Exhaustiveness covers types, not null, because null inhabits every reference type.

Q: Pattern switch vs the visitor pattern? — Both dispatch over a closed set. Visitor scatters behavior across an accept method on every type and a growing visitor interface; pattern switch keeps each operation in one local, readable place and gets compiler-checked exhaustiveness for free. Visitor wins only when the type set is open to third parties — where sealed wouldn’t apply anyway.

STAGE 6 15 MIN

Java 21 — virtual threads, structured concurrency & the platform

┃ VIRTUAL THREADS (standard) — the headline feature

// One virtual thread per task — the executor creates them on demand
try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (Order o : orders) exec.submit(() -> enrich(o));  // 100k blocking calls? Fine.
}
Thread.ofVirtual().start(() -> doWork());

The platform-thread problem they solve

A platform thread is a thin wrapper over an OS thread: ~1 MB of reserved stack, scheduled by the kernel, and expensive to create. That caps you at a few thousand. So for high-concurrency I/O, the industry went asynchronous — callbacks, futures, reactive — trading readable, debuggable, blocking code for throughput. Virtual threads remove that trade: they make blocking cheap.

How the scheduling actually works (M:N)

A virtual thread is a Java object holding a continuation plus a small heap-allocated stack — not an OS thread. Many virtual threads are multiplexed onto a small pool of carrier threads (platform threads in a dedicated ForkJoinPool, defaulting to the CPU count). The lifecycle:

  • Mount: to run, a virtual thread mounts onto a carrier — its stack frames are copied onto the carrier’s stack and it executes as normal Java code.
  • Unmount: when it hits a blocking call the JDK has instrumented (socket read, Thread.sleep, most java.util.concurrent waits, blocking I/O), the runtime captures its continuation, copies its stack back to the heap, and unmounts it — freeing the carrier to run another virtual thread immediately.
  • Resume: when the I/O completes, the virtual thread is rescheduled onto any free carrier and continues from exactly where it parked.

The result: code is written in plain blocking style (var resp = client.send(req)), but the carrier never actually blocks — so a handful of OS threads serve hundreds of thousands of concurrent in-flight operations. Thread-per-request becomes a sane architecture again.

Pinning — the one failure mode to know cold. A virtual thread that cannot unmount while blocked is pinned to its carrier, which sits idle — reintroducing the exact thread-starvation virtual threads were meant to eliminate. The two causes: (1) blocking inside a synchronized block or method — the JDK can’t safely unmount across a monitor; (2) blocking in a native (JNI) call. Fix: replace synchronized with ReentrantLock in hot, blocking paths — the lock’s park is unmount-aware. Detect it with -Djdk.tracePinnedThreads=full, which prints a stack trace at every pinning event. Pinning during brief, non-blocking critical sections is harmless; pinning around I/O is the bug.

What virtual threads are NOT: not faster for CPU-bound work (you’re still limited by cores — they help concurrency, not compute); not pooled (you create one per task and let it die — pooling them is an anti-pattern); and not a reason to size them with semaphores unless you’re protecting a limited downstream resource (a DB connection pool), in which case you bound concurrency at that resource, not at the threads.

┃ STRUCTURED CONCURRENCY (preview)

Virtual threads make spawning concurrency cheap; structured concurrency makes it safe. The problem it fixes: with a bare executor, if you fork two subtasks and the first fails, the second keeps running — orphaned, burning a connection, its result destined for the garbage. Error handling and cancellation become manual and leaky. Structured concurrency gives concurrent tasks the same lifecycle discipline as a code block.

try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var user  = scope.fork(() -> fetchUser(id));    // each fork → its own virtual thread
    var order = scope.fork(() -> fetchOrder(id));
    scope.join();                                  // wait for ALL forks to finish/fail
    scope.throwIfFailed();                          // propagate the first failure as an exception
    return new Response(user.get(), order.get());     // both guaranteed complete here
}  // scope close: no subtask can outlive this block
  • ShutdownOnFailure — the “I need ALL results” policy: the first subtask to throw triggers cancellation (interruption) of its still-running siblings, so you don’t wait on work whose result you’ll discard.
  • ShutdownOnSuccess — the “I need ANY result” policy: the first success cancels the rest. This is hedged requests — fire the same query at two replicas, take the faster, cancel the slower.
  • The structural guarantee: the try-with-resources block cannot exit until every fork has joined. Cancellation propagates down to children, exceptions propagate up to the parent, and a thread dump shows the real parent–child tree instead of a flat pool — observability that bare executors never gave you.

┃ SCOPED VALUES (preview)

ThreadLocal was the old way to carry per-request context (a request id, a tenant, a security principal) without threading it through every method signature. It has three flaws that virtual threads turn fatal: it’s mutable anywhere downstream (action at a distance), it has an unbounded lifetime until explicitly removed (leaks, and a forgotten remove() in a pooled thread is a classic bug), and with a million virtual threads each carrying a mutable map, the memory cost explodes.

static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();

ScopedValue.where(REQUEST_ID, "req-8861").run(() -> handle());
// Anywhere inside handle() and everything it transitively calls:
String id = REQUEST_ID.get();   // reads the value bound for THIS dynamic scope

Scoped values flip every flaw: the binding is immutable (you can only rebind in a nested scope, never mutate), its lifetime is exactly the dynamic extent of the run/call — automatically torn down when the block exits, so no leaks — and they’re inherited by forked subtasks in a StructuredTaskScope, so request context flows correctly across the fan-out. The two features are co-designed: scoped values are the context mechanism for the structured-concurrency + virtual-thread world.

┃ SEQUENCED COLLECTIONS (standard)

A twenty-year inconsistency, fixed. “First element” was spelled differently on every collection (list.get(0), deque.getFirst(), sortedSet.first()) and LinkedHashSet had no way to reach its last element without a full iteration. JEP 431 retrofits three interfaces — SequencedCollection, SequencedSet, SequencedMap — into the existing hierarchy:

// One vocabulary across List, Deque, LinkedHashSet, LinkedHashMap, SortedSet/Map
history.addFirst(latest);  history.removeLast();  var newest = history.getFirst();
for (var e : auditLog.reversed().entrySet()) { ... }   // descending iteration

The standout is reversed(): it returns a view, not a copy — O(1) to create, and writes through to the backing collection. Iterating a LinkedHashMap newest-first used to require copying keys into a list and reversing; now it’s allocation-free. One sharp edge worth knowing: positional writes like addFirst on a SortedSet throw UnsupportedOperationException — a sorted set’s order comes from its comparator, so you can read its ends but can’t impose a position.

┃ GENERATIONAL ZGC (standard)

ZGC (Day 4) already gave sub-millisecond pauses on huge heaps via colored pointers and load barriers. Java 21 makes it generational — separate young and old regions — exploiting the weak generational hypothesis from Day 3 (most objects die young). Collecting the young generation far more often than the old one means less total GC work, lower CPU overhead, and lower allocation stalls, all while keeping the sub-ms pause guarantee. It becomes the recommended default ZGC mode: the same latency, materially better throughput. This ties the whole memory arc together — Day 3’s hypothesis, Day 4’s ZGC mechanics, now the production-grade refinement.

Coming next: Gatherers (Java 22+ preview)

Collectors customize a stream’s terminal step; Stream.gather(...) finally customizes the intermediate step — windowFixed (batching), windowSliding (moving averages), scan (running totals), and custom Gatherers. Java 21 is the current LTS, so know it exists and what gap it fills — the full treatment waits until it’s standard.

🎯 INTERVIEW QUESTIONS

Q: Walk me through what happens when a virtual thread blocks on a socket read. — The instrumented blocking call captures the thread’s continuation, copies its stack to the heap, and unmounts it from the carrier, which immediately runs another virtual thread. When the read completes the virtual thread is rescheduled onto a free carrier and resumes from the park point. The OS thread never blocked.

Q: What is pinning, what causes it, how do you detect and fix it? — A virtual thread that can’t unmount while blocked, holding its carrier idle. Caused by blocking inside synchronized or a JNI call. Detect with -Djdk.tracePinnedThreads=full; fix by switching hot blocking sections from synchronized to ReentrantLock.

Q: When would you prefer virtual threads over Project Reactor? — For blocking I/O where you want simple, debuggable, synchronous-looking code that still scales to high concurrency. Reactor still wins for backpressure, streaming, and complex async composition; virtual threads remove the “go reactive purely to scale blocking calls” pressure.

Q: Why can’t you use ThreadLocal effectively with virtual threads, and what replaces it? — A million virtual threads each holding mutable, unbounded-lifetime ThreadLocals is a memory and leak hazard. Scoped values replace it: immutable, lifetime bound to a dynamic scope (auto-torn-down), and inherited by structured-concurrency subtasks.

You move a service to virtual threads and throughput barely improves; a profiler shows carriers mostly idle. One synchronized method guards a call to a slow external API. Explain exactly what’s happening and the one-line-ish fix.

STAGE 7 15 MIN

The challenge — diagnose a production incident

A real on-call scenario built from today’s internals. Answer all three parts (50+ characters each), then reveal the model answers.

SCENARIO · READ CAREFULLY BEFORE ANSWERING

The 504s under load

System: a Spring Boot service. A controller handler runs a CPU-light aggregation using list.parallelStream().map(...).collect(...) over a few hundred items per request.

Symptom: under production load the service randomly returns 504 Gateway Timeout. CPU is not pegged. It works perfectly in load tests that hit one endpoint at a time. The same code with a sequential stream never times out.

YOUR ANSWERS · MINIMUM 50 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 50 characters
MODEL ANSWER

Root cause: common ForkJoinPool starvation. parallelStream() submits its tasks to ForkJoinPool.commonPool(), which is shared across the entire JVM and sized at cores−1 (often just 3–7 threads). Each in-flight request consumes pool threads for its parallel pipeline. Under concurrent load, many requests compete for that tiny shared pool; requests queue waiting for a pool thread, latency climbs past the gateway timeout, and you get 504s — even though CPU isn’t pegged, because threads are waiting, not computing. It doesn’t reproduce in single-endpoint tests because one-request-at-a-time never contends for the pool. (If the mapping also did any blocking I/O, the starvation would be far worse, since blocked threads hold pool slots without doing CPU work.)

0 / 50 characters
MODEL ANSWER

Submit the parallel pipeline from inside a dedicated ForkJoinPool: var pool = new ForkJoinPool(8); pool.submit(() -> list.parallelStream().map(...).collect(...)).get(); A parallel stream run from within a custom pool uses that pool’s threads, not the common pool — so this request’s parallelism can no longer starve unrelated work (or be starved by it). Caveats: this relies on a documented-by-behavior (not formally specified) mechanism; you must size and lifecycle-manage the pool (don’t create one per request — make it a managed singleton); and you’ve only contained contention, not removed it — if the dedicated pool is also too small for the load, you’ll starve within it instead. Sizing must reflect actual concurrency × per-request parallelism.

0 / 50 characters
MODEL ANSWER

Just use a sequential stream: list.stream().map(...).collect(...). Run the checklist — the workload fails it: only a few hundred elements (N too small), CPU-light per element (Q too small), so fork/join orchestration overhead likely exceeded any speedup even before the contention. And in a web handler the server already provides parallelism by running many requests concurrently; parallelizing within each request just multiplies pressure on shared threads. Removing parallel makes each request’s latency predictable and independent — usually the right call for request-scoped work. Parallel streams are justified for batch/CPU-bound work: large N (100k+), expensive per-element computation, a well-splitting source (ArrayList/array), stateless ops, cheap merge, and a context you control (a background job, not a request handler) — ideally on a dedicated pool.

Key Takeaways
  • A lambda is invokedynamic + LambdaMetafactory, not an anonymous class: runtime-linked, singleton when non-capturing, allocating only when it captures.
  • Streams are lazy, fused, one-pass pipelines over spliterators — which is what makes short-circuiting and infinite sources work, and why they’re not reusable.
  • Collectors are the 5-function recipe; the combiner is load-bearing — a collector that can’t merge partial results is broken under parallelism.
  • Parallel streams share the common pool — never in a request handler; CPU-bound, well-splitting, big-N batch work only.
  • Java 21: exhaustive pattern switch + record patterns complete the algebraic-data-type story; virtual threads make blocking code scale; structured concurrency + scoped values are built for that world.
