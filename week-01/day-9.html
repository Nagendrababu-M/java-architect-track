STAGE 1 30 MIN

Functional interfaces & what a lambda really is

SAM types and @FunctionalInterface

A functional interface is any interface with exactly one abstract method (a SAM — Single Abstract Method — type). default and static methods don’t count, and neither do abstract methods that match Object’s public methods (which is why Comparator can declare equals and still be functional). @FunctionalInterface changes nothing at runtime — it makes the compiler fail if someone adds a second abstract method later. It’s a contract annotation, like @Override.

The core four (and their primitive cousins)

InterfaceSignatureUse
Supplier<T>() → TLazy values, factories
Consumer<T>T → voidSide effects (forEach)
Function<T,R>T → RTransformations (map)
Predicate<T>T → booleanFiltering, validation

The primitive specializations — IntFunction, ToLongFunction, DoublePredicate, etc. — exist to avoid autoboxing. Stream<Integer>.map(i -> i * 2) boxes and unboxes on every element; IntStream.map(i -> i * 2) works on raw ints. In a hot loop over millions of elements that’s the difference between memory-bound and CPU-bound.

The four kinds of method references

// 1. Static method
Function<String,Integer> f1 = Integer::parseInt;      // s -> Integer.parseInt(s)
// 2. Instance method of a PARTICULAR object (bound)
Supplier<Integer> f2 = myString::length;               // () -> myString.length()
// 3. Instance method of an ARBITRARY object of a type (unbound — receiver becomes param 1)
Function<String,Integer> f3 = String::length;          // s -> s.length()
// 4. Constructor
Supplier<ArrayList<String>> f4 = ArrayList::new;       // () -> new ArrayList<>()

The bound/unbound distinction (2 vs 3) is the one interviewers test: String::length as a Function<String,Integer> takes the receiver as its argument.

Capture semantics: effectively final

A lambda can only capture local variables that are effectively final — never reassigned after initialization. Why? Captured locals are copied into the lambda object at creation. If the variable could change afterwards, the lambda’s copy and the original would silently diverge — Java forbids the divergence rather than allow the confusion. Fields and statics are not captured (the lambda reads them live through this), which is why they don’t need to be final — and why mutating a field from a lambda in a parallel stream is a data race.

invokedynamic — why a lambda is not an anonymous class

The compiler does not generate a class file per lambda. It emits an invokedynamic instruction.

The lambda body becomes a private static method in the enclosing class. The invokedynamic call site, on first execution, calls a bootstrap method — LambdaMetafactory — which spins a small implementation class at runtime and links it. Subsequent executions hit the linked call site directly. Consequences: no .class file clutter, non-capturing lambdas are cached as singletons (zero allocation per call!), capturing lambdas allocate per evaluation, and the JVM is free to improve the strategy in future releases without recompiling your code. An anonymous class, by contrast, is a compile-time-fixed class file with an allocation on every new.

🎯 INTERVIEW QUESTIONS

Q: Is a lambda just syntax sugar for an anonymous class? — No. Anonymous class = compile-time class file + allocation per instantiation + its own this. Lambda = invokedynamic + LambdaMetafactory at runtime, singleton if non-capturing, and this is lexically the enclosing instance.

Q: Why effectively final? — Locals are captured by value (copied). Allowing mutation would mean the copy and the variable diverge; banning reassignment removes the trap.

Q: When does a lambda allocate? — Only if it captures. x -> x * 2 is created once, ever. x -> x * factor allocates each time the expression is evaluated (it must carry factor).

STAGE 2 30 MIN

Stream internals — the lazy pipeline

A stream pipeline has three parts: a source (collection, array, generator), zero or more intermediate operations (filter, map, sorted), and one terminal operation (collect, reduce, forEach, findFirst). The fact every candidate states — and few can explain — is laziness:

Intermediate operations do nothing. They build a linked chain of pipeline stages. All work happens when the terminal operation pulls.

When collect() runs, the pipeline is fused: each element flows through all stages one at a time — filter, then map, then accumulate — before the next element starts. There is no intermediate collection between filter and map. One pass, no materialization, and short-circuit operations (findFirst, anyMatch, limit) can stop pulling the moment they’re satisfied — which is why Stream.iterate(0, i -> i + 1).filter(...).findFirst() terminates on an infinite stream.

Spliterators — the engine underneath

Every stream source provides a Spliterator (“splittable iterator”). It does two jobs: tryAdvance(consumer) pushes one element through the pipeline (sequential traversal), and trySplit() carves off a chunk for another thread (parallel decomposition). It also reports characteristics — SIZED (knows its element count), SUBSIZED (splits also know theirs), ORDERED (encounter order matters), SORTED, DISTINCT, IMMUTABLE — which the pipeline uses to skip work: distinct() on a DISTINCT source is a no-op; toArray() on a SIZED source preallocates exactly.

Stateless vs stateful operations

KindOperationsCost
Stateless filter, map, flatMap, peek Each element processed independently — fuses perfectly, parallelizes perfectly
Stateful sorted, distinct, limit, skip Must see some/all elements before emitting — sorted buffers the ENTIRE stream; barriers in the pipeline

sorted() in the middle of a pipeline breaks the one-pass model: everything upstream runs to completion into a buffer, the buffer sorts, then downstream resumes. In parallel pipelines, stateful ops are where the scalability dies.

Encounter order

An ordered stream (from a List) promises results in source order. forEach ignores ordering for speed; forEachOrdered enforces it. unordered() explicitly drops the promise — which can genuinely speed up parallel distinct() and limit(), because maintaining “the first N” across threads requires coordination that “any N” doesn’t.

flatMap vs mapMulti (Java 16)

// flatMap: one Stream object allocated per element — expensive when most map to 0–1 results
orders.stream().flatMap(o -> o.lines().stream())

// mapMulti: push results into a consumer — no per-element stream allocation
orders.stream().<OrderLine>mapMulti((o, push) -> o.lines().forEach(push))

mapMulti wins when the mapping is mostly 0-or-1 elements (filter-and-transform in one step) or when the per-element Stream allocation of flatMap shows up in the profile. flatMap stays clearer when each element genuinely expands into a meaningful sub-stream.

🎯 INTERVIEW QUESTIONS

Q: Why can’t you reuse a stream? — A stream is a one-shot pipeline over a spliterator, not a data structure. The terminal op consumes the spliterator; a second terminal op throws IllegalStateException. Re-create from the source (or use a Supplier<Stream>).

Q: Does peek always run? — No. Laziness + short-circuiting means elements never pulled are never peeked, and ops may even be elided entirely (e.g. count() on a SIZED source can skip traversal). Never use peek for required side effects.

Q: map(x -> x*2).filter(x -> x > 10) vs filter first — does order matter? — Semantically here, and always for performance: filter early so fewer elements flow through later stages.

STAGE 3 30 MIN

Collectors — from groupingBy to building your own

The Collector contract

A Collector<T, A, R> is four functions plus characteristics: supplier() creates the mutable container A, accumulator() folds an element T into it, combiner() merges two containers (used by parallel streams), finisher() transforms A into the result R. Characteristics: IDENTITY_FINISH (A is R, skip the finisher), CONCURRENT (one shared container, thread-safe accumulate), UNORDERED.

Downstream collectors — the composition pattern

// Count trades per symbol
Map<String, Long> counts = trades.stream()
    .collect(groupingBy(Trade::symbol, counting()));

// Per symbol: total notional value — groupingBy + mapping-free reduce
Map<String, BigDecimal> notional = trades.stream()
    .collect(groupingBy(Trade::symbol,
             reducing(BigDecimal.ZERO, Trade::value, BigDecimal::add)));

// Multi-level: symbol → side → list of trades (report-style nested grouping)
Map<String, Map<Side, List<Trade>>> bySymbolBySide = trades.stream()
    .collect(groupingBy(Trade::symbol, groupingBy(Trade::side)));

// filtering vs pre-filter: keep EMPTY groups for symbols with no large trades
Map<String, List<Trade>> large = trades.stream()
    .collect(groupingBy(Trade::symbol,
             filtering(t -> t.quantity() > 10_000, toList())));

That last distinction is a classic probe: .filter(...).collect(groupingBy(...)) drops symbols entirely if nothing passes; groupingBy(..., filtering(...)) keeps the symbol with an empty list. Different shapes for different reports.

Collectors.teeing (Java 12)

// Min and max in ONE pass
record PriceRange(Optional<Trade> low, Optional<Trade> high) {}

PriceRange range = trades.stream().collect(teeing(
    minBy(comparing(Trade::price)),
    maxBy(comparing(Trade::price)),
    PriceRange::new));

teeing feeds every element to two collectors and merges their results. The pattern: any time you’re tempted to traverse twice (average + count, sum + max), teeing does it in one pass.

Writing a custom Collector

// Collect into an immutable, comma-joined, capped summary — the 4 functions explicitly
Collector<String, StringJoiner, String> top3Summary = Collector.of(
    () -> new StringJoiner(", "),          // supplier: the mutable container
    (j, s) -> { if (j.length() < 60) j.add(s); },  // accumulator
    (j1, j2) -> j1.merge(j2),                 // combiner: parallel merge
    StringJoiner::toString);                 // finisher

The interview point: the combiner is not optional decoration. If your collector can’t correctly merge two partial containers, it’s broken under parallelStream() even if it “works” sequentially.

🎯 INTERVIEW QUESTIONS

Q: toMap throws on duplicate keys — how do you handle it? — The 3-arg form takes a merge function: toMap(Trade::symbol, Trade::price, (a, b) -> b) (keep latest). Forgetting the merge function is a top-5 production stream bug.

Q: reduce vs collect? — reduce is for immutable accumulation (each step produces a new value — right for ints, wrong for StringBuilder). collect is mutable reduction: one container per thread, mutated in place, merged. Folding strings with reduce(String::concat) is O(n²); joining() is linear.

STAGE 4 30 MIN

Parallel streams — when they hurt

parallelStream() is one method call that looks free and isn’t. Underneath: the spliterator trySplit()s the source into chunks, chunks are processed as ForkJoin tasks on ForkJoinPool.commonPool(), and results merge via the collector’s combiner.

The common pool is shared by the entire JVM — sized at cores−1.

Every parallel stream, every CompletableFuture default async, everything that doesn’t name a pool shares these threads. One parallel stream doing blocking I/O starves every other parallel computation in the process. That’s the #1 production parallel-stream incident: someone parallelized a stream of HTTP calls, the common pool filled with blocked threads, and unrelated code across the service stopped making progress.

The decision checklist — parallel only if ALL of these hold

  • N × Q is large: element count times per-element cost. 10,000 elements of cheap work loses to sequential; the fork/join orchestration costs more than it saves. Rule of thumb: don’t consider parallel below ~100k cheap elements or unless per-element work is genuinely expensive.
  • The source splits well: ArrayList and arrays split perfectly (SIZED + SUBSIZED — index ranges). LinkedList splits terribly (must walk to find the midpoint). Stream.iterate can’t split at all (each element depends on the previous); LongStream.range splits perfectly.
  • Operations are stateless & non-blocking: CPU-bound pure functions. No I/O, no locks, no shared mutable state.
  • Merge is cheap: summing longs merges in O(1); merging HashMaps merges in O(n). groupingBy in parallel pays a real merge cost — groupingByConcurrent (one ConcurrentHashMap, UNORDERED) often wins if you don’t need encounter order.
  • You don’t depend on encounter order: limit() and findFirst() on an ordered parallel stream force coordination; findAny() and unordered() release it.

The shared-state trap

// BROKEN under parallel — ArrayList is not thread-safe; loses elements or throws
List<String> results = new ArrayList<>();
trades.parallelStream().map(Trade::symbol).forEach(results::add);   // ✗ data race

// CORRECT — let the collector own the mutable state, one container per thread + merge
List<String> results = trades.parallelStream().map(Trade::symbol).toList();  // ✓

Isolating from the common pool

A parallel stream submitted from inside a custom ForkJoinPool runs its tasks in that pool — the standard (if officially undocumented) isolation trick: new ForkJoinPool(8).submit(() -> data.parallelStream()...).get(). For I/O-bound fan-out, don’t use parallel streams at all — that’s what virtual threads / StructuredTaskScope (Day 8) are for. Parallel streams are for CPU-bound data crunching.

🎯 INTERVIEW QUESTIONS

Q: Why was my parallel stream slower than sequential? — Walk the checklist: too few elements / too cheap per element; badly-splitting source (LinkedList, iterate); a stateful op (sorted/limit) serializing the pipeline; expensive merge; or common-pool contention from elsewhere in the JVM.

Q: Is forEach on a parallel stream ordered? — No, deliberately: elements are processed as threads finish. forEachOrdered restores order at the cost of parallel benefit in that stage.

Q: Where do parallel stream threads come from? — ForkJoinPool.commonPool(), sized cores−1, shared JVM-wide; the calling thread also participates. Sizing override: -Djava.util.concurrent.ForkJoinPool.common.parallelism=N.

STAGE 5 30 MIN

Optional discipline & Gatherers — the missing operations

Optional — the API and the anti-patterns

// The chain — alternative sources, transformation, fallback action
String display = repo.findById(id)                     // Optional<User>
    .or(() -> cache.find(id))                          // try another source (lazy)
    .flatMap(User::nickname)                           // nickname() returns Optional — flatMap, not map
    .filter(n -> !n.isBlank())
    .orElseGet(() -> expensiveDefault());              // orElseGet is lazy; orElse always evaluates

ifPresentOrElse(u -> audit(u), () -> log.warn("missing"));

// Optional → Stream bridge: from a stream of Optionals, keep the present values
ids.stream().map(repo::findById).flatMap(Optional::stream).toList();
Anti-patternWhy it’s wrongInstead
if (o.isPresent()) o.get()Re-creates the null-check; get() can throwmap/ifPresent/orElseThrow
Optional as a fieldNot serializable; adds a box to every read; identity-based equals riskNullable field, Optional-returning getter
Optional method parametersCaller forced to wrap; three states (null, empty, present)Overloads or @Nullable param
orElse(compute())Always evaluates, even when presentorElseGet(() -> compute())

The one-line philosophy for interviews: Optional is a return type — a contract that “absence is a normal outcome” — not a general-purpose null replacement.

Stream.iterate + takeWhile — bounded generation (pagination pattern)

// Pull pages from an API until an empty page comes back — lazy, no upfront count needed
Stream.iterate(fetchPage(0), page -> fetchPage(page.number() + 1))
      .takeWhile(page -> !page.items().isEmpty())
      .flatMap(page -> page.items().stream())
      .toList();

Gatherers (Java 22+) — custom intermediate operations

Collectors customize the terminal step; until Gatherers, you couldn’t customize an intermediate step. Stream.gather(...) fills the gap with built-ins that were previously impossible or awkward:

  • Gatherers.windowFixed(3) — [1,2,3,4,5,6] → [1,2,3],[4,5,6] (batching!)
  • Gatherers.windowSliding(2) — [1,2,3] → [1,2],[2,3] (pairwise comparison, moving averages)
  • Gatherers.scan(() -> 0, Integer::sum) — running totals: emits every intermediate accumulation
  • Gatherers.fold(...) — ordered, sequential fold emitting one result

A custom Gatherer<T,A,R> has an initializer, an integrator (sees each element + state + downstream; returns false to short-circuit), an optional combiner, and a finisher. The canonical example interviewers love: a moving average over a price stream — before Gatherers that needed an external mutable deque (broken in parallel); now it’s a self-contained, reusable pipeline stage.

For the trading platform: windowSliding(20) over a tick stream is a 20-tick moving window for a rolling VWAP — expressed declaratively, no hand-rolled ring buffer. And the pagination pattern (iterate + takeWhile) is exactly how you wrap a broker’s paged historical-data API as a lazy stream: pages are fetched only as the consumer pulls, and a limit(n) downstream stops the HTTP calls automatically — laziness doing real work.

🎯 INTERVIEW QUESTIONS

Q: Why is Optional-as-field discouraged? — Optional isn’t Serializable, costs an extra object per read, and was explicitly designed (per Brian Goetz) as a return-type vehicle for “no result” — fields and params re-introduce the null problem one level up (the Optional itself can be null).

Q: How would you batch a stream into chunks of 100 before Java 22, and after? — Before: collect to a list and partition by index arithmetic (eager, loses laziness) or an external library. After: stream.gather(Gatherers.windowFixed(100)) — lazy, composable, parallel-aware.

STAGE 6 35 MIN

Streams & lambdas across the LTS versions — Java 8, 11, 17, 21

Day 8 gave you the platform view of the four LTS releases. This stage is the same timeline through today’s lens: how the functional/streams toolkit grew from 8 to 21. Interviewers use this to date your experience — a candidate still writing collect(Collectors.toList()) on Java 17+ is telegraphing that they stopped learning at 8.

Java 8 (2014) — the foundation

Everything in Stages 1–3 starts here:

  • Lambdas + invokedynamic + LambdaMetafactory — the runtime machinery from Stage 1, designed so future JVMs could optimize lambda creation without recompiling your code.
  • java.util.function — the core four (Supplier, Consumer, Function, Predicate) plus ~40 primitive specializations to dodge autoboxing.
  • Stream API v1 — filter/map/flatMap/reduce/collect, the spliterator engine, parallel streams on the common pool.
  • Collectors v1 — toList, toMap, joining, groupingBy, partitioningBy, downstream composition with counting/mapping.
  • Optional v1 — map, flatMap, filter, orElse, orElseGet — but no or(), no ifPresentOrElse, no stream() bridge yet.
  • Known gaps at the time: no way to conditionally stop a stream (takeWhile didn’t exist), Stream.iterate was infinite-only, Optional-to-Stream required an ugly map(Stream::of).orElseGet(Stream::empty).

Java 11 (2018) — the gap-fillers (bundling 9 & 10)

  • takeWhile / dropWhile (9) — ordered, conditional truncation. The pagination pattern from Stage 5 (iterate + takeWhile) became possible here.
  • 3-arg Stream.iterate (9) — iterate(seed, hasNext, next): a for-loop as a stream, finally bounded.
  • Stream.ofNullable (9) — 0-or-1 element stream from a possibly-null value; perfect inside flatMap.
  • Optional grows up: or() for lazy alternative sources (9), ifPresentOrElse (9), stream() (9) — enabling the flatMap(Optional::stream) idiom — and isEmpty() (11).
  • Collectors: flatMapping and filtering as downstream collectors (9) — the keep-empty-groups pattern from Stage 3; toUnmodifiableList/Set/Map (10).
  • Predicate.not() (11) — turns a positive method reference negative: filter(Predicate.not(String::isBlank)) instead of a lambda.
  • String.lines() (11) — a stream of lines straight off a string; pairs with Files.lines() for lazy file processing.
  • var in lambda params (11) — exists solely so you can annotate lambda parameters: (@Nonnull var s) -> ...

Java 17 (2021) — ergonomics & performance (bundling 12–16)

  • Stream.toList() (16) — the new default. Shorter than collect(toList()), returns an unmodifiable list, and doesn’t need the collector machinery. Subtle interview point: code relying on mutating the result of collect(toList()) breaks if mechanically migrated.
  • mapMulti (16) — Stage 2’s allocation-free flatMap alternative for 0-or-1 expansions.
  • Collectors.teeing (12) — Stage 3’s two-collectors-one-pass composition.
  • Records as stream tuples (16) — the missing piece for multi-value pipelines: map(t -> new SymbolStats(t.symbol(), t.value())). Before records, intermediate pipeline shapes meant AbstractMap.SimpleEntry or a hand-written class; local records (declared inside a method!) made one-off pipeline shapes free.
  • Switch expressions (14) — an expression fits inside map(): map(order -> switch (order.type()) { ... }).

Java 21 (2023) — the concurrency intersection (and 22’s Gatherers)

  • Virtual threads change the parallel-stream calculus — Stage 4’s rule hardens: parallel streams for CPU-bound work on the common pool; virtual threads (cheap, blocking-friendly) for I/O-bound fan-out. Before 21, people abused parallel streams for I/O because threads were expensive; that excuse is gone.
  • Sequenced Collections meet streams — map.reversed().entrySet().stream(): descending iteration over a LinkedHashMap without copying.
  • Record patterns in pipelines — deconstruction inside stream ops via pattern switch: map(result -> switch (result) { case Approved(var id, var amt) -> ...; ... }) — Day 8’s exhaustiveness guarantees, now inside your pipeline.
  • Gatherers (22, preview) — Stage 5: windowFixed, windowSliding, scan, fold, custom Gatherer<T,A,R> — custom intermediate operations, closing the last gap in the API’s extensibility (Collectors only ever customized the terminal step).

The evolution in one sentence per LTS:

8 invented the model (lambdas, streams, collectors, Optional). 11 filled the gaps (takeWhile, bounded iterate, Optional.or/stream, filtering/flatMapping). 17 polished ergonomics (toList, mapMulti, teeing, records as tuples). 21/22 drew the concurrency boundary (virtual threads for I/O, parallel streams for CPU) and opened the pipeline itself (Gatherers).

🎯 INTERVIEW QUESTIONS

Q: Stream.toList() vs collect(Collectors.toList()) — just style? — No. toList() returns an unmodifiable list; Collectors.toList() happens to return a mutable ArrayList (documented as no-guarantees). Migrating mechanically can break code that mutates the result — and toList() can also be faster (no accumulator indirection, sized allocation).

Q: How did Optional change after Java 8? — Java 9 added or() (lazy alternative Optional), ifPresentOrElse() (both branches), and stream() (the flatMap bridge); 10 added orElseThrow() no-arg; 11 added isEmpty(). If a candidate only knows isPresent/get, they learned Optional in 2014 and stopped.

Q: You need to batch a stream into groups of 500 for bulk DB inserts — answer for Java 8, 17, and 22. — Java 8/17: no native support — collect to a list and partition by index, or use a library (Guava Lists.partition); laziness is lost. Java 22: stream.gather(Gatherers.windowFixed(500)) — lazy and composable. Knowing the feature’s absence and arrival dates your knowledge as current.

The streams deep-dive round

Prompt A: “Walk me through exactly what the JVM does when this line executes for the first time: list.stream().filter(s -> s.length() > max).count() — from lambda creation to result.”

Prompt B: “This code intermittently returns wrong counts in production. Find every bug: Map<String,Integer> counts = new HashMap<>(); orders.parallelStream().forEach(o -> counts.merge(o.symbol(), 1, Integer::sum));”

Prompt C: “Design a single stream pipeline producing a report: for each symbol, the trade count, total notional, and the largest single trade — one pass over 10 million trades. Sequential or parallel, and why?”

MODEL ANSWER

Lambda: s -> s.length() > max captures max, so it allocates a Predicate instance per execution of this line (a non-capturing lambda would be a cached singleton). First execution hits the invokedynamic call site: the JVM calls the bootstrap method, LambdaMetafactory.metafactory, which spins a hidden class implementing Predicate whose test() delegates to the compiler-generated private static method holding the body; the call site links and never bootstraps again. Pipeline: list.stream() wraps the list’s spliterator (SIZED, ORDERED) in a head stage; filter(...) allocates one stateless-op stage object linked to the head — no elements move. Terminal: count() triggers evaluation. Wrinkle worth naming: with no peek, count() on a SIZED source could shortcut, but the filter changes the count, so it must traverse: the spliterator’s forEachRemaining pushes each element through the fused sink chain — test the predicate, increment if it passes — one pass, no intermediate collection, result returned.

MODEL ANSWER

Bug 1 — data race: HashMap is not thread-safe; concurrent merge() from ForkJoin threads loses updates (lost counts), can corrupt buckets, and in resize races can even loop or throw. Intermittent-wrong-counts is the textbook symptom. Bug 2 — pattern violation: side-effecting forEach into external mutable state is exactly what the Stream API’s “mutable reduction” exists to avoid — the stream should own the container. Fix (idiomatic): Map<String,Long> counts = orders.parallelStream().collect(groupingBy(Order::symbol, counting())); — per-thread maps, merged by the combiner. Fix (high-contention alternative): groupingByConcurrent with a single ConcurrentHashMap, valid because counting is UNORDERED. Bonus point: swapping HashMap for ConcurrentHashMap alone would make the original correct (merge is atomic there) but still non-idiomatic and contended — saying that distinction out loud is senior-level signal. Then question whether parallel is justified at all (Stage 4 checklist).

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • A lambda is invokedynamic + LambdaMetafactory, not an anonymous class: runtime-linked, singleton when non-capturing, allocating only when it captures.
  • Streams are lazy fused pipelines over spliterators: intermediate ops build stages, the terminal op pulls each element through all stages in one pass — which is what makes short-circuiting and infinite sources work.
  • Collectors are 4 functions; the combiner is load-bearing. A collector that can’t merge partial results is broken under parallelism. teeing and downstream collectors compose multi-aggregate one-pass reports.
  • Parallel streams: CPU-bound, well-splitting, stateless, cheap merge, big N — and they all share the common pool. Blocking I/O in a parallel stream is a JVM-wide incident; that workload belongs to virtual threads.
  • Optional is a return-type contract, not a null replacement: never a field or parameter, orElseGet over orElse, flatMap(Optional::stream) to cross into streams.
