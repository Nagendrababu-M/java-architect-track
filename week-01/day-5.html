Java Memory Model (JMM) & happens-before Why concurrent code is hard, what actually guarantees one thread sees another’s writes, and how to publish objects safely.

schedule 160 min · layers 6 stages
RECAP FROM DAY 4

You now own the JVM memory picture: the heap is shared by every thread, and the GC manages it underneath you. Day 5 confronts the consequence of that sharing: when two threads touch the same heap data, what does each one actually see? The answer is not “the latest value” — it’s whatever the Java Memory Model permits, which is far weaker and stranger than intuition suggests. The JMM is the contract that says which writes are guaranteed visible to which reads. Master it and concurrency stops being guesswork; ignore it and you ship bugs that only appear in production, on a different CPU, under load.

STAGE 1 25 MIN

Why concurrency is hard — reordering, caching, and the three guarantees

Your source code is a sequence of statements. The CPU that runs it is not obligated to execute them in that order — only to preserve the appearance of order for a single thread. Three layers reorder your program, each for performance:

  • The compiler / JIT reorders independent statements and hoists reads out of loops.
  • The CPU executes instructions out of order and speculatively.
  • The memory subsystem — each core has its own caches and store buffers, so a write by one core may sit in that core’s buffer, invisible to others for an unbounded time.

Single-threaded, none of this is observable — the JVM guarantees your thread sees its own actions in program order (the “as-if-serial” rule). The instant a second thread reads the same data, all bets are off.

The canonical reordering bug.

// Thread A                       // Thread B
value = 42;                        while (!ready) { }     // spin until ready
ready = true;                       System.out.println(value);

Intuition says B prints 42. The JMM allows three other outcomes: B may never see ready become true (the value is stuck in A’s cache and B’s read is hoisted out of the loop — an infinite spin); or B may see ready == true but value == 0, because A’s two writes were reordered, or B’s two reads were. Nothing here establishes that A’s write to value is visible before B’s read of it. The code is racy.

Three distinct guarantees — don’t conflate them

Most concurrency confusion comes from treating these as one thing. They are independent:

PropertyQuestion it answersBroken example
AtomicityDoes this operation happen all-or-nothing?count++ is read-modify-write — two threads can both read 5 and both write 6
VisibilityWill another thread ever see my write?A non-volatile flag written by one thread may never be observed by another
OrderingDo my writes appear in the order I wrote them?value then ready may be seen in the reverse order by another thread

A correct concurrent program needs the right combination of all three. volatile gives you visibility + ordering but not atomicity; synchronized and locks give all three for the guarded region; atomics give atomicity + visibility for a single variable. Knowing which tool gives which property is the whole game.

In the reordering example, name the two distinct failures that can occur, and which guarantee (visibility or ordering) each one violates.

STAGE 2 30 MIN

happens-before — the contract that tames the chaos

The JMM doesn’t promise “writes are instantly visible.” It gives you something precise instead: a partial ordering called happens-before. The single rule that matters:

If action X happens-before action Y, then the effects of X (all its writes) are guaranteed visible to Y, and X is guaranteed to appear ordered before Y. If there is no happens-before edge between two conflicting accesses (at least one a write to the same variable), you have a data race — and the result is undefined.

You never reason about caches or barriers directly. You establish happens-before edges, and the JVM inserts whatever fences the hardware needs. These are the edges the JMM gives you:

EdgeThe guarantee
Program orderWithin a single thread, each action happens-before every later action in source order.
Monitor lockUnlocking a monitor happens-before every subsequent lock of the same monitor. (This is what synchronized gives you.)
VolatileA write to a volatile field happens-before every subsequent read of that same field.
Thread startthread.start() happens-before every action inside the started thread.
Thread joinEvery action in a thread happens-before another thread returns from thread.join() on it.
TransitivityIf X happens-before Y and Y happens-before Z, then X happens-before Z.

Transitivity is the workhorse. It’s how a lock or a volatile makes ordinary (non-volatile) writes visible. Revisit Stage 1 with ready made volatile:

// Thread A                          // Thread B
value = 42;            // (1)         while (!ready) { }     // (3) volatile read
ready = true;          // (2) volatile write  print(value);          // (4)

The chain: (1) happens-before (2) by program order; (2) happens-before (3) by the volatile rule (B’s read sees the true that A wrote); (3) happens-before (4) by program order. By transitivity, (1) happens-before (4) — so B is guaranteed to see value == 42, even though value itself is an ordinary field. The volatile write “publishes” everything that came before it.

🎯 INTERVIEW QUESTIONS

Q: What is a data race, precisely? — Two accesses to the same variable, at least one a write, with no happens-before edge ordering them. Not the same as a race condition (a logic-level timing bug); a data race is the JMM-level absence of ordering, and its result is undefined.

Q: Does synchronized only provide mutual exclusion? — No — that’s the common half-answer. It also creates a happens-before edge (unlock → subsequent lock), so it provides visibility and ordering for everything done inside the block, not just atomicity.

Q: How does making one field volatile fix visibility of other, non-volatile fields? — Transitivity: writes before the volatile write are ordered before it; a reader that sees the volatile value is guaranteed to see those earlier writes too.

QUICK CHECK

Two threads share a plain int counter guarded by a synchronized block on both increment and read. Which happens-before edge makes the reader see the latest value — and would it still hold if only the writer synchronized?

STAGE 3 25 MIN

volatile — visibility and ordering, never atomicity

volatile is the lightest synchronization tool. It does exactly two things, and the interview trap is believing it does a third.

What volatile guarantees

  • Visibility: every read of a volatile field sees the most recent write — the value is never served from a stale cache.
  • Ordering: it establishes the happens-before edge from Stage 2, and acts as a barrier — reads/writes before a volatile write can’t be reordered after it, and reads after a volatile read can’t be moved before it.

What volatile does NOT guarantee: atomicity

volatile int count = 0;
count++;   // THREE operations: read count, add 1, write count

Two threads can both read 5, both compute 6, both write 6 — one increment lost. volatile made each read and write visible, but the read-modify-write as a whole is not atomic. For that you need AtomicInteger.incrementAndGet() (a CAS loop) or a lock. Rule of thumb: volatile is correct only when a write does not depend on the current value — a flag, a one-shot published reference, a configuration snapshot. The moment the new value is a function of the old one, volatile alone is wrong.

The one pattern volatile nails

private volatile boolean shutdown = false;
public void shutdown() { shutdown = true; }          // writer
public void run() { while (!shutdown) { doWork(); } }  // reader sees it promptly

Without volatile the worker loop may never observe the flag and spin forever — the classic “my thread won’t stop” bug.

The 64-bit gotcha: non-atomic long & double

The JMM permits a 64-bit long or double write to be split into two 32-bit writes. A concurrent reader (without synchronization) can observe a torn value — the high 32 bits of one write and the low 32 bits of another, a number that was never assigned. Marking the field volatile makes 64-bit reads and writes atomic, closing the gap. (References and all other primitives are always atomic to read/write; only long/double have this hole.)

🎯 INTERVIEW QUESTIONS

Q: Is volatile enough to make count++ thread-safe? — No. It fixes visibility/ordering but not the read-modify-write atomicity. Use AtomicInteger or a lock.

Q: When is volatile the right tool? — When the new value doesn’t depend on the old: status flags, safe publication of an immutable object, double-checked-locking guards.

Q: Why can a non-volatile long read return a value that was never written? — 64-bit writes may be split into two 32-bit stores; a reader can see a half-updated value (word tearing). volatile makes them atomic.

QUICK CHECK

You have a volatile long requestId incremented by many threads with requestId++. Two bugs hide here — the volatile fixes one of them. Which one does it fix, and which remains?

STAGE 4 25 MIN

final fields & safe publication

Publication is making an object reachable from another thread. The danger: a thread can obtain the reference before the object’s constructor has finished writing its fields — and see a partially constructed object. This is the deepest JMM trap, because the object “looks” fine.

final field semantics

The JMM makes a special promise for final fields: if an object is constructed correctly, any thread that later sees the object is guaranteed to see its final fields fully initialized — without any synchronization. There’s a “freeze” action at the end of the constructor that orders the final-field writes before the object reference can be published.

public final class Config {
    private final String url;
    private final int timeout;
    public Config(String url, int timeout) { this.url = url; this.timeout = timeout; }
    // Any thread that gets a Config reference sees url and timeout set — no sync needed.
}

The escape that breaks the guarantee. The final-field promise holds only if this does not escape during construction. If a constructor publishes the partially-built object — registering a listener, starting a thread, assigning this to a static field before the constructor returns — another thread can see the object before the freeze, and the guarantee is void. Never let this leak from a constructor.

The four safe-publication idioms

To hand an object to another thread safely, the publication itself must create a happens-before edge. There are exactly four standard ways:

IdiomMechanism
Static initializerClass-init locking guarantees visibility: static final X x = new X();
volatile / AtomicReference fieldThe volatile-write → volatile-read edge publishes the object and its prior writes
final fieldThe constructor freeze publishes the referenced object (if this didn’t escape)
Lock-guarded fieldWriting inside a lock and reading inside the same lock (the monitor edge)

An effectively immutable object (no mutation after construction) published by any of these is safe to share across threads forever. A truly immutable object (all fields final) is safe even if published unsafely — the final-field guarantee carries it. This is the deep reason immutability is the architect’s default for shared state: it removes the publication problem entirely.

🎯 INTERVIEW QUESTIONS

Q: What is unsafe publication? — Making an object visible to other threads without a happens-before edge, so a reader can see a reference to an object whose fields aren’t yet visible (or a constructor that hasn’t finished).

Q: Why are final fields special? — The constructor freeze guarantees their initialized values are visible to any thread that sees the object, with no synchronization — provided this didn’t escape during construction.

Q: Why is an immutable object thread-safe by default? — No mutation means no write-after-publication races, and all-final fields make even unsafe publication safe. Sharing requires zero synchronization.

QUICK CHECK

A class registers itself as an event listener in its constructor (bus.register(this)) and sets a final field afterward. Why might another thread’s callback see that final field uninitialized?

STAGE 5 25 MIN

Double-checked locking — the cautionary tale, and what to do instead

Double-checked locking (DCL) is a lazy-singleton optimization: check the field without a lock (fast path), and only lock if it’s null. It is famous because the “obvious” version is broken, and understanding why is the perfect JMM capstone.

The broken version (pre-Java 5 idiom, still wrong without volatile)

private static Singleton instance;   // NOT volatile — the bug
public static Singleton get() {
    if (instance == null) {                     // (1) unlocked check
        synchronized (Singleton.class) {
            if (instance == null)
                instance = new Singleton();    // (2) construct + publish
        }
    }
    return instance;                            // (3)
}

Line (2) is not atomic. instance = new Singleton() is really: allocate memory, run the constructor, then assign the reference. The JMM permits the JIT/CPU to reorder the assignment before the constructor finishes. Now thread B hits line (1), sees a non-null instance, skips the lock, and returns a reference to an object whose constructor hasn’t run — fields still at defaults. There’s no happens-before edge between the constructor’s writes and B’s unlocked read. A latent, load-dependent, “works-on-my-machine” crash.

The fix: one keyword

private static volatile Singleton instance;   // volatile closes the hole

The volatile write of instance happens-before any subsequent read of it (Stage 2), and the volatile barrier forbids the reorder — the constructor’s writes are ordered before the reference is published. DCL has worked correctly since Java 5 only with volatile. If an interviewer shows you DCL, the first thing you check is that keyword.

Better: skip DCL entirely

DCL is rarely worth its subtlety. Two cleaner idioms get lazy, thread-safe singletons for free:

// 1. Initialization-on-demand holder — lazy AND lock-free
private static class Holder { static final Singleton INSTANCE = new Singleton(); }
public static Singleton get() { return Holder.INSTANCE; }
// The JVM loads Holder only on first get(); class-init locking guarantees safe publication.

// 2. Enum singleton — Joshua Bloch's recommendation; serialization-safe too
public enum Singleton { INSTANCE; public void work() { } }

The holder idiom leans on a JMM guarantee you got for free in Stage 4: class initialization is locked by the JVM, so static final fields are safely published with zero explicit synchronization and zero per-call cost.

🎯 INTERVIEW QUESTIONS

Q: Why was double-checked locking broken before Java 5? — The reference could be published before the constructor finished (reordering), with no happens-before edge to the unlocked reader — so a thread could see a non-null but partially-constructed object. The pre-Java-5 memory model couldn’t even express the fix.

Q: What exactly does volatile fix in DCL? — It forbids the publish-before-construct reorder and creates the write→read happens-before edge, so a reader that sees non-null also sees the fully-constructed object.

Q: Best way to write a lazy singleton today? — The initialization-on-demand holder idiom (lazy, lock-free, simple) or an enum (also serialization- and reflection-safe). DCL only if you must lazily init an instance field where a holder class doesn’t fit.

The challenge — find the races in a Singleton

The capstone. Read the code, then answer all three parts (50+ characters each) and reveal the model answers. These are open-ended JMM diagnoses — reason in terms of happens-before, not vibes.

SCENARIO · READ CAREFULLY BEFORE ANSWERING

A cache singleton that “works in testing”

public class PriceCache {
    private static PriceCache instance;            // A
    private Map<String,Double> prices = new HashMap<>();
    private int hitCount = 0;

    private PriceCache() { load(); }

    public static PriceCache getInstance() {        // B
        if (instance == null) instance = new PriceCache();
        return instance;
    }

    public Double get(String sym) {                 // C
        hitCount++;
        return prices.get(sym);
    }
}

Called concurrently from many request-handling threads. It passes every single-threaded test.

YOUR ANSWERS · MINIMUM 20 CHARS EACH TO UNLOCK MODEL ANSWER
0 / 20 characters
MODEL ANSWER

Race 1 — the singleton check (B). if (instance == null) instance = new PriceCache() is unsynchronized and non-atomic: two threads can both see null and both construct an instance (broken atomicity), and a reader can see a non-null reference to a partially-constructed object because there’s no happens-before edge publishing the constructor’s writes (visibility + ordering — the DCL trap). Race 2 — the shared HashMap (C). HashMap is not thread-safe; concurrent get/put during load() can corrupt internal state or (historically) spin forever on resize — a visibility/atomicity failure on the map’s structure. Race 3 — hitCount++ (C). Read-modify-write with no synchronization: lost updates (broken atomicity), and even the reads have no visibility guarantee.

MODEL ANSWER

Singleton: use the initialization-on-demand holder idiom — private static class Holder { static final PriceCache INSTANCE = new PriceCache(); } and return Holder.INSTANCE. Class-init locking guarantees safe publication of a static final field with no per-call cost (Stage 4/5). Map: replace HashMap with ConcurrentHashMap — thread-safe reads/writes with internal happens-before edges, no external locking. Make prices final so it’s safely published with the instance. Counter: replace int hitCount + hitCount++ with an AtomicLong (incrementAndGet, a CAS) or, for high write contention, a LongAdder — both give atomic increment plus visibility. Net: the holder gives safe publication, ConcurrentHashMap and the atomic give per-operation atomicity + visibility.

🪞 Reflection & Self-Assessment ▾
Key Takeaways
  • Concurrency has three independent guarantees — atomicity, visibility, ordering. Always know which one a tool gives you.
  • happens-before is the whole model. No edge between conflicting accesses = data race = undefined. The edges: program order, monitor, volatile, thread start/join, transitivity.
  • volatile = visibility + ordering, never atomicity. Correct only when the new value doesn’t depend on the old; also makes 64-bit long/double atomic.
  • Safe publication needs a happens-before edge: static initializer, volatile/AtomicReference, final field, or lock. Immutable objects sidestep the problem entirely.
  • DCL needs volatile; better still, use the holder idiom or an enum.
