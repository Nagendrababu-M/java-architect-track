Stage 6 — three escalating interview prompts. Explain the internals, debug the production bug, design the pipeline. Try each one before comparing with the model answers. Here's how strong answers sound.

Prompt A: trace list dot stream, filter with a lambda comparing against a captured variable, count. Start with the lambda: it captures max, so it allocates a Predicate each time the line runs — call that out, then contrast: a non-capturing lambda would be a cached singleton. First execution hits invokedynamic, LambdaMetafactory spins the hidden class, the call site links once. Then the pipeline: stream wraps the list's spliterator — sized, ordered. Filter builds a stage object; nothing moves. Count is the trigger: the spliterator pushes each element through the fused chain — test, increment, next element. One pass, no intermediate collections. Bonus point: count on a sized source could shortcut without traversing, but the filter changes the count, so it must traverse. Naming that wrinkle shows real depth.

Prompt B: the parallel counting bug. The symptom — intermittently wrong counts — is the fingerprint of a data race. HashMap is not thread-safe; concurrent merge calls from ForkJoin threads lose updates and can corrupt buckets. But name the second defect too: side-effecting forEach into external state violates the model — the stream should own its mutable container. The idiomatic fix: collect with groupingBy and counting. Per-thread maps, merged by the combiner, no shared state at all. The senior-level garnish: swapping in ConcurrentHashMap would technically fix correctness — merge is atomic there — but it's still non-idiomatic and contended. Saying "correct but not right" out loud is signal. Then question whether two hundred orders even justifies parallel.

Prompt C: the one-pass report — count, total notional, and largest trade per symbol, over ten million trades. Shape: groupingBy symbol with a composed downstream. Teeing composes two collectors, so three aggregates means nesting teeing — or, cleaner, a small custom Collector with a mutable stats accumulator, and saying you'd prefer the custom collector for readability is a good instinct. Sequential or parallel? Walk the checklist out loud: ten million elements, check. Source must be an ArrayList or array for clean splitting — if the data arrives as a linked structure or a file stream, fix the source first. Stateless, CPU-bound, check. The cost center is the map merge across threads, so the honest answer is: parallel is defensible, benchmark both with JMH before committing. "I'd measure" is the senior answer — confidence about the checklist, humility about the numbers.
