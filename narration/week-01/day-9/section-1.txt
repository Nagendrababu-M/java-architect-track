Day 9. Everyone can write filter-map-collect. Today we go underneath, because senior interviews test the why, not the what. Stage 1: what a lambda actually is.

First, the vocabulary. A functional interface is any interface with exactly one abstract method — a SAM type, single abstract method. Default and static methods don't count. Neither do abstract methods matching Object's public methods — that's why Comparator can declare equals and still be functional. The FunctionalInterface annotation changes nothing at runtime; it makes the compiler fail if someone adds a second abstract method later. A contract annotation, like Override.

The core four: Supplier produces a value from nothing. Consumer takes a value, returns nothing. Function transforms. Predicate tests. And their primitive cousins — IntFunction, DoublePredicate — exist for one reason: avoiding autoboxing. A Stream of boxed Integers boxes and unboxes on every element. IntStream works on raw ints. Over millions of elements, that's the difference between memory-bound and CPU-bound code.

Method references come in four kinds, and the distinction interviewers test is bound versus unbound. Integer colon-colon parseInt — static. myString colon-colon length — bound to a particular object, takes no arguments. String colon-colon length — unbound: the receiver becomes the first parameter, so it works as a Function from String to Integer. And ArrayList colon-colon new — a constructor reference.

Capture semantics: a lambda can only capture locals that are effectively final. Why? Because captured locals are copied into the lambda at creation. If the variable could change later, the lambda's copy and the original would silently disagree. Java forbids the divergence instead of allowing the confusion. Fields are different — they're read live through this, not copied — which is why they don't need to be final, and why mutating a field from inside a parallel stream is a data race.

Now the deep cut: a lambda is NOT an anonymous class. The compiler does not generate a class file per lambda. It puts the lambda body into a private static method and emits an invokedynamic instruction. On first execution, the JVM calls a bootstrap method — LambdaMetafactory — which spins a small implementation class at runtime and links the call site. After that, every execution hits the linked call site directly.

The consequences are what you say in interviews. Non-capturing lambdas are cached singletons — zero allocation, ever. Capturing lambdas allocate per evaluation, because they must carry their captured values. And because the strategy lives in the JVM, not in your class files, the platform can make lambdas faster without you recompiling. An anonymous class is a fixed class file and an allocation on every new. Different machinery entirely.
