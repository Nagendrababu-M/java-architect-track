Stage 4: parallel streams, and specifically when they hurt — because the interview question is never "how do you parallelize a stream," it's "why was your parallel stream slower."

First, what actually happens. ParallelStream makes the spliterator trySplit the source into chunks, each chunk becomes a ForkJoin task, and results merge through the collector's combiner. And all those tasks run on ForkJoinPool dot commonPool — one pool, shared by the entire JVM, sized at cores minus one.

Let that sink in, because it's the number one production incident. Every parallel stream in the process, every CompletableFuture that didn't name an executor — all sharing the same handful of threads. Someone parallelizes a stream of HTTP calls. The common pool fills with threads blocked on I/O. And suddenly, unrelated code on the other side of the service stops making progress. The speedup looked great in testing — testing didn't have anything else competing for the pool.

So here's the checklist. Parallel only when ALL of these hold.

One: N times Q is large — element count times per-element cost. Ten thousand cheap elements lose to sequential; the orchestration costs more than it saves.

Two: the source splits well. ArrayList and arrays split perfectly — they're index-addressable, so trySplit is arithmetic. LinkedList splits terribly — finding the midpoint means walking half the list. Stream dot iterate can't split at all, because each element depends on the previous one.

Three: operations are stateless and non-blocking. CPU-bound pure functions. No I/O, no locks, no shared mutable state. The classic trap: forEach adding to an ArrayList from a parallel stream. ArrayList isn't thread-safe — you lose elements or throw. The fix isn't a synchronized list; it's letting the collector own the state — toList gives each thread its own container and merges.

Four: the merge is cheap. Summing longs merges in constant time. Merging HashMaps is linear — groupingBy in parallel pays real merge costs. GroupingByConcurrent trades encounter order for a single shared ConcurrentHashMap.

Five: you don't depend on encounter order. Limit and findFirst on an ordered parallel stream force cross-thread coordination — "the FIRST n elements" is expensive when work finishes out of order. FindAny and unordered release that constraint.

Last piece: for I/O-bound fan-out, parallel streams are the wrong tool entirely. That's what virtual threads and StructuredTaskScope are for — remember Day 8. Parallel streams are for CPU-bound data crunching. Say that boundary clearly in an interview and you've answered the question they were actually asking.
