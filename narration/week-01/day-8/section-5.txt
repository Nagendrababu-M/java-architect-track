Stage 5: Optional discipline, the pagination pattern, and Gatherers — the newest piece of the stream API.

Optional first. The philosophy in one sentence, straight from the API's designers: Optional is a return type — a contract that absence is a normal outcome — not a general-purpose null replacement.

The good chain: findById returns an Optional. Dot or tries another source, lazily. FlatMap when the next step itself returns an Optional — nickname returns Optional, so map would give you Optional of Optional; flatMap flattens. Filter, then orElseGet for the fallback. Why orElseGet and not orElse? OrElse evaluates its argument ALWAYS — even when the value is present. If the default is expensive, you just paid for it every time. OrElseGet takes a supplier and only runs it on empty.

The anti-patterns, because interviewers ask for them by name. IsPresent followed by get — you've recreated the null check with extra steps. Optional as a field — it's not serializable, costs an object per read, and the Optional itself can be null, so you've moved the problem up a level. Optional as a method parameter — now the caller juggles three states: null, empty, present. Return type. That's it.

And the bridge: from a stream of Optionals, flatMap with Optional colon-colon stream keeps the present values and drops the empties. One idiom, very common in interviews.

Next, the pagination pattern — Stream dot iterate plus takeWhile. Fetch page zero, iterate to the next page, takeWhile pages are non-empty, flatMap the items. The beauty is laziness doing real work: pages are fetched only as the consumer pulls, so a limit downstream literally stops the HTTP calls. This wraps any paged API or DB cursor as a lazy stream.

Finally, Gatherers, Java 22. Collectors customize the terminal step. Until Gatherers, you could NOT customize an intermediate step — and some operations were simply impossible. Stream dot gather fills the gap. WindowFixed of three turns one-two-three-four-five-six into two batches of three — batching, the most-requested stream feature in history. WindowSliding gives overlapping windows — pairwise comparison, moving averages. Scan emits running totals. And a custom Gatherer has an initializer, an integrator that sees each element plus state plus the downstream — returning false to short-circuit — and a finisher.

The example that lands in interviews: a moving average over a price stream. Before Gatherers, that required an external mutable deque — broken under parallelism, ugly everywhere. Now it's a self-contained, reusable pipeline stage. For our trading context: windowSliding of twenty over a tick stream is a rolling VWAP window, declared in one line.
