Welcome to Day 8. Today and tomorrow are interview-prep sessions, and we start with the feature trio that defines modern Java: records, sealed classes, and pattern matching. This stage is records.

A record is a class whose state is its API. One line — record Trade with symbol, quantity, and price — and the compiler generates the constructor, the accessors, equals, hashCode, and toString. But don't sell it in an interview as "less boilerplate." Sell the semantics: a record is a promise that this type is a transparent, immutable carrier of its components. All fields are private and final. There are no setters. Equality is component-wise — two records with equal components are equal. A record can't extend a class and can't be extended, though it can implement interfaces.

The compact constructor is where validation lives. You write the constructor with no parameter list at all, and inside it you validate and normalize. Here's the subtle part: when you write symbol equals symbol dot toUpperCase, you're reassigning the parameter, not the field. The fields get assigned after your compact constructor finishes. That's how normalization works without ever touching a field directly.

Now, three traps interviewers love. First: records are only shallowly immutable. If a component is a List, the reference is final, but the list itself can still be mutated by anyone holding it. The fix is a defensive copy in the compact constructor — items equals List dot copyOf of items. One line, deep immutability.

Second: records are not JavaBeans. No no-arg constructor, no getters with the get prefix — the accessor for symbol is just symbol(). Older frameworks that reflect on bean conventions can choke.

Third, and this one is a genuinely senior insight: records are safer for untrusted data. Regular Java deserialization bypasses constructors entirely — your validation never runs. Record deserialization goes through the canonical constructor, always. Your compact constructor validation cannot be skipped. If an interviewer asks why you'd prefer a record for an API payload, that's the answer that lands.

And know when not to use them: JPA entities need mutability and identity-based equality — records have neither. Records model values; entities model identity over time. Say that sentence in an interview and you've separated yourself from the crowd.
