Stage 4: sampling versus instrumentation, safepoint bias, and profiling modes. This is the stage that explains why one profiler can lie to you and another tells the truth.

There are two fundamental ways to profile. Sampling periodically captures every thread's stack — say every ten milliseconds. Overhead is low and tunable, accuracy is statistical, so hot paths naturally dominate the samples. Instrumentation instead injects timing code into every method's entry and exit. You get exact call counts, but the overhead is high and — this is the trap — it distorts what it measures. Adding code to every method makes cheap, frequently-called methods look expensive, because the measurement cost dominates their real cost, and it defeats inlining. So the profile lies. That's why sampling is the production default.

Now the deep one: safepoint bias, the reason most Java samplers lie even when sampling. Traditional Java profilers, and JFR's method sampler to a degree, can only capture a stack at a safepoint — a point where the JVM has paused the thread in a known state. But the JIT only inserts safepoints at certain spots, like loop back-edges and method calls. A hot, tight loop with no safepoint inside never gets sampled there. So the real hot code is invisible, and the samples pile up at the next safepoint, blaming the wrong method. That's safepoint bias: the profile is systematically skewed toward safepoint locations.

async-profiler sidesteps this. It uses OS-level signals — perf_events, or SIGPROF — together with the JVM's AsyncGetCallTrace API to capture a thread's real stack at any instruction, not just at safepoints. It catches the tight loops and JNI-heavy code that JFR can miss, at very low overhead.

And async-profiler has modes that map straight onto Stage 1's bottleneck classes. CPU mode, with asprof dash d thirty, finds CPU-bound hot code. Alloc mode, with dash e alloc, finds the call sites churning the heap — straight back to GC pressure from Day 4. Lock mode, dash e lock, finds where threads block. And wall-clock mode, dash e wall, includes time spent BLOCKED and WAITING, so it finds the I/O and lock waits that a pure CPU profile renders completely invisible.

That last point is the connective tissue of the whole day: pick the profiler mode that matches your hypothesis. If you suspect I/O, a CPU profile will look misleadingly idle — you want wall-clock. If you suspect GC churn, you want allocation mode. Suspect, then measure with the matching tool.
