Stage 2: the atomic family — counters, references, and functional updates. The java.util.concurrent.atomic package wraps CAS in friendly classes: AtomicInteger, AtomicLong, AtomicBoolean, and the general AtomicReference of T. The convenience methods hide the retry loop. incrementAndGet is an atomic plus-plus with a CAS loop inside. getAndAdd returns the old value and adds atomically. And compareAndSet of false-to-true is a one-shot guard, perfect for do-once patterns.

Java 8 added functional updates — CAS loops you don't have to write yourself. updateAndGet and accumulateAndGet run a CAS loop with your function. accumulateAndGet with Math max gives you an atomic running maximum. updateAndGet with a lambda gives you an atomic transform of a reference. But here's a crucial caution: the function you pass may be retried — the loop re-runs it on CAS failure — so it must be pure and side-effect-free. No logging, no mutation, no I/O inside it. That's a frequent subtle bug.

Now the architect's pattern, which ties Day 5 and Day 8 together: AtomicReference plus immutable snapshots. This is the lock-free way to manage compound state safely. Picture a Config record — immutable, from Day 8 — held in an AtomicReference. To update, you call updateAndGet and build a whole new Config snapshot, and CAS the reference. Readers just call get and see a consistent, fully-built snapshot — safe publication from Day 5 — never a torn half-update, with zero locking. It's copy-on-write at the reference level, and it's how you publish configuration, routing tables, and feature flags in a hot path.

For the last drop of performance, there are also AtomicIntegerFieldUpdater and AtomicReferenceFieldUpdater, which apply CAS to a volatile field directly, avoiding a wrapper object per field. Niche, but worth knowing the name exists.

Two interview answers. Implement a thread-safe counter without synchronized? AtomicInteger with incrementAndGet, a CAS loop — and for very high write contention, prefer LongAdder, which is Stage 3. And why must the lambda passed to updateAndGet be side-effect-free? Because it can be re-executed on CAS failure, so any side effect would run multiple times for one logical update.
