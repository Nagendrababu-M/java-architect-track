Stage 4: the ABA problem. CAS asks "is the value still what I expected?" But "the same value" is not the same as "nothing happened." If a value goes A to B to A between your read and your CAS, the CAS succeeds — it sees A and assumes nothing changed — even though the world moved underneath you. That's the ABA problem.

Here's why ABA is usually harmless for numbers but dangerous for references. For a counter, A to B to A genuinely leaves the value at A — no harm done. The danger is with pointers in lock-free structures. A node can be removed, and a different node recycled to the same memory address, so the reference compares equal — it's "A" — but it now points at a different logical object. Your CAS succeeds and you splice corrupted links: a lost node, a cycle, or a resurrected element.

The fix is version stamps. AtomicStampedReference pairs the reference with an integer version. You read the value and the stamp together. When you CAS, you must match both the reference and the stamp, and you bump the stamp on success. Now A to B to A changes the stamp at each transition, so the comeback-to-A no longer looks unchanged — the CAS fails and you retry correctly. There's also a lighter cousin, AtomicMarkableReference, which carries a single boolean mark bit instead of a counter — enough when you only need to flag something as logically deleted.

So the two interview answers. What is the ABA problem? A value changes A to B to A between a thread's read and its CAS; the CAS succeeds because the value matches, missing that intervening changes occurred — dangerous for pointer-based lock-free structures where a recycled address looks identical. And how do you fix it? AtomicStampedReference — reference plus version stamp — so any change bumps the stamp and a returned-to-A value fails the CAS; or AtomicMarkableReference for a single mark bit. And note: languages with garbage collection reduce, but don't fully eliminate, ABA, because a popped node can't be reused while someone still references it — which is exactly the precondition the next stage's challenge exploits with an object pool.
