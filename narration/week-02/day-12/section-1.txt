Day 12: atomics, compare-and-swap, and lock-free patterns. Day 11 made shared state safe with locks — pessimistic concurrency: assume a conflict, exclude everyone else. That's correct, but it serializes, and contention silently caps throughput. Today is the other philosophy: optimistic concurrency — assume no conflict, do the work, and only retry if someone beat you. Its primitive is compare-and-swap, a single hardware instruction, and it's the foundation of every atomic, every lock-free data structure, and the entire java.util.concurrent package.

So what is CAS? One atomic CPU instruction with three operands: a memory location, an expected value, and a new value. Its meaning: if the location still holds the expected value, set it to the new value and report success; otherwise change nothing and report failure. The compare and the swap happen indivisibly — no other thread can sneak in between.

The shape of every atomic update is the CAS retry loop. Read the current value. Compute the new value. Then compareAndSet — swap if and only if the value is unchanged, otherwise loop and retry with the fresh value. If another thread changed the value between your read and your CAS, compareAndSet returns false and you retry. No thread ever blocks; a thread only ever redoes its own work. Under the hood this is a single instruction — CMPXCHG on x86, load-linked store-conditional on ARM — cheaper than a lock, which needs OS arbitration when contended.

Here's the key memory point: CAS gives all three Day 5 guarantees for a single variable. The atomic classes are built on volatile fields plus CAS, so you get atomicity — the read-modify-write is indivisible; visibility — the volatile read and write flush caches; and ordering — a successful CAS establishes a happens-before edge, just like a lock release. That's why an AtomicInteger needs no synchronized: the hardware does the mutual exclusion for that one word.

And the mental model — optimistic versus pessimistic. A lock is pessimistic: acquire first, assume a conflict, block others. CAS is optimistic: do the work, then check; pay only when there's an actual conflict. Optimistic wins when conflicts are rare, which is the common case — no blocking, no context switches, no scheduler. But internalize the catch: under contention, CAS doesn't block, it spins and retries, burning CPU. We'll see in Stage 3 that under very high contention, a lock can actually win.
