Stage 5: lock-free structures, progress guarantees, and when not to roll your own.

Start with the Treiber stack — lock-free in about six lines. There's an AtomicReference to the top node. push builds a new node, sets its next to the current top, and CAS-es the top from the old head to the new node, retrying on failure. pop reads the current top, returns null if empty, and CAS-es the top from the current node to its next, retrying on failure. That pop CAS is exactly where ABA can strike. It's the canonical lock-free stack: each operation is a CAS retry loop on the head pointer. Elegant — and a minefield.

Now the vocabulary of progress guarantees, which architects are expected to know. Lock-free: the system always makes progress — some thread completes in a bounded number of steps. Individual threads may starve, by repeatedly losing the CAS, but the structure never stalls. CAS loops are lock-free. Wait-free is stronger: every thread completes in a bounded number of steps — no starvation possible. It's hard to build and rare. And obstruction-free is the weakest: a thread completes if it runs in isolation long enough — StampedLock's optimistic read is in this family.

Here's the architect's rule, and it's important: almost never hand-write a lock-free structure. They are extraordinarily hard to get right — ABA, safe memory reclamation, subtle reorderings — and even harder to debug. Reach instead for the battle-tested lock-free collections in java.util.concurrent. ConcurrentLinkedQueue and ConcurrentLinkedDeque are lock-free and are the right answer to "I need a concurrent stack or queue." ConcurrentHashMap, LongAdder, all built on CAS, all correct. For the extreme high-throughput case, study the LMAX Disruptor — a ring buffer using CAS and cache-line padding that powers high-frequency trading systems — but use the library, don't reinvent it. Roll your own only with a very good reason and a lot of testing, with a tool like jcstress.

One modernization note: hand-written lock-free code historically used sun.misc.Unsafe. The supported API since Java 9 is VarHandle, which exposes CAS plus fine-grained memory-ordering modes — plain, opaque, acquire-release, volatile. You almost never need it, but that's what the JDK's own concurrent classes use under the hood.

Interview-ready: lock-free guarantees system-wide progress, some thread advances; wait-free guarantees per-thread progress, every thread finishes in bounded steps. CAS loops are lock-free, not wait-free. And if you need a concurrent stack or queue, you do not write a Treiber stack — you use ConcurrentLinkedDeque.
