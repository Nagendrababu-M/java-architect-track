Stage 4: ReadWriteLock and StampedLock — scaling read-heavy state. A plain lock serializes everything, even reads that don't conflict. For read-heavy data — a config, a cache, a routing table read thousands of times per write — that's wasteful, because concurrent reads are perfectly safe as long as nobody is writing.

ReentrantReadWriteLock splits the lock in two. The read lock can be held by many readers simultaneously. The write lock is exclusive — it blocks all readers and all other writers. Three things to know. One: multiple readers concurrently, one writer exclusively. Two: downgrade is allowed — you can hold the write lock, acquire the read lock, then release the write lock, to keep reading what you just wrote. But upgrade is not allowed — if you hold the read lock and try to acquire the write lock, you deadlock, because the writer waits for all readers to release, including yourself. Three: write starvation — under a constant stream of readers, a writer may never get in. Use the fair constructor if writes must make progress. And the net: a read-write lock wins only when reads greatly outnumber writes and the critical section is long enough that the extra bookkeeping pays off. For very short sections, a plain lock can actually be faster.

StampedLock, from Java 8, adds an optimistic read. You call tryOptimisticRead, which takes no lock — it just returns a version stamp. You read your fields. Then you call validate with the stamp: if no writer intervened, you're done, having acquired nothing. Only if a writer did intervene do you fall back to a real read lock and re-read. For overwhelmingly read-heavy data, this beats ReadWriteLock by avoiding even read-lock contention. The catch: StampedLock is not reentrant, and not directly tied to Condition objects — so it's a specialized tool, not a drop-in replacement.

Three interview answers. Why can't you upgrade a read lock to a write lock? Because the write lock waits for all readers to release; holding the read lock while requesting write means waiting for yourself — deadlock. Downgrade, write to read, is fine. When does ReadWriteLock not help? Short critical sections, where its bookkeeping costs more than it saves, or write-heavy workloads, where writers are exclusive anyway. And what does StampedLock add? Optimistic reads — read without locking, then validate, falling back only on a real conflict — best for very read-heavy data, but non-reentrant.
