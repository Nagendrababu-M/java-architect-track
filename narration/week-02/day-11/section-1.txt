Day 11: synchronization primitives. Day 5 gave you the rules — shared mutable state needs a happens-before edge or you have a data race. Day 10 gave you the execution machinery — threads and pools. Today connects them: the primitives that actually create those happens-before edges and enforce mutual exclusion. The architect's theme: correctness is the floor, not the goal. Any lock makes code correct; the skill is making it correct and scalable, because lock contention silently destroys throughput.

Let's start with synchronized and the intrinsic monitor. Every Java object has an invisible lock attached to it — its intrinsic lock, or monitor. The synchronized keyword acquires it on entry and releases it on exit, normally or via an exception. You can synchronize on an explicit lock object, or use synchronized on an instance method, which locks "this", or on a static method, which locks the Class object. Under the hood the compiler emits monitorenter and monitorexit bytecodes, and the lock state lives in the object header's mark word. Historically the JVM optimized this with biased and lightweight locking, but biased locking was removed in JDK 15 and later, so don't cite it as current.

Here's the most important point, and most people name only the first part: synchronized provides three things, not one. Mutual exclusion: one thread in the block at a time per monitor. Visibility: on release, all writes are flushed; on acquire, the next thread sees them. And ordering: the unlock happens-before the next lock of the same monitor — that's the Day 5 edge, and it's what makes ordinary fields written inside the block visible to the next thread. The consequence that trips people up: a thread that only reads a shared field must also synchronize, on the same monitor, to be guaranteed the latest value. Exclusion on the writer alone is not enough; visibility needs the matching acquire.

Two more essentials. Reentrancy: intrinsic locks are reentrant — a thread holding a monitor can re-acquire it without deadlocking itself, because the JVM tracks a hold count. That's why a synchronized method can call another synchronized method on the same object. And choosing the lock object: don't lock on something a caller can also lock — not synchronized-on-this in a public class, not a String literal or boxed Integer that might be interned and shared. Prefer a private final lock object: it can't be locked by anyone else, so you can actually reason about who contends for it.
