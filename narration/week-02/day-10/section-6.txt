Stage 6 is your capstone — pick the right executor for three different workloads. For each, you design the executor: type, queue, size, saturation policy, and justify it. Let me set them up, then walk the model answers.

Workload A: an image-resizing service. Each task is pure CPU — decode, resize, encode — runs about two hundred milliseconds, on a sixteen-core box, and requests can burst far above capacity. CPU-bound on sixteen cores means core and max pool size of about sixteen or seventeen, cores plus one. More threads only add context-switch tax, since there's no spare CPU. Use a bounded ArrayBlockingQueue so excess bursts don't blow memory, plus CallerRunsPolicy so that when saturated, the submitting thread runs the resize itself, throttling intake to what sixteen cores can sustain. Build it as an explicit ThreadPoolExecutor, not newFixedThreadPool, which would use an unbounded queue and ignore max pool size. Under a burst, the service slows gracefully instead of OOM-ing or thrashing.

Workload B: an order-enrichment service. Each task makes three sequential REST calls — about a hundred fifty milliseconds of wait, five milliseconds of compute — and must never drop an order, but must not OOM under a spike. This is I/O-bound: wait over compute is about thirty, so threads is roughly cores times thirty-one — on sixteen cores, hundreds, because threads are mostly parked. With platform threads, use a generous pool, a bounded queue, and CallerRunsPolicy. "Never drop" rules out the Discard policies; the bounded queue plus CallerRuns gives backpressure, so a spike slows the producer instead of growing memory. The producer experiences rising latency — the system degrades, it doesn't fall over. But the better answer here is virtual threads, Day 8: this is blocking I/O, so newVirtualThreadPerTaskExecutor gives near-unlimited cheap concurrency without pool sizing — and you bound concurrency at the real constraint, the downstream service or a semaphore, not at the threads.

Workload C: a market-data fan-out, pushing the latest quote to subscribers every hundred milliseconds; if the system falls behind, stale quotes are worthless — only the newest matters. Use a ScheduledExecutorService for the cadence. Because only the freshest quote matters, DiscardOldestPolicy — or an explicit latest-value buffer — is correct: drop stale queued pushes, keep the newest. Prefer scheduleWithFixedDelay if push duration varies, to avoid runs bunching. And the pitfall to guard against: if the push task ever throws an uncaught exception, the scheduler silently cancels all future executions — the feed just stops. Wrap the task body in try-catch, and alert on "no tick in N seconds."
