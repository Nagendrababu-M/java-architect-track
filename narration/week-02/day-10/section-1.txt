Welcome to Week 2 — concurrency mastery — and Day 10: threads, thread pools, and the ExecutorService framework. Week 1 taught you how memory works and how the JVM behaves under load. Week 2 is about doing many things at once, correctly. Day 5 gave you the rules — the memory model and happens-before. Now you learn the machinery that runs concurrent work.

Here's the architect's counterintuitive lesson for today: you will spend almost no time creating threads, and almost all your time deciding how many, what queue, and what happens when the system is overwhelmed. Those three decisions separate a service that degrades gracefully from one that falls over.

Let's start with the thread itself. A Java platform thread is a thin wrapper over an operating-system thread. When you call thread dot start, the JVM asks the OS for a real kernel thread, and the OS scheduler time-slices it across cores. That one-to-one mapping has three consequences. Memory: each thread reserves a stack, roughly half a megabyte to a megabyte by default — so ten thousand threads is gigabytes of stacks before any work happens. Creation cost: it's a syscall into the kernel, expensive relative to the tiny work most tasks do. And the context-switch tax: when the OS juggles far more threads than cores, it burns CPU on bookkeeping instead of your code.

This is exactly why thread pools exist. Threads are costly enough that you reuse a fixed set rather than spawning one per task. Day 8's virtual threads change this calculus for I/O-bound work, but platform threads and pools remain the model for CPU-bound work, and the foundation you must understand first.

Now the thread lifecycle — and these are the same states you read in Day 6's thread dumps. NEW: created but start not yet called. RUNNABLE: executing or ready to execute, including running in native or I/O. BLOCKED: waiting to acquire a monitor lock another thread holds. WAITING: parked indefinitely, on wait, join, or park, until signalled. TIMED_WAITING: parked with a timeout, like sleep or a timed wait. And TERMINATED: finished or threw.

Finally, how you describe work. A Runnable returns nothing and can't throw checked exceptions. A Callable returns a value and can throw. And a Future is a handle to a result that may not exist yet. When you submit a Callable, you get a Future, and future dot get blocks the calling thread until the result is ready — so always prefer the timed overload, future dot get with a timeout, so a stuck task can't hang you forever. And future dot cancel requests cancellation, interrupting the running thread.
