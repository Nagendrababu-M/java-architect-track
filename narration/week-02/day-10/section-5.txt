Stage 5: sizing, scheduling, and graceful shutdown.

First, how many threads? The wrong size is either starvation, too few, or context-switch tax, too many. Brian Goetz's formula, derived from Little's Law: threads equals cores times target utilization times one plus wait time over compute time. Two cases fall out of that. CPU-bound work, where wait is roughly zero: threads is about cores plus one. More threads just thrash the scheduler — there's no more CPU to use. And I/O-bound work, where wait dwarfs compute: many more threads, because most are parked waiting on I/O, not using a core. A task that waits nine milliseconds for every one millisecond of compute can keep about ten times the core count busy.

And that's exactly why virtual threads exist — Day 8. For I/O-bound work, the right platform-thread count is huge and expensive; virtual threads make that count effectively free. For CPU-bound work, a small platform-thread pool sized near the core count stays correct.

Next, ScheduledExecutorService for periodic work. scheduleAtFixedRate runs every period measured from the start time — a steady cadence; but if a run overruns the period, the next starts immediately, so runs can bunch up. scheduleWithFixedDelay always leaves a fixed gap after each run finishes — safer when run durations vary. And here's the silent-death pitfall, which is a favorite interview gotcha: if a scheduled task throws an uncaught exception, the scheduler cancels all future executions of that task, silently. Your periodic job just stops, with no error. Always wrap the task body in try-catch so an exception can never escape.

Finally, graceful shutdown — the step everyone forgets. shutdown stops accepting new tasks and finishes the queued ones — it's graceful and drains the queue. awaitTermination waits for in-flight work for a bounded time. And if that times out, shutdownNow interrupts the running tasks and returns the ones it dropped. A pool whose threads are non-daemon will keep the entire JVM alive if you never shut it down — that's the classic "my app won't exit" bug and a resource leak. Also, name your threads with a ThreadFactory so your Day 6 thread dumps are actually readable.

Three interview answers to bank. Sizing CPU-bound versus I/O-bound: cores plus one versus cores times one plus wait over compute. fixedRate versus fixedDelay: period from each start versus gap after each finish. And shutdown versus shutdownNow: drain the queue versus interrupt and return undone tasks — pair shutdown with awaitTermination, then escalate.
