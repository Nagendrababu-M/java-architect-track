Stage 3: ThreadPoolExecutor internals — the real engine. Every factory method from Stage 2 just calls this one constructor with different arguments. Understand its seven parameters and you understand all thread pools.

corePoolSize: the threads kept alive even when idle. maximumPoolSize: the hard ceiling on threads. keepAliveTime: how long an idle non-core thread survives before it dies. The workQueue: a blocking queue holding tasks waiting for a thread — and you want this bounded. The threadFactory: which names your threads, sets daemon status, and the uncaught-exception handler. And the rejection handler: what to do when the pool is full, which is Stage 4.

Now the single most important thing to know about thread pools — the exact algorithm for how a submitted task flows, and it's a frequent interview question. Step one: if fewer than corePoolSize threads are running, start a new thread for the task — even if other threads are idle. Step two: otherwise, try to enqueue the task in the work queue. Step three: if the queue is full, start a new thread, up to maximumPoolSize. Step four: if the pool is at max and the queue is full, reject the task via the saturation policy.

That ordering leads to a counterintuitive consequence that trips up almost everyone: max pool size does nothing with an unbounded queue. Because step three only fires when the queue is full, an unbounded queue — the newFixedThreadPool default — can never fill. So the pool never grows past corePoolSize, and maximumPoolSize is silently ignored. If you want extra threads to kick in under load, you need a bounded queue. So if someone sets core to four, max to fifty, and only ever sees four threads under heavy load — that's why. The unbounded queue never fills, so step three never triggers.

Finally, the work queue choices. ArrayBlockingQueue: bounded, fixed capacity — the architect's default, because it gives backpressure. LinkedBlockingQueue: optionally bounded, but unbounded by default — that's the trap. SynchronousQueue: zero capacity, each task must be handed directly to a thread, which is what newCachedThreadPool uses to force thread creation. And PriorityBlockingQueue: unbounded, ordered by priority.
